# Claude Code ベストプラクティス

**原文:** https://code.claude.com/docs/en/best-practices

---

Claude Codeはエージェント型のコーディング環境だ。質問に答えて待機するチャットボットとは異なり、Claude Codeはファイルを読み、コマンドを実行し、変更を加え、あなたが見守っている間も、リダイレクトしている間も、あるいは完全に離席している間も、自律的に問題を解決できる。

これにより働き方が変わる。自分でコードを書いてClaudeにレビューを頼む代わりに、何が欲しいかを説明すれば、Claudeがどう構築するかを考え出す。Claudeは探索し、計画し、実装する。

しかし、この自律性には学習曲線が伴う。Claudeは理解すべき特定の制約の中で動作する。

このガイドでは、Anthropicの社内チームや、様々なコードベース、言語、環境でClaude Codeを使用するエンジニアの間で効果的であることが証明されたパターンを紹介する。エージェントループの内部動作については、[How Claude Code works](https://code.claude.com/docs/en/how-claude-code-works)を参照してほしい。

---

ほとんどのベストプラクティスは1つの制約に基づいている：**Claudeのコンテキストウィンドウ（context window）は急速に埋まり、埋まるにつれてパフォーマンスが低下する。**

Claudeのコンテキストウィンドウは、すべてのメッセージ、Claudeが読むすべてのファイル、すべてのコマンド出力を含む会話全体を保持する。しかし、これはすぐに埋まってしまう。1回のデバッグセッションやコードベース探索で、数万トークンを生成・消費することもある。

これが重要なのは、コンテキストが埋まるにつれてLLMのパフォーマンスが低下するからだ。コンテキストウィンドウがいっぱいになりかけると、Claudeは以前の指示を「忘れ」たり、より多くのミスを犯し始めたりする可能性がある。コンテキストウィンドウは管理すべき最も重要なリソースだ。トークン使用量を削減する詳細な戦略については、[Reduce token usage](https://code.claude.com/docs/en/costs#reduce-token-usage)を参照してほしい。

## Claudeに作業を検証する方法を与える

テスト、スクリーンショット、または期待される出力を含めることで、Claudeが自己チェックできるようにする。**これが最もレバレッジの高いことだ。**

Claudeは自分の作業を検証できるとき、つまりテストを実行し、スクリーンショットを比較し、出力を検証できるとき、劇的に良いパフォーマンスを発揮する。

明確な成功基準がないと、正しく見えるが実際には機能しないものを生成する可能性がある。あなたが唯一のフィードバックループとなり、すべてのミスにあなたの注意が必要になる。

| 戦略 | Before | After |
| --- | --- | --- |
| **検証基準を提供する** | _「メールアドレスを検証する関数を実装して」_ | _「validateEmail関数を書いて。テストケース例：user@example.comはtrue、invalidはfalse、user@.comはfalse。実装後にテストを実行して」_ |
| **UI変更を視覚的に検証する** | _「ダッシュボードをもっと良くして」_ | _「[スクリーンショットを貼り付け] このデザインを実装して。結果のスクリーンショットを撮って元と比較して。違いをリストアップして修正して」_ |
| **症状ではなく根本原因に対処する** | _「ビルドが失敗してる」_ | _「ビルドがこのエラーで失敗する：[エラーを貼り付け]。修正してビルドが成功することを確認して。エラーを抑制するのではなく、根本原因に対処して」_ |

UI変更は[Claude in Chrome拡張機能](https://code.claude.com/docs/en/chrome)を使用して検証できる。ブラウザを開き、UIをテストし、コードが動作するまで反復する。

検証はテストスイート、リンター、または出力をチェックするBashコマンドでもよい。検証を堅牢にすることに投資しよう。

## まず探索し、次に計画し、それからコーディングする

調査と計画を実装から分離し、間違った問題を解決することを避ける。

Claudeにいきなりコーディングさせると、間違った問題を解決するコードが生成される可能性がある。[Plan Mode](https://code.claude.com/docs/en/common-workflows#use-plan-mode-for-safe-code-analysis)を使用して、探索と実行を分離しよう。

推奨されるワークフローには4つのフェーズがある：

### 1. 探索

Plan Modeに入る。Claudeは変更を加えずにファイルを読み、質問に答える。

```
claude (Plan Mode)

/src/authを読んで、セッションとログインをどう処理しているか理解して。
また、シークレット用の環境変数をどう管理しているかも見て。
```

### 2. 計画

Claudeに詳細な実装計画を作成するよう依頼する。

```
claude (Plan Mode)

Google OAuthを追加したい。どのファイルを変更する必要がある？
セッションフローは？計画を作成して。
```

`Ctrl+G`を押すと、Claudeが進む前に計画をテキストエディタで直接編集できる。

### 3. 実装

Normal Modeに戻り、Claudeに計画に対して検証しながらコーディングさせる。

```
claude (Normal Mode)

計画からOAuthフローを実装して。コールバックハンドラのテストを書いて、
テストスイートを実行して失敗があれば修正して。
```

### 4. コミット

Claudeに説明的なメッセージでコミットさせ、PRを作成させる。

```
claude (Normal Mode)

説明的なメッセージでコミットしてPRを開いて
```

Plan Modeは便利だが、オーバーヘッドも追加する。

スコープが明確で修正が小さいタスク（タイポの修正、ログ行の追加、変数名の変更など）では、Claudeに直接やらせよう。

計画が最も有用なのは、アプローチが不確かなとき、変更が複数のファイルを修正するとき、または変更されるコードに不慣れなときだ。差分を一文で説明できるなら、計画をスキップしよう。

## プロンプトに具体的なコンテキストを提供する

指示が正確であるほど、修正が少なくて済む。

Claudeは意図を推測できるが、心を読むことはできない。特定のファイルを参照し、制約を述べ、パターン例を指し示そう。

| 戦略 | Before | After |
| --- | --- | --- |
| **タスクのスコープを定める。** どのファイル、どのシナリオ、テストの好みを指定する。 | _「foo.pyのテストを追加して」_ | _「foo.pyのテストを書いて、ユーザーがログアウトしているエッジケースをカバーして。モックは避けて。」_ |
| **ソースを指し示す。** 質問に答えられるソースをClaudeに指示する。 | _「ExecutionFactoryはなぜこんな奇妙なAPIなの？」_ | _「ExecutionFactoryのgit履歴を調べて、APIがどのようにして今の形になったかまとめて」_ |
| **既存のパターンを参照する。** コードベースのパターンをClaudeに指し示す。 | _「カレンダーウィジェットを追加して」_ | _「ホームページで既存のウィジェットがどう実装されているか見て、パターンを理解して。HotDogWidget.phpが良い例。そのパターンに従って、ユーザーが月を選択し、前後にページネートして年を選べる新しいカレンダーウィジェットを実装して。コードベースで既に使われているライブラリ以外は使わずにゼロから構築して。」_ |
| **症状を説明する。** 症状、おそらくの場所、「修正された」とはどういう状態かを提供する。 | _「ログインバグを修正して」_ | _「ユーザーがセッションタイムアウト後にログインが失敗すると報告している。src/auth/の認証フローを確認して、特にトークンリフレッシュを。問題を再現する失敗するテストを書いてから修正して」_ |

曖昧なプロンプトは、探索していて軌道修正する余裕があるときに有用だ。`「このファイルで何を改善する？」`のようなプロンプトは、聞こうと思わなかったことを浮き彫りにできる。

### リッチコンテンツを提供する

`@`でファイルを参照し、スクリーンショット/画像を貼り付け、またはデータを直接パイプする。

Claudeにリッチデータを提供する方法はいくつかある：

- **`@`でファイルを参照する** - コードがどこにあるか説明する代わりに。Claudeは応答前にファイルを読む。
- **画像を直接貼り付ける** - コピー/ペーストまたはドラッグ＆ドロップでプロンプトに画像を入れる。
- **URLを与える** - ドキュメントやAPIリファレンス用。`/permissions`を使用して頻繁に使用するドメインを許可リストに追加できる。
- **データをパイプで渡す** - `cat error.log | claude`を実行してファイル内容を直接送信する。
- **Claudeに必要なものを取得させる** - Bashコマンド、MCPツール、またはファイル読み込みを使用してClaudeに自分でコンテキストを取得させる。

## 環境を設定する

いくつかのセットアップ手順で、すべてのセッションでClaude Codeが大幅に効果的になる。拡張機能の概要とそれぞれをいつ使用するかについては、[Extend Claude Code](https://code.claude.com/docs/en/features-overview)を参照してほしい。

### 効果的なCLAUDE.mdを書く

`/init`を実行して現在のプロジェクト構造に基づいたCLAUDE.mdのスターターファイルを生成し、時間をかけて改善しよう。

CLAUDE.mdはClaudeがすべての会話の開始時に読む特別なファイルだ。Bashコマンド、コードスタイル、ワークフロールールを含める。これにより、Claudeに**コードだけでは推測できない永続的なコンテキスト**を与える。

`/init`コマンドはコードベースを分析してビルドシステム、テストフレームワーク、コードパターンを検出し、改善するための堅固な基盤を提供する。

CLAUDE.mdファイルに必須のフォーマットはないが、短く人間が読めるものにしよう。例えば：

```
CLAUDE.md

# Code style
- Use ES modules (import/export) syntax, not CommonJS (require)
- Destructure imports when possible (eg. import { foo } from 'bar')

# Workflow
- Be sure to typecheck when you're done making a series of code changes
- Prefer running single tests, and not the whole test suite, for performance
```

CLAUDE.mdは毎セッション読み込まれるので、広く適用されることだけを含めよう。特定のときだけ関連するドメイン知識やワークフローには、代わりに[skills](https://code.claude.com/docs/en/skills)を使用しよう。Claudeはすべての会話を肥大化させることなく、オンデマンドでそれらを読み込む。

簡潔に保とう。各行について問いかけよう：_「これを削除するとClaudeがミスをする原因になる？」_ そうでなければ、削除する。肥大化したCLAUDE.mdファイルは、Claudeがあなたの実際の指示を無視する原因になる！

| 含めるべきもの | 除外すべきもの |
| --- | --- |
| Claudeが推測できないBashコマンド | Claudeがコードを読めば分かるもの |
| デフォルトと異なるコードスタイルルール | Claudeが既に知っている標準的な言語規約 |
| テスト指示と好みのテストランナー | 詳細なAPIドキュメント（代わりにドキュメントにリンク） |
| リポジトリのエチケット（ブランチ命名、PR規約） | 頻繁に変わる情報 |
| プロジェクト固有のアーキテクチャ決定 | 長い説明やチュートリアル |
| 開発環境の癖（必要な環境変数） | コードベースのファイルごとの説明 |
| よくある落とし穴や非自明な動作 | 「クリーンなコードを書く」のような自明な慣行 |

ルールがあるにもかかわらずClaudeが望まないことをし続ける場合、おそらくファイルが長すぎてルールが埋もれている。CLAUDE.mdに答えがあることをClaudeが質問する場合、表現が曖昧かもしれない。CLAUDE.mdをコードのように扱おう：問題が起きたときにレビューし、定期的に剪定し、Claudeの動作が実際に変わるかどうか観察して変更をテストしよう。

強調（例：「IMPORTANT」や「YOU MUST」）を追加することで、遵守を改善するために指示を調整できる。CLAUDE.mdをgitにチェックインして、チームが貢献できるようにしよう。ファイルの価値は時間とともに複利で増加する。

CLAUDE.mdファイルは`@path/to/import`構文を使用して追加ファイルをインポートできる：

```
CLAUDE.md

See @README.md for project overview and @package.json for available npm commands.

# Additional Instructions
- Git workflow: @docs/git-instructions.md
- Personal overrides: @~/.claude/my-project-instructions.md
```

CLAUDE.mdファイルは複数の場所に配置できる：

- **ホームフォルダ（`~/.claude/CLAUDE.md`）**: すべてのClaudeセッションに適用
- **プロジェクトルート（`./CLAUDE.md`）**: gitにチェックインしてチームと共有するか、`CLAUDE.local.md`と名付けて`.gitignore`する
- **親ディレクトリ**: モノレポで`root/CLAUDE.md`と`root/foo/CLAUDE.md`の両方が自動的に取り込まれる場合に有用
- **子ディレクトリ**: Claudeはそれらのディレクトリ内のファイルで作業するときに、オンデマンドで子CLAUDE.mdファイルを取り込む

### 権限を設定する

`/permissions`を使用して安全なコマンドを許可リストに追加するか、`/sandbox`でOSレベルの分離を行う。これにより中断が減り、制御を維持できる。

デフォルトでは、Claude Codeはシステムを変更する可能性のあるアクション（ファイル書き込み、Bashコマンド、MCPツールなど）に対して許可を要求する。これは安全だが面倒だ。10回目の承認の後は、実際にはレビューしておらず、ただクリックしているだけだ。これらの中断を減らす方法は2つある：

- **許可リスト**: 安全だと分かっている特定のツールを許可する（`npm run lint`や`git commit`など）
- **サンドボックス**: ファイルシステムとネットワークアクセスを制限するOSレベルの分離を有効にし、定義された境界内でClaudeがより自由に作業できるようにする

あるいは、`--dangerously-skip-permissions`を使用して、リントエラーの修正やボイラープレートコードの生成などの封じ込められたワークフローですべての権限チェックをバイパスする。

> Claudeに任意のコマンドを実行させると、データ損失、システム破損、またはプロンプトインジェクションによるデータ流出が発生する可能性がある。インターネットアクセスのないサンドボックスでのみ`--dangerously-skip-permissions`を使用すること。

[権限の設定](https://code.claude.com/docs/en/settings)と[サンドボックスの有効化](https://code.claude.com/docs/en/sandboxing#sandboxing)の詳細を読もう。

### CLIツールを使用する

外部サービスとやり取りするとき、`gh`、`aws`、`gcloud`、`sentry-cli`などのCLIツールを使用するようClaude Codeに伝えよう。

CLIツールは外部サービスとやり取りする最もコンテキスト効率の良い方法だ。GitHubを使用している場合、`gh` CLIをインストールしよう。Claudeはissueの作成、プルリクエストのオープン、コメントの読み取りにそれを使用する方法を知っている。`gh`がないと、ClaudeはまだGitHub APIを使用できるが、認証されていないリクエストはしばしばレート制限に達する。

Claudeはまだ知らないCLIツールを学習することにも効果的だ。`「'foo-cli-tool --help'を使ってfooツールについて学び、それを使ってA、B、Cを解決して」`のようなプロンプトを試してみよう。

### MCPサーバーを接続する

`claude mcp add`を実行してNotion、Figma、またはデータベースなどの外部ツールを接続する。

[MCPサーバー](https://code.claude.com/docs/en/mcp)を使用すると、Claudeにissueトラッカーから機能を実装させたり、データベースにクエリを実行させたり、モニタリングデータを分析させたり、Figmaからデザインを統合させたり、ワークフローを自動化させたりできる。

### フックを設定する

例外なく毎回実行しなければならないアクションにはフックを使用する。

[フック](https://code.claude.com/docs/en/hooks-guide)はClaudeのワークフローの特定のポイントで自動的にスクリプトを実行する。アドバイス的なCLAUDE.mdの指示とは異なり、フックは決定論的であり、アクションが確実に実行されることを保証する。

Claudeはあなたのためにフックを書くことができる。_「すべてのファイル編集後にeslintを実行するフックを書いて」_ や _「migrationsフォルダへの書き込みをブロックするフックを書いて」_ のようなプロンプトを試してみよう。インタラクティブな設定には`/hooks`を実行するか、`.claude/settings.json`を直接編集しよう。

### スキルを作成する

`.claude/skills/`に`SKILL.md`ファイルを作成して、Claudeにドメイン知識と再利用可能なワークフローを与える。

[スキル](https://code.claude.com/docs/en/skills)は、プロジェクト、チーム、またはドメインに固有の情報でClaudeの知識を拡張する。Claudeは関連性があるときに自動的にそれらを適用するか、`/skill-name`で直接呼び出すことができる。

`.claude/skills/`に`SKILL.md`を含むディレクトリを追加してスキルを作成する：

```
.claude/skills/api-conventions/SKILL.md

---
name: api-conventions
description: REST API design conventions for our services
---
# API Conventions
- Use kebab-case for URL paths
- Use camelCase for JSON properties
- Always include pagination for list endpoints
- Version APIs in the URL path (/v1/, /v2/)
```

スキルは直接呼び出す繰り返し可能なワークフローも定義できる：

```
.claude/skills/fix-issue/SKILL.md

---
name: fix-issue
description: Fix a GitHub issue
disable-model-invocation: true
---
Analyze and fix the GitHub issue: $ARGUMENTS.

1. Use `gh issue view` to get the issue details
2. Understand the problem described in the issue
3. Search the codebase for relevant files
4. Implement the necessary changes to fix the issue
5. Write and run tests to verify the fix
6. Ensure code passes linting and type checking
7. Create a descriptive commit message
8. Push and create a PR
```

`/fix-issue 1234`で呼び出す。手動でトリガーしたい副作用のあるワークフローには`disable-model-invocation: true`を使用する。

### カスタムサブエージェントを作成する

`.claude/agents/`に特化したアシスタントを定義し、Claudeが孤立したタスクを委任できるようにする。

[サブエージェント](https://code.claude.com/docs/en/sub-agents)は、独自のコンテキストと独自の許可されたツールセットで実行される。多くのファイルを読むタスクや、メインの会話を乱雑にすることなく特化した焦点が必要なタスクに有用だ。

```
.claude/agents/security-reviewer.md

---
name: security-reviewer
description: Reviews code for security vulnerabilities
tools: Read, Grep, Glob, Bash
model: opus
---
You are a senior security engineer. Review code for:
- Injection vulnerabilities (SQL, XSS, command injection)
- Authentication and authorization flaws
- Secrets or credentials in code
- Insecure data handling

Provide specific line references and suggested fixes.
```

Claudeに明示的にサブエージェントを使用するよう伝える：_「サブエージェントを使用してこのコードをセキュリティ問題についてレビューして。」_

### プラグインをインストールする

`/plugin`を実行してマーケットプレイスを閲覧する。プラグインは設定なしでスキル、ツール、統合を追加する。

[プラグイン](https://code.claude.com/docs/en/plugins)は、コミュニティとAnthropicからのスキル、フック、サブエージェント、MCPサーバーを単一のインストール可能なユニットにバンドルする。型付き言語で作業している場合、[コードインテリジェンスプラグイン](https://code.claude.com/docs/en/discover-plugins#code-intelligence)をインストールして、Claudeに正確なシンボルナビゲーションと編集後の自動エラー検出を与えよう。

スキル、サブエージェント、フック、MCPのいずれを選択するかのガイダンスについては、[Extend Claude Code](https://code.claude.com/docs/en/features-overview#match-features-to-your-goal)を参照してほしい。

## 効果的にコミュニケーションする

Claude Codeとのコミュニケーション方法は、結果の品質に大きく影響する。

### コードベースについて質問する

シニアエンジニアに聞くような質問をClaudeに聞こう。

新しいコードベースにオンボーディングするとき、学習と探索にClaude Codeを使用しよう。他のエンジニアに聞くのと同じ種類の質問をClaudeに聞ける：

- ロギングはどう機能する？
- 新しいAPIエンドポイントをどう作る？
- `foo.rs`の134行目の`async move { ... }`は何をしている？
- `CustomerOnboardingFlowImpl`はどんなエッジケースを処理している？
- なぜこのコードは333行目で`bar()`ではなく`foo()`を呼んでいる？

この方法でClaude Codeを使用することは効果的なオンボーディングワークフローであり、立ち上がり時間を改善し、他のエンジニアへの負荷を軽減する。特別なプロンプティングは不要：直接質問しよう。

### Claudeにインタビューさせる

大きな機能については、最初にClaudeにインタビューさせよう。最小限のプロンプトから始めて、Claudeに`AskUserQuestion`ツールを使用してインタビューするよう依頼する。

Claudeは技術的な実装、UI/UX、エッジケース、トレードオフなど、まだ考えていなかったかもしれないことについて質問する。

```
[簡潔な説明]を構築したい。AskUserQuestionツールを使用して詳細にインタビューして。

技術的な実装、UI/UX、エッジケース、懸念事項、トレードオフについて聞いて。
明白な質問はせず、私が考えていなかったかもしれない難しい部分を掘り下げて。

すべてをカバーするまでインタビューを続け、それから完全な仕様をSPEC.mdに書いて。
```

仕様が完成したら、それを実行するために新しいセッションを開始する。新しいセッションは実装に完全に焦点を当てたクリーンなコンテキストを持ち、参照する書面の仕様がある。

## セッションを管理する

会話は永続的で可逆的だ。これを活用しよう！

### 早期かつ頻繁に軌道修正する

Claudeが軌道から外れていることに気づいたら、すぐに修正しよう。

最良の結果は、タイトなフィードバックループから生まれる。Claudeは時々最初の試みで問題を完璧に解決するが、一般的には素早く修正することでより良いソリューションがより速く生まれる。

- **`Esc`**: `Esc`キーでClaudeをアクション中に停止させる。コンテキストは保持されるので、リダイレクトできる。
- **`Esc + Esc`または`/rewind`**: `Esc`を2回押すか`/rewind`を実行して巻き戻しメニューを開き、以前の会話とコード状態を復元する。
- **`「それを元に戻して」`**: Claudeに変更を元に戻させる。
- **`/clear`**: 無関係なタスク間でコンテキストをリセットする。無関係なコンテキストを持つ長いセッションはパフォーマンスを低下させる可能性がある。

1回のセッションで同じ問題についてClaudeを2回以上修正した場合、コンテキストは失敗したアプローチで乱雑になっている。`/clear`を実行し、学んだことを組み込んだより具体的なプロンプトで新しく始めよう。より良いプロンプトを持つクリーンなセッションは、蓄積された修正を持つ長いセッションをほぼ常に上回る。

### コンテキストを積極的に管理する

無関係なタスク間で`/clear`を実行してコンテキストをリセットする。

Claude Codeはコンテキスト制限に近づくと自動的に会話履歴を圧縮し、重要なコードと決定を保持しながらスペースを解放する。

長いセッション中、Claudeのコンテキストウィンドウは無関係な会話、ファイル内容、コマンドで埋まる可能性がある。これはパフォーマンスを低下させ、時にClaudeの注意を散漫にする。

- タスク間で頻繁に`/clear`を使用してコンテキストウィンドウを完全にリセットする
- 自動圧縮がトリガーされると、Claudeはコードパターン、ファイル状態、重要な決定など、最も重要なことを要約する
- より多くの制御には、`/compact <instructions>`を実行する、例えば`/compact APIの変更に焦点を当てて`
- 重要なコンテキストが要約を生き残ることを確実にするために、CLAUDE.mdで`「圧縮時は、変更されたファイルの完全なリストとすべてのテストコマンドを常に保持すること」`のような指示で圧縮動作をカスタマイズする

### 調査にサブエージェントを使用する

`「サブエージェントを使用してXを調査して」`で調査を委任する。サブエージェントは別のコンテキストで探索し、メインの会話を実装のためにクリーンに保つ。

コンテキストが根本的な制約であるため、サブエージェントは利用可能な最も強力なツールの1つだ。Claudeがコードベースを調査するとき、多くのファイルを読み、それらすべてがコンテキストを消費する。サブエージェントは別のコンテキストウィンドウで実行し、要約を報告する：

```
サブエージェントを使用して、認証システムがトークンリフレッシュを
どう処理しているか、再利用すべき既存のOAuthユーティリティがあるか調査して。
```

サブエージェントはコードベースを探索し、関連ファイルを読み、発見を報告する、すべてメインの会話を乱雑にすることなく。

Claudeが何かを実装した後の検証にもサブエージェントを使用できる：

```
サブエージェントを使用してこのコードのエッジケースをレビューして
```

### チェックポイントで巻き戻す

Claudeが行うすべてのアクションはチェックポイントを作成する。会話、コード、または両方を任意の以前のチェックポイントに復元できる。

Claudeは変更前に自動的にチェックポイントを作成する。`Escape`をダブルタップするか`/rewind`を実行してチェックポイントメニューを開く。会話のみを復元する（コード変更を維持）、コードのみを復元する（会話を維持）、または両方を復元できる。

すべての動きを慎重に計画する代わりに、Claudeにリスクのあることを試すよう伝えることができる。うまくいかなければ、巻き戻して別のアプローチを試す。チェックポイントはセッション間で永続するので、ターミナルを閉じても後で巻き戻すことができる。

> チェックポイントは_Claudeによって_行われた変更のみを追跡し、外部プロセスは追跡しない。これはgitの代わりにはならない。

### 会話を再開する

`claude --continue`を実行して中断したところから再開するか、`--resume`を使用して最近のセッションから選択する。

Claude Codeは会話をローカルに保存する。タスクが複数のセッションにまたがるとき（機能を開始し、中断され、翌日に戻る）、コンテキストを再説明する必要がない：

```
claude --continue    # 最新の会話を再開
claude --resume      # 最近の会話から選択
```

`/rename`を使用してセッションに説明的な名前を付けよう（`"oauth-migration"`、`"debugging-memory-leak"`など）、後で見つけやすくなる。セッションをブランチのように扱おう。異なるワークストリームは別々の永続的なコンテキストを持つことができる。

## 自動化とスケール

1つのClaudeで効果的になったら、並列セッション、ヘッドレスモード、ファンアウトパターンで出力を倍増させよう。

ここまでのすべては、1人の人間、1つのClaude、1つの会話を想定している。しかし、Claude Codeは水平方向にスケールする。このセクションのテクニックは、より多くのことを成し遂げる方法を示す。

### ヘッドレスモードを実行する

CI、pre-commitフック、またはスクリプトで`claude -p "prompt"`を使用する。ストリーミングJSON出力には`--output-format stream-json`を追加する。

`claude -p "your prompt"`で、インタラクティブセッションなしでClaudeをヘッドレスで実行できる。ヘッドレスモードは、CIパイプライン、pre-commitフック、または任意の自動化ワークフローにClaudeを統合する方法だ。出力フォーマット（プレーンテキスト、JSON、ストリーミングJSON）により、結果をプログラムで解析できる。

```bash
# ワンオフクエリ
claude -p "このプロジェクトが何をするか説明して"

# スクリプト用の構造化出力
claude -p "すべてのAPIエンドポイントをリストして" --output-format json

# リアルタイム処理用のストリーミング
claude -p "このログファイルを分析して" --output-format stream-json
```

### 複数のClaudeセッションを実行する

複数のClaudeセッションを並列で実行して、開発を高速化し、孤立した実験を実行し、複雑なワークフローを開始しよう。

並列セッションを実行する主な方法は2つある：

- [Claude Desktop](https://code.claude.com/docs/en/desktop): 複数のローカルセッションを視覚的に管理する。各セッションは独自の孤立したworktreeを取得する。
- [Claude Code on the web](https://code.claude.com/docs/en/claude-code-on-the-web): Anthropicの安全なクラウドインフラストラクチャで、孤立したVMで実行する。

作業の並列化に加えて、複数のセッションは品質重視のワークフローを可能にする。新鮮なコンテキストは、Claudeが書いたばかりのコードに偏らないため、コードレビューを改善する。

例えば、Writer/Reviewerパターンを使用する：

| Session A (Writer) | Session B (Reviewer) |
| --- | --- |
| `APIエンドポイントのレートリミッターを実装して` |  |
|  | `@src/middleware/rateLimiter.tsのレートリミッター実装をレビューして。エッジケース、競合状態、既存のミドルウェアパターンとの一貫性を探して。` |
| `これがレビューのフィードバックだ：[Session Bの出力]。これらの問題に対処して。` |  |

テストでも同様のことができる：1つのClaudeにテストを書かせ、別のClaudeにそれをパスするコードを書かせる。

### ファイル間でファンアウトする

各タスクに対して`claude -p`を呼び出すループを作成する。バッチ操作の権限をスコープするには`--allowedTools`を使用する。

大規模なマイグレーションや分析では、多くの並列Claude呼び出しに作業を分散できる：

#### 1. タスクリストを生成する

Claudeにマイグレーションが必要なすべてのファイルをリストさせる（例：`マイグレーションが必要なすべての2,000個のPythonファイルをリストして`）

#### 2. リストをループするスクリプトを書く

```bash
for file in $(cat files.txt); do
  claude -p "Migrate $file from React to Vue. Return OK or FAIL." \
    --allowedTools "Edit,Bash(git commit *)"
done
```

#### 3. いくつかのファイルでテストしてから、スケールで実行する

最初の2-3ファイルでうまくいかなかったことに基づいてプロンプトを改善し、完全なセットで実行する。`--allowedTools`フラグはClaudeができることを制限する、これは無人で実行するときに重要だ。

既存のデータ/処理パイプラインにClaudeを統合することもできる：

```bash
claude -p "<your prompt>" --output-format json | your_command
```

開発中のデバッグには`--verbose`を使用し、本番環境ではオフにする。

### 安全な自律モード

`claude --dangerously-skip-permissions`を使用して、すべての権限チェックをバイパスし、Claudeを中断なしで作業させる。これはリントエラーの修正やボイラープレートコードの生成などのワークフローに適している。

> Claudeに任意のコマンドを実行させることはリスクがあり、データ損失、システム破損、またはデータ流出（例：プロンプトインジェクション攻撃経由）を引き起こす可能性がある。これらのリスクを最小化するために、インターネットアクセスのないコンテナで`--dangerously-skip-permissions`を使用すること。

サンドボックスを有効にすると（`/sandbox`）、より良いセキュリティで同様の自律性が得られる。サンドボックスはすべてのチェックをバイパスするのではなく、事前に境界を定義する。

## よくある失敗パターンを避ける

これらはよくある間違いだ。早期に認識することで時間を節約できる：

- **キッチンシンクセッション。** 1つのタスクから始め、無関係なことをClaudeに聞き、最初のタスクに戻る。コンテキストは無関係な情報でいっぱいだ。

  > **修正**: 無関係なタスク間で`/clear`する。

- **何度も修正する。** Claudeが何か間違えて、修正して、まだ間違っていて、また修正する。コンテキストは失敗したアプローチで汚染されている。

  > **修正**: 2回の失敗した修正の後、`/clear`して学んだことを組み込んだより良い初期プロンプトを書く。

- **過剰に指定されたCLAUDE.md。** CLAUDE.mdが長すぎると、重要なルールがノイズに埋もれて、Claudeは半分を無視する。

  > **修正**: 容赦なく剪定する。Claudeが指示なしで既に正しく何かをしている場合、削除するかフックに変換する。

- **信頼してから検証のギャップ。** Claudeはもっともらしく見えるがエッジケースを処理しない実装を生成する。

  > **修正**: 常に検証（テスト、スクリプト、スクリーンショット）を提供する。検証できない場合、出荷しない。

- **無限の探索。** スコープを定めずにClaudeに何かを「調査して」と頼む。Claudeは数百のファイルを読み、コンテキストを埋める。

  > **修正**: 調査を狭くスコープするか、サブエージェントを使用して探索がメインのコンテキストを消費しないようにする。

## 直感を養う

このガイドのパターンは絶対的なものではない。一般的にうまく機能する出発点だが、すべての状況で最適とは限らない。

時にはコンテキストを蓄積_させるべき_だ、なぜなら1つの複雑な問題に深く入っていて、履歴が価値があるからだ。時には計画をスキップしてClaudeに解決させるべきだ、なぜならタスクが探索的だからだ。時には曖昧なプロンプトがまさに正しい、なぜならClaudeが問題をどう解釈するか、制約する前に見たいからだ。

何がうまくいくか注意を払おう。Claudeが素晴らしい出力を生成するとき、何をしたか注意する：プロンプトの構造、提供したコンテキスト、どのモードにいたか。Claudeが苦労するとき、なぜか聞いてみる。コンテキストがノイズが多すぎた？プロンプトが曖昧すぎた？タスクが1回のパスには大きすぎた？

時間とともに、どのガイドも捕らえられない直感を養うことになる。いつ具体的で、いつオープンエンドにするか、いつ計画し、いつ探索するか、いつコンテキストをクリアし、いつ蓄積させるかが分かるようになる。
