// METHOD · 01

監査の前提と方法

判定基準はかいじゅうの指定通り「その指示が消えたら、実際に壊れる挙動・起きる事故があるか」。settings.json の hook 配線 (ツール実行に割り込む自動チェックスクリプトの設定)・permissions.deny (ツール権限の拒否リスト)・skill (再利用可能な手順書) 群への grep 検索で、依存関係を全て実測した上で判定しておる。優先順位も指定通り「誤削除のほうが冗長見逃しより重い」── 迷った指示は全て維持に倒し、迷った旨を 05 章に明記した。

判定を支える実測事実は 2 つある。

実測事実 1 — rules/ は全ファイルが毎セッション自動注入される。 本セッションへの注入内容を確認したところ、rules/ 配下の 8 ファイル全てが CLAUDE.md と同格の「全プロジェクト共通指示」として全文注入されておった。つまり CLAUDE.md 側にある「reference: rules/X.md」だけの節は、既に注入済みの同一情報への 2 本目の道標であり、削除しても挙動は何も変わらない。これが削除候補 5 件の根拠じゃ。

実測事実 2 — 「壁と地図」の構造。 rm・curl/wget・秘密ファイル読取・一括置換・git amend・plans 出力先・ヘッジ語 (「場合による」等の曖昧表現)・コード内絵文字・生成署名コメントは、hook と permissions.deny が「壁」として既に機械的にブロック済み。一見すると CLAUDE.md の対応指示は重複に見えるが、これらは「壁の地図」── 壁に衝突する前に代替手段 (trash を使え、pbpaste で受け取れ、MCP を使え) へ誘導する案内であり、消すと Claude が壁に繰り返し衝突して停滞する。よって地図系の指示は全て維持と判定した。

一覧表の判定バッジは 維持 短縮候補 削除候補 の 3 種で表記する。

Before (原文・全文)

# Citation Format (出典表記)

skill / rule / doc に外部出典を書くとき、プロジェクト固有の略記 (例: `note 260405`) だけで終わらせない。その略記は、当該プロジェクト文脈を持たない別セッション / 別PJ の Claude には解決不能な暗号になる。

理由: グローバルハーネス (`~/.claude/`) は特定プロジェクト情報を持たない前提で動く (CLAUDE.md Harness 節)。番号略記は発行元 PJ でしか辿れない。

## ルール

本文中は略記でよいが、末尾に出典セクションを置き、どのセッションの Claude でも機械的に解決できる形を必ず添える。
