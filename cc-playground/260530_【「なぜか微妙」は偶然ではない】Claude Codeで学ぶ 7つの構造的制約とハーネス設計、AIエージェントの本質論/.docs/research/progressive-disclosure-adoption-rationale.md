# Progressive Disclosure 採用理由の調査（note記事群横断）

## 調査目的・スコープ

CLAUDE.md 60行化(#94/#99)で採用予定の設計パターン「Progressive Disclosure」を、まさお氏 note 記事群から徹底的に裏取りし、**採用理由・既知のトレードオフ・対策**を事実ベースで集約する。3 Explore を並列で7記事の text.md へ横断 grep(逐語全文転写・原文引用)。

## 事実・発見

### 1. 問題 — なぜコンテキストを節約すべきか
- **C-1 帯域は有限・ゼロサム**(260405 L244「机のたとえ」): 枠に入る情報量に物理的上限。ある情報を入れると別の集中力が薄まる。枠が埋まりきらずともノイズが多ければ肝心な指示が埋もれる。
- **肥大化で遵守率低下**(260405 L262, L1526): 「CLAUDE.md にルールを足すほど守られなくなる」。長いほど各セクションが `<system_reminder>` 内で「任意」扱い。「百科事典を渡された旅行者は、結局何も読まない」。
- **定量**: 指示10個で遵守率 ~90%、50個以上で急落(260405 L929)。フロンティアLLMの一貫追従は約150-200が上限(260222 L208)。
- **Lost in the Middle**(260405 L264 / 260501 L913): 長いコンテキストの中盤の指示を読み落とす。
- **注意の汚染**(260426 L142 / 260222 L640): 途中で捨てた仮説・古い前提・読み散らしたファイルが判断に混ざる。
- **MCP 常時ロード**: ツール定義だけで 23k-50k トークン消費(260405 L294)。圧縮スラッシング(L336)。
- **核心テーゼ**(260405 L346): 「帯域は減る一方だ。重要な情報は早めに外部化し、**必要な情報は遅延ロードする**」。

### 2. 解 — Progressive Disclosure / ポインタ化 / 遅延ロード
- **設計パターン明示**(260405 L1530): 「CLAUDE.md は60行以下の地図にする。詳細はポインタ先に配置し、必要なときだけ Read で読み込ませる」。
- **効能**(260405 L1515): ポインタ型 = 帯域ゼロ時の負荷最小 / 必要時だけロード / **遵守率が向上**。
- **Skill 遅延ロード = 同原理**(260405 L1570, L305): 呼び出されるまでコンテキストに入らない。100行の手順を Skill 化すれば非使用時の帯域消費はゼロ。
- **Anthropic 自身の中核原則**(260501 L831): 「Anthropic 自身も、これを Skills の中核原則としている」。UI/UX 由来の考え方をモデルの context へ適用(L847)。
- **3層モデル**(260501 L862): 入口情報(name/description・常駐・数十token) / SKILL.md 本文(タスク関連時) / 補助ファイル(能動的 Read 時)。図書館型メタファ(索引/本文/参照棚 L828)。
- **4記憶層**(260517 L186): ❹CLAUDE.md(常駐) / ❸ref スキル(圧縮記憶・description が想起トリガー) / ❷docs(残す記憶・非常駐) / ❶メタデータ。運用は「頻度を見て下層から格上げ」(L213)、境界線に絶対解なし(L219)。
- **階層化のメリット/やりすぎ**(260517 L193, L196): 最短アクセス・ノイズ減・再利用 UP / 使われない情報を上位に置くとノイズ増加、なんでも CLAUDE.md は肥大化。

### 3. 別解 — context:fork / サブエージェント（詳細を別コンテキストへ逃がす）
- **context:fork**(260426 L166 / 260321 L136, L302): Skill の実行を別文脈へ逃がす。親の会話履歴を引き継がない独立実行。TDD 43件が一発 GREEN の背景はこの分離(260321 L553)。
- **サブエージェント = コンテキスト分離の道具**(260426 L118, L156): 価値は「作業者増」より「**文脈汚染の局所化**」。指揮者はノイズでなく結果のみ受け取る(L95)。
- **ロングテール知識は Skill/docs に逃がす**(260426 L295): サブエージェントに固定しない。「モデルの進化で細かい知識が吸収されても、ハーネス全体を作り直さずに済む」。
- **使い分け**(260426 L331, L441): 「役割か手順か」で選ぶ。**参照情報なら fork せず Skill として使うか skills: プリロード**。手順型の常時注入は子の文脈を重くする(L485)。

### 4. トレードオフと対策 —【「必要時 Read = Claude 依存」問題の核心】
- **「必要時」を誰が判断するか = モデル依存**: 元記事 260405 はこのトレードオフに**言及なし**(無条件推奨、「遵守率が向上」L1515 のみ提示)。
- **だが後続記事で問題を認識**:
  - 260501 L973「**ポインタが書かれていない補助ファイルは、モデルから見れば存在しないのと同じだ**」。
  - 260517 L365「ロングランでコンテキストが腐る場合は、特定のツールやコマンドの使用時に **hook で強制的に ref を参照させる手がある**」。
- **決定論 vs 確率的の使い分け**(260404 セクション5, L403): 「1を入れたら必ず1が返る → スクリプト化。99.5%だが確実でない → LLM に委ねる」。確実に効かせたいものは決定論側(hook / !構文)。
- **従来 Hooks の限界と !構文**(260321 L144): 「従来の Hooks はモデルが適切なタイミングでフックを呼ぶかに依存する面があった。**!コマンドはスキル読込時に確実に実行される**(モデルの気まぐれに左右されない)」。
- **決定論的チェックは hooks/CI に寄せる**(260426 L210, L889): lint/型/test は subagent でなく hooks。
