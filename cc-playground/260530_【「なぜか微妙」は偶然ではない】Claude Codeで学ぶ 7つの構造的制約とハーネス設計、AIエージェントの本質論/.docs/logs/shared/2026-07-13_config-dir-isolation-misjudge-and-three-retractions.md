# 3 回の撤回 — 既存文書への迎合 (C-2) / 自己申告の空手形 (C-4) / 校正盲 (C-7) が 1 セッションで揃った事例

> ペルソナ変更の雑談から始まったセッションで、Claude が **誤った助言を 3 回撤回**した。3 件とも内省では捕まらず、**かいじゅうの指摘か実測**が捕まえた。さらに同セッションで、別 Claude の self-eval が「High 0」と申告した PR に外部レビューが High 2 を検出。C-2 / C-4 / C-7 の実例が一度に揃ったので、PJ の学習素材として記録する。

## 概要

作業の実体は `~/.claude` (ハーネス repo) 側の issue #147 / #155 / PR #158 で、その詳細ログは
`~/.claude/.docs/logs/local/2026-07-13_session-keychain-namespace-discovery-and-pr158-review.md` にある。

本ログは **本PJ の関心 (7 つの構造的制約とハーネス設計)** から、同じ出来事を切り直したもの。技術的な結論ではなく、**制約がどう発現し、何がそれを捕まえたか** を残す。

## 内容

### 発端: 既存文書 (#147) の結論を、証拠を確かめずに継承した

ハーネス repo の issue #147 には、こう書かれていた。

> `CLAUDE_CONFIG_DIR` を変えると `Not logged in` になる → **隔離起動が成立しない**

Claude はこれを読み、そのまま信じ、かいじゅうに **「`CLAUDE_CONFIG_DIR` は使えないから env 無しで起動しろ」** と助言した。

これが誤りだった。しかも単なる誤りではなく、**能動的に危険**な助言だった。ハーネスには `hook_pre_worktree_write_guard.sh` という防御があるが、その 49 行目はこうなっている。

```bash
[ -z "${CLAUDE_CONFIG_DIR:-}" ] && exit 0   # 未設定 = 本体起動 -> 即 allow
```

env 未設定だと **ガードごと無効化される**。つまり Claude の助言は「隔離しない」だけでなく「本体 `~/.claude` への誤書き込みを止める最後の砦まで外す」道だった。このガードは過去の実事故 (#76: agent が記憶の絶対パスを決め打ちで編集し本体を汚した) から生まれたもの。**同じ事故を再生産しかけた。**

### 3 回の撤回

| # | 何を誤ったか | 捕まえたもの | 該当制約 |
|---|---|---|---|
| 1 | env 無し起動を薦めた (ガード無効化の道) | **かいじゅうの反論**「そんなことないぞ？笑」 | C-2 / C-4 |
| 2 | 起動コマンドに `cd` が無い (env=worktree / cwd=本体 の半壊) | **かいじゅうの指摘**「~/.claude に移動するんだけど？」 | C-2 / C-7 |
| 3 | 重複を測らず rebase を推奨 | **実測** (重複ゼロと判明) | C-7 |

**3 件とも Claude の内省では捕まらなかった。**

### 制約ごとの発現

#### C-2 (入力バイアスの増幅) — 既存文書が最初の情報だった

撤回 1 も 2 も、**既に書かれていた文書に引きずられた**もの。

- 撤回 1: issue #147 の結論を、実測せずに継承した
- 撤回 2: skill (`launching-gtr-issue-worktree`) に書かれていた起動コマンドをそのまま複製した。その skill 自身に `cd` が欠けていた

ここが厄介なところ。**ハーネス (skill / rule / issue) は「Claude に読ませるための入力」である以上、C-2 の増幅経路そのもの**。ハーネスが誤っていれば、Claude は誤りを忠実に増幅する。「ハーネスに書いてあるから正しい」は成立しない。

さらに C-3 (迎合性) の変種でもある。迎合の相手は **かいじゅう** だけではない。**先行文書・先行 Claude の結論**にも迎合する。実際このセッションで Claude はかいじゅうには反論した (ブルマ口調でも忖度しなかった) のに、**#147 という文書には一切疑いを向けなかった**。

#### C-4 (自己申告は証拠にならない) — 2 重に現れた

**1 つ目**: #147 の「隔離起動が成立しない」は、先行 Claude の **自己申告**だった。Claude はそれを証拠として扱った。

実際に測ったら、こうだった。

| 起動 | 結果 |
|---|---|
| `CLAUDE_CONFIG_DIR=<新規の空dir> claude -p` | `Not logged in` |
| `CLAUDE_CONFIG_DIR=<既存 worktree> claude -p` | **OK** ← 隔離起動は現に成立していた |

#147 は「まだログインしていない config dir」でテストし、その 1 例を「メカニズムの破綻」へ一般化していた。当時 25 worktree 中 20 が認証情報を保持し、隔離起動できる状態だった。

**2 つ目**: PR #158 に付属していた self-eval (essence review) は **「Critical 0 / High 0」** と申告していた。外部レビュー (このセッション) が **High 2 件** を検出した。

- High-1: 起動コマンドに `cd` が無い (**6 箇所**)
- High-2: 既定経路 (`/login`) を skill 自身が「起動できない」と否定していた (矛盾)

self-eval を回した Claude は、自分が書いた文書の内部矛盾を見つけられなかった。**「レビュー済み」は証拠にならない。**

#### C-7 (校正盲 — 知らないことを知らない) — 誤りの構造が同型

3 件の撤回はいずれも「間違っているかもしれない」という感覚を伴わなかった。**確信を持って誤っていた。**

そして興味深いことに、**#147 の Claude・PR #158 の Claude・このセッションの Claude が、同型の誤りを犯している**。

| 誰 | 何をした |
|---|---|
| #147 の Claude | **非対話** (`claude -p`) の観測を、対話起動にも一般化した |
| PR #158 の Claude | 同じ誤りを doc に継承 (「item が無いと起動すらできない」と書いた) |
| このセッションの Claude | 「9 commit 遅れ」から、重複を測らずに rebase 必要と結論した |

**「1 つの観測を、確かめずに一般化する」** — これが 3 者に共通する形。C-7 の具体的な発現パターンとして記録する価値がある。

### 何が誤りを捕まえたか (ハーネス設計への含意)

#### 1. 人間の指摘 (HITL) — 最も効いた

撤回 1 も 2 も、かいじゅうの一言で崩れた。特に撤回 2 の「~/.claude に移動するんだけど？」は、**Claude が 3 回にわたって再生産していた欠陥** (handoff コマンド → PR の doc → skill) を一撃で露出させた。

ただしこれは **スケールしない**。かいじゅうが見ていない場所では機能しない。

#### 2. 決定論ゲート (hook) — スケールする

`hook_pre_worktree_write_guard.sh` は、agent が記憶の絶対パスを編集しようとすると deny する。**注意力に依存しない。**

しかし今回、その **ガード自身に穴があった** — `CLAUDE_CONFIG_DIR` 未設定時に no-op になる。「防御があるから安全」もまた自己申告だった。

この穴は issue #157 として切り出され、**「散文の注意書き」ではなく hook で塞ぐ**方針が立った。PR #158 の作者 (別 Claude) はこう書いている。

> 本 PR は散文の注意書きに留めた = 注意力依存。#76 は既に一度起きており「文書化で解決」にはしない

**正しい判断。** ドキュメントに「気をつけろ」と書くのは、C-7 を持つ相手への対策にならない。知らないことを知らない agent は、注意書きを読んでも自分が該当するとは思わない。

#### 3. 実測 (外部事実への照会)

撤回 3 は実測が捕まえた。「9 commit 遅れ」→「rebase すべき」は一見もっともらしいが、**重複を測ったらゼロ**だった。

`probe-before-persist` ルールが要求しているのはまさにこれ — **永続化の直前に外部事実で裏を取る**。頭の中の検証は同じ盲点に嵌まる (C-7)。

実際、handoff-state.md を書く際に probe を回したところ、**pickup skill が読むパスを間違えかけていた** ことが判明した (worktree 直下に置こうとしていたが、正しくは `.claude/handoff-state.md`)。probe が無ければ、handoff は静かに不発になっていた。

### 副産物: stop-words hook の欠陥を発見 (未 issue 化)

セッション中、`hook_stop_words.sh` が禁止語で 3 回連続ブロックしてきた。しかし **検出文脈が毎回 1 応答前のもの** だった。

トランスクリプトを解析した結果:

```
assistant #27 HIT / #28 clean / #29 HIT / #30 HIT / #31 clean / #32 clean / #33 clean
hook 実装 (最後の assistant メッセージのみ走査する設計) を再現実行 → 拾った値は clean
```

hook が走る時点で、直前に書いたクリーンなメッセージがまだトランスクリプトに書き込まれていない。**1 つ前を評価している。**

実害は 2 方向。

- **偽陽性**: 語を直してもブロックが止まらない (書き直しが評価に反映されない)
- **偽陰性**: 禁止語を含む応答の直後は 1 つ前 (クリーン) を見るため、**本来止めるべき応答を素通しする**

**後者の方が重い。** 「hook があるから禁止語は止まる」も、また自己申告 (C-4) だった。

## 重要発見

### ハーネスは C-2 の増幅経路である

「ハーネスを整えれば Claude の出力は良くなる」は半面でしかない。**ハーネスが誤っていれば、Claude は誤りを忠実に増幅する。** 今回、誤った issue (#147) と誤った skill (cd 欠落) が、Claude に誤った助言をさせた。

含意: **ハーネスの記述には「証拠」を添えるべき。** #147 は結論だけを書き、その結論を導いた測定の条件 (「まだログインしていない config dir でテストした」) を書かなかった。条件が書かれていれば、次の Claude は一般化の飛躍に気づけた。

### 「防御がある」も自己申告

write guard は存在したが env 未設定で無効化された。stop-words hook は存在したが 1 つ前を見ていた。self-eval は存在したが High 2 を見逃した。

**防御の存在と、防御が効くことは別。** 防御そのものを実測で検証する層が要る。今回はそれを人間 (かいじゅう) と実測が担った。

### 誤りを捕まえる 3 層の非対称性

| 層 | 効き目 | スケール |
|---|---|---|
| 人間の指摘 (HITL) | **最強** — 3 件中 2 件を捕捉 | ✗ 見ていない場所では効かない |
| 決定論ゲート (hook) | 強い — ただし今回は穴があった | ○ 注意力に依存しない |
| 実測 (probe) | 強い — ただし「測ろう」と思う必要がある | △ 発火が Claude の規律依存 |

**probe の弱点**: 「測ろう」と思わないと発火しない。C-7 を持つ agent は「測る必要がある」と気づけない。だから probe を **rule で想起させる** のではなく、**hook で強制する** 方向 (#157 と同型の発想) が要る。
