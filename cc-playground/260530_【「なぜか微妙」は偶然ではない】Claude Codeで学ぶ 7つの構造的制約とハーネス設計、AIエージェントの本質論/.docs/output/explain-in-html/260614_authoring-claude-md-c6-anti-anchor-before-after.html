01背景 — anti-anchor とは何か

大規模言語モデル (LLM) は、明示しないと訓練データで頻出する「主流」の選択肢を選ぶ。テストなら Jest、HTTP クライアントなら axios、型を書かなければ素の JavaScript。これは note の構造的制約 C-6「訓練データの断崖」そのものじゃ。一度出力された主流名は アンカー (思い込みの錨) として後続の生成を引っ張り続ける (C-2 アンカリング)。

対策が anti-anchor。非主流のライブラリを採用する時、主流側を Vitest (Jest ではない) のように 明示的に排除して書く。「Vitest」と書くだけでは弱く、「Jest ではない」と添えて初めて訓練データ由来のアンカーを上書きできる。

調査 (/review-agent-essence) の結果、この手法は CLAUDE.md 作成 skill に まったく反映されていなかった。該当語句の grep は 0 件。それどころか「良いサンプル」が Vitest を裸で列挙し、手法未適用の反例がお手本として同梱されていた。本 PR でこの穴を後方互換に埋めた。

anti-anchor = 非主流を選ぶ時、主流側を「X ではない」と明示排除して、LLM の訓練データ由来の思い込み (アンカー) を打ち消す書き方。「何を使うか」だけでなく「何を使わないか」を書くことが、生成の方向を確定させる。

04適用ガード — 「足しすぎ」を防ぐ仕掛け

手法を無条件に入れると、別の原則を壊す。レビューが指摘した 2 つの落とし穴に、ガードを最初から組み込んだ。

  • 非主流かつ実害のあるカテゴリのみに付ける — 全行に ではない を付けると行数が膨張し、原則3「Less is More (200行以下)」と正面衝突する。だから sample でも 1 箇所だけに留めた。
  • 対抗馬名は最小限、判断理由を主にする — 「Jest」という名前そのものより「訓練データ偏りの上書き」という理由を中心に置く。名前は従。
  • 主流判定の腐敗を避ける — 何が主流かはモデルの更新で逆転しうる (状態管理の主流の変遷など)。対抗馬名をハードコードして依存しすぎない。

06教訓 — 前回はなぜ転んだか

この再実装はやり直しじゃ。最初の試みは静かに失敗していた。両者の違いが、この作業の本質的な学びになる。

前回検証ゼロで「完了」と自己申告した

何が起きたかstatus を completed にして archived (完了済み置き場) に移動した。だが実際は 編集 4・5 が未実施、編集 3 には -->() という異物の欠陥が混入していた。検証 (スクリプト実行・再 grep・検収チェック) を一切通さなかったため、未完成を見逃した。
正体構造的制約 C-4「自己申告は完了の証拠にならない」の違反。「やったつもり」が「やった」に化けた。

今回検証を通してから PR、completed は merge 後に回した

何を変えたか各検証を実コマンドで実証 → ブランチで commit → PR #1 作成、の順に固定。completed は merge されるまで付けない。実装が終わった ≠ 取り込まれた、を status で区別する。
効いた仕掛け編集 5 の Check 6 がまさにこの C-4 対策。生成物に手法が入っているかを機械が見るので、人間やエージェントの「言ったつもり」に依存しない。

07総評

この PR の価値は「anti-anchor 手法を足したこと」だけではない。足し方そのものにある。過剰適用を防ぐガードを手法と同時に設計し、終了コードを変えない WARN で機械検証を相乗りさせ、検証を通すまで完了を宣言しない。手法の追加が別の原則を壊さないことを、実コマンドで証明してから人の判断 (merge) に渡す — これが「動く」と「最善」の差じゃ。
