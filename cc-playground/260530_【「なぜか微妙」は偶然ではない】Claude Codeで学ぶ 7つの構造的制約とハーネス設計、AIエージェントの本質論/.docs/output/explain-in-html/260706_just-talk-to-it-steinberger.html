02中核主張 — Just Talk To It

論の背骨は「モデルは十分に賢くなったのだから、その賢さを回避する茶番 (charade) をやめろ」。plan mode、spec駆動、subagent、RAG (検索拡張生成 = 外部知識を検索して回答に反映する仕組み)、無数のラッパーツール — これらは多くの場合、モデルの非効率を回避するための遠回りだと断じる。代わりに推すのは直接対話だ。「let's discuss」「give me options」と一言添えれば、codex はリポジトリのファイルを読み込んで慎重に待つ。専用の仕組みは要らない、と。

Don't waste your time on stuff like RAG, subagents, Agents 2.0 or other things that are mostly just charade. Just talk to it. Play with it. Develop intuition.

捨てるもの理由代わりに

04実践のコツ

  • Blast Radius (爆発半径) — 変更が触るファイル数・時間を先に見積もる。小さな爆弾を多数投げれば atomic commit (1変更=1コミット) と巻き戻しが容易。長引いたら escape キーで「what's the status」と現状確認。
  • 短いプロンプト + 画像 — codex では1〜2文で足りる。プロンプトの半分以上にスクリーンショットを添える。モデルは画像内の文字列を見つけ該当箇所へ直行する。
  • continue を予約投入 — 長い作業は「continue」をキューに積んで放置。完了後の余分なメッセージは無害に無視される。
  • feature の後にテストを書かせる — 同じコンテキストのまま依頼すると良いテストになり、実装のバグも露見しやすい。
  • trigger words (誘い水) — 難所では「take your time」「comprehensive」「read all code that could be related」「create possible hypothesis」を添えると解ける。
  • tmux で背景実行 — codex に背景タスク管理が無いので「run via tmux」と言うだけ。専用の agent 定義は不要。
