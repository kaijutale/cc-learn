02何が壊れていたか (根因)

Claude Code の事前チェック (PreToolUse hook) には「保護パス (~/.claude/settings や ~/.claude/hooks 等) への Bash 経由書き込みを止める」ルールがある。この判定に使う正規表現が、書き込んでいない読み取り専用コマンドまで誤って拒否 (over-block) していた。Phase A の検証層を作っている最中に、メインの Claude 自身が実際に踏んで発覚した — 既存の監査60件には含まれない新規バグである。

核心は [^|]* という部分。これは「パイプ記号 | 以外なら空白も含めて貪欲に飲み込む」意味になる。結果、2>/dev/null の > (エラー出力の捨て先) を起点に、行のずっと後ろにある別コマンドの引数 .claude/hooks (これは cat など読み取りの対象) までを一続きとして橋渡しし、マッチさせていた。実際のリダイレクト先は /dev/null なのに、「> と .claude/hooks が同じ行に同居している」だけで発火していた。

05レビュー (harness-essentials-reviewer 2巡)

HIGH → 解消「二次防御だから許容」は事実誤認だった

指摘当初 tee の見逃し (tee a ~/.claude/settings.json の2番目以降が保護パス) を「本丸は permissions.deny だから二次防御として許容」と整理していた。これが誤り — permissions.deny は Edit / Write ツールしか止めず、Bash 経由の redirect / tee / cp 書き込みはこの hook が壁。
