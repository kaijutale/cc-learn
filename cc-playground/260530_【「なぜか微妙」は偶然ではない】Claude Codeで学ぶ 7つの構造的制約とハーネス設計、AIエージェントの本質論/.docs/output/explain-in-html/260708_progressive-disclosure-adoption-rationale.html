03かいじゅうの疑問への回答 — 「必要時 Read は誰の判断か」

「必要なときだけ Read」の必要なときを判断するのはモデル自身 —— つまり確率的で、読み損ねれば詳細ルールは効かぬ。これは重要なトレードオフじゃが、記事群を横断すると答えが見えてくる。

盲点元記事はトレードオフに言及していない

対策1読まれないポインタは「存在しない」と自覚する

対策2確実に効かせたいものは hook で強制参照

対策3決定論 vs 確率的の線引き

260404「1を入れて必ず1が返る → スクリプト化。確実でない → LLM に委ねる」。確実に効かせたい核は決定論(hook / !構文)側へ。

04別解 — context:fork(別コンテキストに逃がす)

詳細を外に出すもう一つの手が context:fork(親の会話履歴を引き継がない独立コンテキストで実行する仕組み、260426「コンテキスト分離の道具」)。ただし用途が違う —— fork は重い実行の分離、Progressive Disclosure は参照情報の遅延ロードじゃ。記事は「役割か手順か」で選べと言う(参照情報なら fork せず Skill 参照が正しい)。

05採用結論と「仕分け」

  • 採用は Yes — 帯域ゼロサムへの正解。だが成否は「仕分け」に懸かる。
  • 確実に効かせたい核(人格・応答契約・安全規約) → 本文に死守、または hook で決定論化。
  • 読み損ねても致命的でない詳細(手順・規約) → `.docs/` へポインタ化。
  • 既存の設計判断と完全一致 — わらわらが決めた「憲法 verbatim / Tools 安全2行 inline / 詳細 .docs/ / 必要なら hook」を、記事群が外側から裏書きした。
