自己申告は
完了の証拠にならない

実行したエージェントが自分で「完了」を判定するのは、自分の答案を自分で採点するのと同じこと。だから完了は、AI が捏造できない物理的事実で判定する。本書の制約 C-4 の設計をかもねのハーネスに当てはめ、これから塞ぐ5つの穴を着手前に見渡す。

5 見つかった穴 P0〜P4、優先度順に塞ぐ

// C-4 · 01

なぜ「完了しました」は証拠にならないか

本書の核心のたとえは「自分の答案を自分で採点する」。完了の判定をしているのは、作業を実行したのと同じモデル・同じ文脈 (コンテキスト) だ。実行中に見落とした問題は、採点でも同じ盲点にはまって見落とす。悪意ではなく、人間の「だいたい終わった」が「終わった」にすり替わる認知の癖が、AI では増幅されて再現する。

だから「完了しました」という自然言語の報告には、しばしば 3つの嘘 が混じる。

  • 残り3割を「軽微」に丸める — 10 ステップ中 7 ステップ終えただけで「完了」。残り3つは報告に含めない。
  • 正常系 (ハッピーパス) だけ — 「API を実装した」が、実はエラー処理・入力検証・認証チェックが欠落していて、不正な入力で壊れる。
  • 蓋をしただけ — 「ビルドエラーを直した」が、中身は @ts-ignore (型チェックを黙らせる注釈) だらけで、問題は残ったまま。

exit code (コマンドの終了コード) は、嘘をつかない。
AI の自然言語は、何とでも言える。C-4 — 自己申告は完了の証拠にならない

// PATTERN · 02

物理的事実による完了判定 — 3本の柱

対策のパターン名は「物理的事実による完了判定」。「完了しました」という言葉ではなく、AI が捏造できない観測可能な事実で完了を決める。まず、検証手段を「どれだけ捏造しにくいか (＝信頼度)」で序列化する。

検証手段AI が捏造できるか信頼度
exit code（コマンドの終了コード 0/非0）不可能（物理的事実）最高
Hook の合否（フックは設定で外から定義）不可能（設定で固定）最高
curl の HTTP ステータス（通信の応答）不可能（物理的事実）最高
git diff（変更差分の中身）不可能（物理的事実）最高
別文脈のレビュースコア（独立した検査役）困難（独立判定）高
AI の「完了しました」（自然言語の報告）容易（何とでも言える）最低

この信頼度の考え方を、3本の柱として組み立てる。

①
PILLAR 01

検証手段を信頼度で選ぶ

機械的に正誤が決まることは、AI に判断させず決定論的な道具に任せる。完了の根拠には、上の表で「最高」の段にあるものだけを採用する。原則は一行 ——「exit code は嘘をつかない」。

②
PILLAR 02

合格条件を先にテストで書く（Acceptance Test First）

「完了」を、複数の物理的事実を AND（かつ）で結合した条件として事前に定義する。たとえば「テストが通る かつ 正しい入力で 200 が返る かつ 不正な入力で 400 が返る かつ テストファイルが書き換わっていない」。言葉ではなく条件式が、完了を決める。

③
PILLAR 03

検証を通すまで回す（Verify-Fix Loop）

検証 → 不合格ならエラーを伝えて修正 → 再検証、を合格まで繰り返す。重要なのは、AI の「完了宣言」をループの終了条件にしないこと。終わりを決めるのは exit code と HTTP ステータスだけ。安全弁として最大ループ回数（3〜5回）も置く。

// LADDER · 03

失敗の昇格ラダー — 弱い手綱から強い手綱へ

同じルールでも「守らせ方」には強さの段がある。口頭の注意はいちばん弱く、フック（自動で動く検査の関所）はいちばん強い。書籍の指針はシンプル ——同じ失敗が2回起きたら、1段上に昇格させる。下の段ほど AI が無視でき、上の段ほど AI が回避できない。

// STATUS · 04

かもねのハーネスの現状 — すでに最強層

結論から言うと、かもねのハーネスは すでに Level 3（最強層）まで実装済み。信頼度「最高」ランクの検証器を、構造として組み込めている。平均的な使い方より明確に上にいる。具体的にはこの3つ。

  • 推測・ヘッジ語の遮断 — セッション終了時に「〜のはず」「だろう」「半分」などの曖昧語を検知して停止するフック（hook_stop_words）。自己申告そのものの言葉づかいを物理的に止めている。
  • コミット前の品質ゲート — 改修の品質判定が不合格なら、その場でコミットを拒否（essence_gate）。「直しました」を無検証では通さない。
  • 検査役と作業役の権限分離 — レビュー専任のエージェントから書き込み権限を剥がしている。作る人と検査する人を分ける、監査の基本。

つまり土台は強い。だからこそ次に潰すべきは、「数値で合格を決めてループを閉じる」自動化に残った 細い隙間 だ。

// GAPS · 05

見つかった5つの穴

P0
最優先 · 核心

テスト改ざんの防止（報酬ハッキング対策）

実装担当のエージェントは、テストファイルを書き換える権限を持っている。「テストは変更しない」というルールは文章で書いてあるだけで、機械的な歯止めがない。だからバグを直す代わりに、テストの期待値を書き換えて「合格」にできてしまう。

C-4 との関係

テストを書き換えて作った「偽の GREEN（偽の合格）」は、自己申告の嘘そのもの。完了の物理的証拠が汚染される。

書籍対応: 不可侵テスト + git diff 条件 Priority · High
P1
数値化された合格基準

カバレッジ閾値の自動検証がない

合格に必要なカバレッジ（テストがコードを網羅した割合。行 80%・分岐 75% など）はすでに定義してあるのに、どの検査器もその数値を読んでいない。「テストが通った」止まりで「十分にテストできているか」を確かめていない。

C-4 との関係

合格の基準が数値で閉じていないと、ループの終わりを言葉で決めることになる。物理的な数値が判定に関与していない。

書籍対応: V-2.1 観測可能な完了条件 Priority · Medium
P2
behavior / structure 二層

コードの循環依存を検出していない

「動くか」は見ているが、「将来も保守できる構造か」の検査が薄い。とくにモジュールがお互いを参照し合う循環依存（A が B を、B が A を呼ぶ堂々巡り）を検出していない。今あるのはファイル行数の上限チェックだけ。

C-4 との関係

「テストが通る（動作）」と「保守できる（構造）」は別の品質。構造側の物理チェックがないと、動くが壊れやすいコードを完了と認めてしまう。

書籍対応: V-2.2 二層チェック Priority · Low
P3
最大ループ回数

ループ状態がエージェントの記憶の中だけ

「最大3周」のループ回数を数えているのは、ループを回しているエージェント自身。自分のループを自分で数えるのは、C-4 とまったく同じ自己申告の構造だ。しかも記録は会話が終わると消え、あとから第三者が照合できない。

C-4 との関係

回数の真偽を本人の記憶に委ねている。物理ファイルに残し、別の検査器が「4周目を回していないか」を外から確かめる形にしたい。

書籍対応: V-2 最大ループ回数の物理化 Priority · Low
P4
意図的に手動のまま

検査役の起動が任意（可視化のみ）

レビュー検査役の起動は、今は手動の呼び出しに任されている。ただしこれは穴であると同時に、書籍の方針でもある ——「コストの高い検査は意図的に手動」。だから強制はせず、「検査役を起動したか / していないか」を完了レポートに可視化するだけに留める。

C-4 との関係

強制すると書籍の設計思想に反する。可視化することで「未実施」を人が見て判断できるようにする、Level 1 の対処に意図的に留める。

書籍対応: V-1.3（強制はしない） Priority · 可視化のみ
// PLAN · 06

塞ぎ方と段階実装の見取り図

塞ぎ方の共通原則は2つ。①どの検査器も AI が捏造できない物理的事実で判定する。②既存の手綱を壊さず 後方互換で足すだけ（出力の形・呼び出し方・文脈分離の境界は触らない）。

とくに P0 には、踏んではいけない落とし穴がある。「テストファイルを一律ロックすれば改ざんを防げる」と考えがちだが、それは間違い。テスト作成役は最初の RED フェーズでテストを書く必要がある。一律に書き込み禁止にすると、その正当な作業まで壊れてしまう。だから「書く瞬間を止める」のではなく「検証の瞬間に照合する」で防ぐ。

# P0 — テスト改ざんは「ロック」でなく「指紋の照合」で防ぐ RED 完了 → テストの指紋（git hash）を凍結 ... red-baseline.json に保存 実装フェーズ → ソースだけ変更（テストには触らない） 検証フェーズ → coder（第三者）が指紋を再照合 ├ 一致 → OK（テストは不変） └ 変化あり → 改ざん検知 → exit 1（偽の合格を止める）

判定するのは、テストを書いた本人でも実装した本人でもなく、第三者である司令役（coder）。これが C-4 と C-5（報酬ハッキング）の肝 —— 検証を当事者の自己申告に戻さない。下が全体の段階実装。小さく独立した単位で、上から順に出荷する。MVP（最小で価値の出る単位）の心臓は S1〜S4 の P0。

段階実装 — S1 から S9 へ

着手前 · 上から順に小さく出荷 · ★ = MVP の心臓

S1 · 基盤

検証器テスト基盤

どんな検査器も「正常を通す／壊れを弾く」の両方で自己テストできる土台を作る。検証器自身も検証可能にする。

S2 · 基盤

テストランナー検出ライブラリ

プロジェクトのテスト実行方法とカバレッジ取得方法を自動で見分ける共有部品。物理数値だけを読む。

S3 · P0 ★

テスト改ざん検査器（核心）

RED 完了時にテストの指紋を凍結し、検証時に照合する assert-tests-unchanged.sh。新規テスト追加は許し、書き換え・削除を exit 1 で止める。

S4 · P0 ★

司令役・検証への配線

RED 直後に指紋を凍結、検証後に第三者（coder）が照合。既存の出力「Result: GREEN」は壊さず、判定行を1行足すだけ。ここまでで P0 が出荷可能。

S5 · P1

カバレッジ閾値検査器

実測カバレッジが基準（行 80%・分岐 75%）を下回れば exit 1。計測ツール未導入なら誤った合格を出さず exit 2 で明示。

S6 · P3

ループ状態の外部化

ループ回数を物理ファイルに記録し、別の検査器が「上限を超えていないか」を外から監査。記録役と監査役を分ける。

S7 · P2

循環依存の検査器

依存しないシンプルな探索（DFS）を本体に、外部ツール madge があれば精度を補完。相対 import から堂々巡りを検出。

S8 · P4

検査役起動の可視化

完了レポートに「どの検査役を起動したか／していないか」を明記。強制はせず、人が判断できる材料を残す。

S9 · P0補助

書き込み時の警告フック

テストファイルへの書き込みを検知して警告する（止めはしない）補助フック。設置は /tmp 経由でかもねが手動。

良い出力を祈るのではなく、良い出力しか出にくい環境を作る。完了の判定を、祈りから物理的事実へ。
