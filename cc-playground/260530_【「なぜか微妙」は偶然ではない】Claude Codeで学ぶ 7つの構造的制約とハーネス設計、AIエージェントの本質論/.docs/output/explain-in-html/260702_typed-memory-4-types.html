01なぜ記憶を「型」で分けるのか

Claude Code は毎セッションが「初対面」だ。前回の会話も発見も失敗も、明示的に書き残さなければ消える。これは能力不足ではなく、ステートレス(状態を保持しない。前回の続きを覚えていない性質)という構造的な制約の帰結である。

note K-1 はこれを「毎朝、記憶がリセットされる同僚」にたとえる。口頭で伝えても翌朝には消えるので、重要なことはすべて共有ドキュメントに書き、その同僚が毎朝読めるようにするしかない。必要なのは「思い出させる」ことではなく、何も知らない次のセッションが、メモだけで再開できる仕組みだ。

ステートレスを放置すると、次の4つが構造的に起きる:

  • 状態の揮発 — 判断理由・中間成果・進捗が、セッション終了とともに回収不能になる。
  • 暗黙知の蓄積不能 — 「この API は仕様と挙動が違う」といった発見が、次のセッションに伝わらない。
  • 同じ失敗の反復 — 地雷を踏んで回避策を見つけても、次のセッションがまた踏む。学習曲線がリセットされ続ける。
  • 引き継ぎの不在 — 長時間タスクが中断したとき、どこから再開すべきか分からない。

解決はプロンプトの改善ではなく、状態を外部に永続化するハーネス設計にある。そして記憶をただ放り込むのではなく「型」で分類することで、記憶の用途と寿命が明確になり、検索性と保守性が上がる。型がなければ、半年後に「これは何の教訓だったか」「まだ有効か」が分からなくなる。型は記憶の棚卸しと整理を可能にする ― これが K-1 の核心思想だ。

024型の記憶分類 ― 正本の定義

note K-1 は、Claude Code の Memory(記憶)システムが記憶を4つの型で分類すると説く。記事はこれを「4つの引き出し」のメタファーで図示する ― ファイルボックスの4段それぞれに型のカードが差さり、必要なとき正しい引き出しを開けて「あ、これだ」と取り出す絵だ。

type(型)用途寿命例(正本ママ)
userユーザーの属性・好み長期(ほぼ不変)「シニアエンジニア、Go が得意、React は初心者」
feedback実際の失敗からの教訓中期(解消まで)「テストは実 DB を使う。モックで事故った」
project進行中の状態短〜中期(タスク完了まで)「リリースフリーズ 3/5 開始」
reference外部リソースへのポインタ中期(リソースが有効な間)「バグは Linear の INGEST プロジェクトで追跡」

4型を貫く軸は「寿命」だ。user はほぼ不変の人物属性、feedback は問題が解消するまでの教訓、project はタスク完了で用済みになる進行状態、reference は外部リソースが有効な間だけ生きるポインタ(行き先を指す道しるべ)。寿命の違うものを同じ箱に入れると棚卸しできない。だから型で引き出しを分ける。

記事はさらに、この記憶を「覚えておく」だけで終わらせない進化サイクルを説く ― 失敗が起きたら注意喚起として書き、それでも再発したら lint(自動検査)や検証スクリプトへ昇格させ、自動で守られるようになったら元の覚え書きを消す。記憶の最終形態は「覚えていなくても守られる構造」だという思想だ。

03「型の定義」と「記憶の実体」は別の層

「4型は構築されているか」を問うには、まず2つの層を分けねばならない。混同すると答えが濁る。

  • スキーマ層(型の定義) — 「user / feedback / project / reference という型がそもそも定義されているか」。設計図の話。
  • 実体層(記憶のインスタンス) — 「各型の記憶ファイルが実際に何件あるか」。中身の話。

この2層は独立だ。型が定義されていても実体が0件はあり得るし、その逆もある。そなたのハーネスを両層で調べた結果、スキーマ層は存在するが「薄い」ことが分かった。4型は記憶テンプレート内の裸の見出しとしてしか定義されていない:

## user
## feedback
## project
## reference

出所は ~/.claude/skills/establishing-knowledge-persistence/SKILL.md(記憶の永続化を司るスキル)の MEMORY テンプレ(120-123 行目)と、review-harness(ハーネス診断スキル)の型別カウント欄だけ。つまりハーネスは4型の名前は知っているが、各型の「用途」「寿命」の意味付けはどこにも書かれていない ― その意味づけは正本(記事)の中にしか存在しない(スキル全体を 寿命 で検索して 0 件)。自動で型を検証するスクリプトも無い。スキーマは名札どまり、意味は人(そなた)の頭の中と記事にある ― これが実体を数える前提だ。

04kaiju ハーネスでの構築状況 ― 全数センサス

実体層を全数調査した。記憶は Claude Code 標準の記憶機構(native memory)として ~/.claude/projects/<プロジェクト名>/memory/ に、Markdown + frontmatter(ファイル冒頭のメタ情報欄。ここに type: を書く)の形式で格納されている。全記憶を横断した型付き実体の件数はこうだ:

USER 型
0 件 (空)
FEEDBACK 型
5 件
PROJECT 型
2 件
REFERENCE 型
1 件

型付きの native 記憶は全 8 件(feedback 5 + project 2 + reference 1)。加えて、型を持たない MEMORY.md(各プロジェクトの索引兼状態ファイル)が 6 件あり、うち 2 件は個別ファイルを作らずプロジェクトの進行状態を直接書き込んだもの(ninmu-ninmu と toybox、いずれも「休止中」の状態記録)。内容は project 型そのものだが、type: を持たないため型付き実体には数えない。グローバルな ~/.claude/memory/ は存在せず、記憶は 100% プロジェクト別に分散している。

型構築状況実体の置き場件数
feedback構築済native memory(型付き、2 プロジェクトに分散)5
project構築済native memory 型付き 2 + MEMORY.md 直書き状態 2(型無し)2 (+2)
reference構築済native memory(型付き)1 ファイル1
usermemory 層は空CLAUDE.md + ~/.claude/rules/ が代替(→ 06)0

運用は徹底して手動キュレーションだ。自動記憶蓄積の設定(autoMemoryEnabled)は false に倒してあり(settings.json)、第三者製の記憶プラグインも無効。記憶を自動で4型に振り分ける hook(イベント連動で自動実行される小さな処理)も、セッション開始時に記憶を自動注入する仕組みも存在しない。読み込みは必要なとき該当ファイルを参照する on-demand(必要時だけ読む)方式。hook が記憶に触れるのは「外部入力を記憶へ勝手に転記するな」という防御ガードと、「この hook はある feedback 記憶から生まれた」という出自コメントの2用途のみ ― 記憶を読む・書く配線は一切無い。

調査中に観測した軽微な綻び(本作業では観測のみ、是正はスコープ外):

  • frontmatter が 2 世代混在 — type: を最上位に置く旧型式 1 件と、metadata: の下に入れ子にする現行式 7 件が同居。素朴な grep "^type:" では 8 件中 1 件しか拾えない。
  • 索引の 1 件が迷子 — claude-code-learn の MEMORY.md が指すファイル 1 件が実在しない(残り 8 リンクは解決)。孤立ファイルは 0 件。

05構築済みの3型 ― 実体の証拠

feedback・project・reference の3型は、いずれも型付き native memory として実ファイルが存在する。中身を挙げる。

FEEDBACK · 5件実際の失敗からの教訓

grep-before-delete削除・移動・改名の前に必ず grep -r で参照を確認する。サイズや名前の類似だけで不要と判断するとパス参照が壊れる。
rules-always-loaded~/.claude/rules/*.md は paths 指定が無いと system prompt(エージェントへの指示文)に常時フルロードされる ― 遅延読み込みではない。
prefer-skill-over-rules文脈限定の知識は常時ロードの rules でなく on-demand の skill に持たせる。ハーネス設計では DRY(同じ記述を繰り返さない原則)が逆効果になる。
verify-source-via-pdf引用は必ず原典 PDF まで遡って裏取りする。コード内コメントや要約 grep だけで断定しない。
public-repo-pre-publish-scan公開前スキャンは秘密情報だけでなく、所属・来歴・内部構造の露出まで見る。

PROJECT · 2件(型付き) +2(直書き状態)進行中の状態

memopo-capture-memo-appそなたの個人開発アプリ memopo(Expo 製のキャプチャ型短文メモアプリ)の本質・技術スタック。
memopo-unified-editor「1 メモ = 1 統合エディタ。入口や種類で分割しない」という設計原則。
直書き状態 2件(ninmu-ninmu / toybox)個別の型ファイルを作らず、MEMORY.md に技術スタック・設計判断・「休止中」ステータスを直接記載するモノリシック(一枚岩)運用。内容は project 型だが型ラベルは持たない。

REFERENCE · 1件外部リソース・確定知識へのポインタ

permission-path-semanticssettings.json の権限パス解釈4種(~/=ホーム相対で効く / 単一スラッシュ=絶対で空振りの罠 / //=ファイルシステム絶対 / **/file=glob 一致)。実機検証済みの確定知識。

06user 型 ― memory 層は空、責務は別基盤が担う

結論から言う。user 型は memory 機構としては未構築だ(全プロジェクトの memory を全数走査して type: user は0件、user_ で始まるファイルも 0 件)。だが、その責務は欠落していない ― ~/.claude/CLAUDE.md と ~/.claude/rules/*.md が肩代わりしている。これは穴ではなく設計判断である。

USER · 0件memory には無い。だが責務は別基盤が担う

責務の所在user 型が記憶すべき「かいじゅう本人の恒久的な属性・好み・規範」は、常時ロードされる CLAUDE.md(persona・username・応答規約・禁止事項)と rules/*.md(9 本の行動規範)に手動配置されている。
なぜ memory に置かないかrules / CLAUDE.md は system prompt に常時ロードされる = 不変の user 知識の置き場として最適。memory の user 型に同じ内容を持たせると二重管理になり冗長。autoMemoryEnabled: false の手動運用と一体の判断だ。
意図的だと分かる証拠アーカイブ済みの計画書(2026-05-23)に、横断的な feedback_* 記憶をあえて rules / CLAUDE.md へ移設した記録が残る。memory に置かないのは手抜かりでなく、文書化された移設判断の結果だ。

記事の4型を「すべて memory ファイルとして作るべき」と字面で読めば、user 型のみ未充足だ。だが「機能が満たされていればよい」と本質で読めば、user 型は充足済み ― ただ置き場が memory ではなく、常時ロードの rules / CLAUDE.md なだけだ。寿命「長期(ほぼ不変)」の知識を、揮発しやすい中期記憶と同じ引き出しに入れず、最も確実に毎回読まれる場所へ置く。これは4型の「寿命で分ける」思想に、むしろ忠実な配置といえる。

user 型相当の知識が、実際どこにあるかの対応:

  • persona・一人称・二人称(ボア・ハンコック) — CLAUDE.md Persona 節
  • username(かいじゅう)・呼称 — CLAUDE.md Username 節
  • 応答規約・禁止事項・好み — CLAUDE.md Response / Prohibition 節
  • 恒久的な行動規範(9 本) — ~/.claude/rules/*.md(decisive-answers ほか、常時ロード)

07総評

4型を2層で診た。スキーマ層(型定義)は存在するが「名札」どまりで、用途・寿命の意味は正本にしか無い。実体層は feedback 5・project 2(+直書き状態 2)・reference 1 が型付きで構築済み、user 型のみ 0 件。だが user の空白は欠落ではなく、寿命「ほぼ不変」の知識を常時ロードの CLAUDE.md + rules に置く設計判断で、移設記録がその意図を裏づける。note K-1 の「型は記憶の棚卸しを可能にする」を、そなたのハーネスは「揮発する中期知識は型付き memory へ、不変の user 知識は常時ロードの rules へ」と層で分けて体現している。記事が説く記憶の最終形態 ―「覚えていなくても守られる構造」― に照らせば、自動蓄積を切り価値あるものだけを手で選ぶこの運用は思想の実装だが、薄いスキーマと 2 世代混在の frontmatter は「型検証の自動化」という次の昇格余地を残している。
