02横断的発見 — 3系統に共通する構造欠陥

最重大評価基準の「正本」が分裂している

review-agent-essence が同梱する評価基準 agent-essence.md (39項目) は 2026-04-05 から実質凍結。いっぽう essence協調一式が参照する正本 essence-docs は、更新装置 (proposing-essence-updates — 人間のレビューを経て原則を追記する仕組み) で成長し続けている。両者をつなぐ同期チェックは存在しない。

なぜ致命的か「誰も更新せず、誰も乖離を検知しない」基準でレビューし続けると、装置は静かに時代遅れの判定を下すようになる。しかも review-harness の25指標も凍結側の原則IDを参照しており、影響は2系統に及ぶ。

未配線迎合・アンカリング対策 — 道具はあるのに接続されていない

実証満点S判定の直後に、別の subagent が弱点を後出しで指摘した記録が残っている (見逃しの実例)。
皮肉な構図review-agent-essence は「迎合 (相手に合わせて甘い判定を出す傾向)」を評価基準として持ちながら、自分の手続きには適用していない。essence協調一式は git log — つまり著者の意図文言 — をレビュー担当へ能動的に渡しており、アンカリング (先に見た情報に判断が引きずられる現象) の源になっている。

分断自己改善ループが essence 系にしか無い

現状self-eval (装置自身の反省メモ) を永続化し、人間確認付きで装置へ反映する仕組み (accumulating-reviewer-feedback) は稼働している。ただし接続されているのは essence協調一式のみで、review-harness と review-agent-essence は運用から学ぶ経路を持たない。

04系統別の要点

要修正review-harness — 診断装置が2026年を知らない

確定バググレード定義が3ファイルで不一致。レポートのテンプレートだけ旧5段階 (E判定の行が無い) のまま、本体・HTML・OGP画像は6段階に更新済み。
カバレッジAgent Teams (複数エージェントの協調機能) / autoMemory (自動記憶) / sandbox (隔離実行) / 新しい hook イベントが、評価基準の全377行に一度も登場しない。
再現性同一ハーネスに対し 50点満点・48点満点・92% と、採点の物差し自体が実行のたびに変わった記録がある。機械的に検証される指標は25個中ほぼ0。
強み修正手順のコピペ可能なスニペットと優先度付けは高品質 — 強化で壊してはならない側。

要修正review-agent-essence — 評価基準が孤立凍結

最重大同梱の評価基準39項目が、更新ループの対象外かつ同期チェックの射程外。成長する正本と別々に管理される「二重帳簿」状態。
精度判定が ○/△/× のみで severity (重大度) が無く「どれから直すべきか」が出力から読めない。根拠に行番号を引用する義務も無い。
統合「エッセンスレビュー」という一般的な言葉は必ずこの旧系統に流れ、後発の3領域並列系統には決して届かない (呼び分けの片方向記述)。参照側には「11原則」という古い数字が3箇所残存 (実態は39項目)。

部分強化essence協調一式 — 最も整備されているが、精度に穴

構造は堅牢出力の形式は検証スクリプトと hook (コマンド実行の前後に自動で走る検査) が機械チェックしており、3系統で唯一 self-eval も稼働。
精度レビュー担当へ commit メッセージ (著者の意図文言) を渡すアンカリング源があり、担当間で重複した指摘の整理は人間頼み — 重大度の集計が二重計上で歪む余地。
鮮度ui 領域の原則だけ更新が止まっている (更新リマインダーは金曜起動時のみ発火する設計)。

05「質を落とさない」ための資産 — 壊してはいけない契約

本調査は欠落だけでなく、改修時に壊してはいけない配線 (契約) も明細化した。強化の絶対条件は「現状の質を落とさない」であり、このリストが全 issue 共通の実装チェックリストになる。

  • review-harness — 出力ファイルの命名規則 / グレード6段階とOGP画像の対応 / HTMLの配色 (本ページの生成装置 explain-in-html が同じ見た目を前提に依存) / 初心者向けスタートアップモードの分岐。
  • review-agent-essence — 標準出力の構造 (原則適用マトリクス〜次のアクション指示) は「不変」と明記された正式成果物。自動改修の禁止指定 (変更は人間主導のみ) も設計意図として尊重する。
  • essence協調一式 — 出力の yaml キー名・件数の併記形式は、検証スクリプトと commit 前 hook が機械的に読み取る契約。fork (分離実行) の境界や評価基準の置き場所パスを含め10項目を確定。

06ここからどうするか — 強化プランと issue

欠落は優先度 P0 (バグ・保守修正) 〜 P2 (カバレッジ・再現性・自己改善) に整理し、GitHub (kaijutale/claude-harness) へ11件を起票済み。実装は baseline (改修前の基準値) 取得 → 改修 → 同一対象へ再実行して比較 の順を必須とし、説明できないスコア変動はリグレッション (品質後退) として差し戻す。特に review-harness は採点が揺れる実証があるため、baseline は2回実行して「自然なブレ幅」を先に測る。

07総評
