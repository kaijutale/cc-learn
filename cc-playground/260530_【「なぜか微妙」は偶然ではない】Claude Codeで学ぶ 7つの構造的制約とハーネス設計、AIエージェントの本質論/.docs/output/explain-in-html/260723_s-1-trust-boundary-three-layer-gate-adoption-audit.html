01note S-1 が求めるもの

一行サマリー: Claude Code が触れるファイル・操作の範囲を意図的に制限する。たとえは「鍵付きの金庫と机の引き出し」── 信頼しているかどうかではなく、万が一のリスクを構造で抑える。守る相手は悪意ではなく 善意の間違い。

核心の論理 (→C-5): どれだけ精密なレビューを組んでも、生成エージェントがレビュー基準を書き換えられるなら検証は無意味。だから「正しい行動以外の経路を物理的に閉じる」── プロンプトのお願いは車線、deny はガードレールじゃ。3層はこう積む:

  • L1 宣言的禁止 — Permissions / settings.json。判断を介さない静的制御。例: Edit(.eslintrc*) → deny
  • L2 条件付き許可 — Hooks / PreToolUse。スクリプトが条件を評価し pass/fail。例: git push は main 以外のみ
  • L3 人間承認 — Permission Mode / 対話的確認。不可逆・外部影響のある操作に適用

03設計差の核心 — 報酬ハッキング2経路の閉じ方

事実テスト本体の deny = 0 件 (.test./.spec./__tests__ の grep 実測)。
理由oracle 自身の設計宣言: 「テストを書く瞬間 (RED) は止めない (一律 deny は RED を壊す) - 照合は verify 時」。ハーネスは TDD でエージェント自身がテストを書く (team-tester) ため、一律 deny は運用と両立しない。
代替予防でなく検出へ移設 ── RED 直後に git hash-object SHA を baseline 凍結し、verify 時に照合。SHA 変化=改ざんとして exit 1 (偽 GREEN 失敗扱い)。判定材料は捏造不能な物理量。射程は TDD の verify を通る場合のみ。

04記事超え点

  • テスト deny の意図的不採用 (予防→検出移設) — note の代表例をあえて外し、TDD (エージェントがテストを書く運用) と C-5 対策を両立。「書けるが、書き換えたら必ずバレる」構造。
  • L2 の字句照合層 — hook_pre_commands が「deny の届かない Bash 経路」(echo リダイレクト・tee・cp 等での保護ファイル書込) を字句で塞ぐ。Edit ツール経路=L1 / Bash 経路=L2 字句、の経路別多重化。
