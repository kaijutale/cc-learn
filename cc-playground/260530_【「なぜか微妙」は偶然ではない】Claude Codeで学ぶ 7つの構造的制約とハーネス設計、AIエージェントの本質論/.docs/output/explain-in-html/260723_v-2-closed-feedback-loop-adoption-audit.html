01note V-2 が求めるもの

V-2「フィードバックループを閉じる」の一行サマリー: チェックするだけでなく、結果を Claude Code に返して修正→再チェックのループを回す。たとえは「体重計に乗るだけ vs 食事を変える」。基本構造は「生成 → 検証 → 合格基準を満たすか判定 → 不合格ならエラー内容を注入して修正 → 再検証」。

この閉じたループには3つの必須要素がある:

  • ① 数値化された合格基準 — exit code 0 / lint エラー0件 / レビュースコア85点以上など。「いい感じになったら完了」ではループが終わらない。
  • ② 構造化されたフィードバック — 「コードに問題があります」ではなく「src/api.ts:42 - 型不一致」のように箇所・内容を明示し、LLM が何を直すか一意に決まる形式。
  • ③ 最大ループ回数 — 無限ループの安全弁。3-5回が妥当。上限に達したら人間にエスカレーションする。

アンチパターンは「開いたループ」— lint を走らせエラーを表示して終わり、検証結果が Claude に返らず修正ループが閉じていない状態。

04反例狩り: 開いたループ (scan §E の12候補を裁定)

note のアンチパターン「開いたループ(測るだけで行動を変えない)」を能動的に狩った。大半は 意図的に開いたループ — 機械では閉じず人間が閉じる設計で、gap ではなくトレードオフ:

  • git-sync reminder — 未commit/未push を検出→通知のみ。まさに「体重計に乗るだけ」だが、閉じるには auto-push が要り「push は人間のみ可」の安全規則に反する → 意図的に人間が閉じる。(本セッションで毎ターン発火し、push を handoff した)
  • self-eval / essence 提案系 — 永続化するが反映は HITL (accept/defer/dismiss)、自動昇格なし。essence 正本への自動 commit も禁止 (評価基準保護)。「昇格の価値があるか」は意味論判断ゆえ機械で閉じない。
  • coder coverage 判定 — 未達を検出してもテストを書けないため親へエスカレーション。スコープ境界で、エスカレーション自体が閉じ機構。

advisory 注入 (loop は注入されるが強制されない): hook_post_lint / hook_post_emoji_check は診断を additionalContext で返すが block しない。これは gap でなく severity 別の階層化 — 安全critical (essence/adr/secret/worktree) は deny で強制閉ループ、style-advisory は注入止まり。note の閉ループ二値を「強制 vs 助言」の勾配に精密化している。
