なぜ隔離しても コンフリクトは起きるのか

2026.07.21git & 隔離worktree / CLAUDE_CONFIG_DIR

01一行の答え

隔離とコンフリクトは、まったく別の話。 「隔離」は どのハーネスを読むか をセッションごとに分ける仕組み。「コンフリクト」は branch (枝分かれした作業の線) をどの時点から切ったか という git の履歴の問題。片方をやっても、もう片方は防げない。

今回コンフリクトが起きたのは、隔離が失敗したからではない。古い土台から branch を切ったから。隔離そのものは正しく効いていた。

02隔離がやること・やらないこと

今回は 2 つで隔離した — worktree (作業コピーを別フォルダに分ける仕組み) と CLAUDE_CONFIG_DIR (どの設定フォルダを読むかを指す環境変数)。この 2 つが分けるのは「実行環境」だけ。

ファイルの編集先分けるworktree = 別フォルダなので上書きし合わない

03今回なぜコンフリクトしたか

origin (共有サーバー側の本流) には、既に f9ce068 という変更があって、harness-worktree-isolation.md の同じ場所 (§1) を直していた。ところが手元 (local) の本流は、それを含む 7 個の変更を取り込んでいなかった (7 個ぶん古い状態)。

あたしはその 古い手元の本流 を土台にして branch を切った。だから f9ce068 の修正が土台に入っていない。そこで同じ §1 を直した — 同じ場所を、知らずに二重に直した。あとで合体すると git は「どっちの §1 が正しい?」を決められず、コンフリクトになる。

[origin の本流]  ●─●─●─●─●─●─●   ← f9ce068 が §1 を直した (最新)
                 |
                 |   7個 未取り込み (手元はここまで古い)
                 v
[手元の本流]     ●   ← この古い土台から branch を切った
                  \
                   ●   ← あたしの fix (同じ §1 を直した)

  合体 (merge) しようとすると → 同じ §1 が 2 通り → コンフリクト

04たとえで言うと

worktree で分けたのは「別々のキッチン」。ぶつからず料理できる。でも 2 人とも書き込むのは 同じ 1 冊のレシピ本 (git の履歴)。片方が先にレシピの 5 ページ目を直し、もう片方が古い写しの 5 ページ目を別々に直せば、本を 1 冊にまとめる時にどちらが正か決まらない。キッチンを分けても、レシピ本の同じページの奪い合いは防げない。

05どうすれば防げたか

  • branch を切る前に git pull — 手元の本流を origin に追いつかせてから枝分かれさせる。そうすれば f9ce068 が土台に入り、コンフリクトは起きない。
  • 二重作業にも気づけた — 最新を土台にしていれば「§1 は既に直っている」と読めた。あたしは古い写しを読んでいたので「まだ直っていない」と誤解して、もう一度直した。
  • 本当の根本原因 — doc の書き方ではなく、手元が 7 個古いまま作業したこと。修正は既に origin にあった。

06総評

隔離は「どのハーネスを読むか」= 実行環境を分ける仕組みで、今回も正しく効いていた。コンフリクトは「どの時点から枝分かれしたか」= git の履歴の問題で、隔離とは無関係。今回の原因は、origin より 7 個古い手元の本流から branch を切り、既に直っていた同じ場所を二重に直したこと。防ぐには、branch を切る前に最新を取り込む。
