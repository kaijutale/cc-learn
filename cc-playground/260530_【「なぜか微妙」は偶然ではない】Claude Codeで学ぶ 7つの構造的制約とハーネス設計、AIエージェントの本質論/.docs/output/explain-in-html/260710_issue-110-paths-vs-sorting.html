04なぜ「判断軸」が本体なのか

仮に #110 を「全 PD 候補に paths を一括で貼る」作業と捉えると、plan-workflow / research-phase にも paths を貼ってしまう。 だが paths はマッチするファイルを Read した時にしか発火せず、Write では発火しない (公式の既知制限)。 この 2 本は「これから新規ファイルを書け」と指示する rule ゆえ、paths を貼った瞬間、いちばん効いてほしい新規作成の直前で確実に不在になる。

つまり paths を貼れるか否かを分けるのは「その rule が Read 依存に耐えるか」という判断であり、 その判断を全 rule に適用する営みこそが #110 の本体じゃ。paths はその判断が Yes だった時に使う道具の 1 つにすぎない。 Tier A の 3 本ですら、glob の書き方 (build-test はコード全域へ拡大、harness はリポジトリ相対へ) という追加判断が要る — ここでも機械操作より判断が先に立つ。
