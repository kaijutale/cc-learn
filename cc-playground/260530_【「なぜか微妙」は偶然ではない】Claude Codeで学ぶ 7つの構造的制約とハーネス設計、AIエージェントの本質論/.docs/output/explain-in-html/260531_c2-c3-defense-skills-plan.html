// PART · 01

何を作るか

新しい skill(Claude に特定の仕事をさせる手順書)を2つ作る。どちらも単独で動く、独立した道具だよ。

①
blinding-review-prompt

レビュー対象を「目隠し」してから渡す道具

レビューを頼むとき、対象から 「作った人の意図・PRの説明文・期待される結論・著者名」を機械的に消して、純粋な変更内容(差分)と客観的な数字(ファイル数・追加削除行)だけに整形して渡す道具。

さらに変更の規模から 「最低 K 件は指摘を挙げよ」という下限を自動で計算して付ける。これで「特に問題ありません」という無難な空返事を構造的に封じる。

構成は SKILL.md(本体)+ blinding-rules.md(消す/残すリスト・K の計算式)の2ファイル。これは「対象を整形して中立な文章を返す」変換の道具なので、分身(後述の fork)は要らない。

③
detecting-framing-bias

同じ対象を「賛成係」と「反対係」で別々に評価する道具

1つの対象を、メリットを最大限に挙げる係と リスクを最大限に挙げる係に、context:fork(お互いの考えを見ない、まっさらな別作業空間)で同時に・独立して評価させる。その後、主導役の Claude が両方を突き合わせて最終判断する。

片方からしか論点が出なかったら「そっちのフレームの掘り起こしが甘い」というバイアスのサインとして、主導役が能動的に補う。いわゆる Devil's Advocate(あえて反対の立場で批判する役)を仕組み化したもの。

構成は本体 + 解説 + 賛成係 fork + 反対係 fork + 賛成係 agent + 反対係 agent の6ファイル。既存の llm-debate(役割で分ける議論の道具)と同じ作りを、2フレームに縮めた形。

①は入力の段階でアンカー(判断を引っ張る錨)を抜く。③は出力の段階で両極に振って相殺する。同じバイアスに、入口と出口の両方から手を打つ。 plan / 2つの skill の関係
// PART · 02

なぜ作るか

理由は2段ある。表の理由は「前回の点検で△だった2点を埋めるため」。奥の理由は「AI は影響されても自覚しない、という性質に対しては、説明を信じるのをやめて仕組みで防ぐしかないため」だよ。

前回 Anthropic の研究を解説したとき出てきた数字を思い出して — ヒントに従う率は99%超、なのにそれを白状する率は2%未満。AI は影響を受けても、受けた自覚がない。だから「理由を説明させて検証する」は効かない。残る手は2つだけ:

  • 入力からアンカーを消す → そもそも引っ張られる材料を与えない(①の発想)
  • 独立した複数文脈で突き合わせる → 1つの頭の自己点検に頼らない(③の発想)

そして「なぜ既存を改造せず新規で作るのか」。かもねのハーネスには10個のエージェントが共有して参照する評価基準ファイル(judging-review-severity)がある。ここに手を入れると10個全部に波及する。だから既存には一切触らず、新しい道具を横に足す方針(契約を壊さない安全側)を取った。

前回の点検結果 (C-2 対処の現状)

② 独立コンテキストで突き合わせ = ✅ 実装済 / ④ 全部OKを疑う = ✅ 実装済 / ③ フレーミング反転 = △(考え方のみ) / ① 目隠しレビュー = △(下限の強制が無い)。今回は △ の2つだけを埋める。

// PART · 03

既存ハーネスの何がどこが強化されるか

「原則は書いてあるが手で守るしかなかった」状態から、「道具を呼べば自動で守られる」状態に変わる。下の表が before → after の対応だよ。

どこが今まで (△)これから (✅)
フレーミング検証
(C-2)		 「順序やフレームを変えて複数回試せ」と原則が1行書いてあるだけ。実際にやるかは Claude の気分次第 ③ 賛成係/反対係を必ず別空間で走らせる仕組みになる。やり忘れが消える
迎合・空返事
(C-3)		 「問題ゼロは疑う」と心構えはあるが、無難な「問題なし」を構造的に止める手段が無い ① 指摘の下限 K を自動付与。K 未達の「問題なし」は再走査を要求する
アンカリング
(C-2)		 レビュアーに著者の意図や期待結論がそのまま渡り、無意識に引っ張られていた ① 著者情報を機械除去した中立プロンプトを生成。引っ張る材料を断つ
既存レビュー群
(gep / llm-debate)		 そのまま動くが、入力の中立化や両フレーム評価は手動で足すしかなかった ①③ 既存レビューの前段に挟める。本体は無改変のまま品質が底上げ

大事なのは 「確率的(やるかもしれない)」から「決定論的(必ずやる)」への格上げ。note記事でいう「祈るのではなく、良い出力しか出にくい環境を作る」を、C-2/C-3 の2点でさらに前に進める変更だよ。しかも①と③は合成できる — ①で目隠しした中立テキストを③の入力に流せば、「アンカー除去済み × 両フレーム評価済み」の二重防御パイプラインになる(ただし合成は任意、各々単独でも動く疎結合)。

// PART · 04

壊さない設計・導入順

この plan の安全性の核心は 「全部が新規追加で、既存には一切触らない」こと。共有資産(10エージェントが参照する評価基準)も、既存レビュー道具の入出力の約束事も、そのまま。だから片方を足しても連鎖崩壊しない。実装は小さくリスクの低い方から刻む。

段階導入の順序

リスク小 → 大、各ステップで実機検証を挟む

Step 1

① blinding を先に1本完成

小さい(2ファイル・分身なし)方を先に。実装初手で 環境変数の既定値展開 が許可設定で拒否されないか実機確認(拒否ならスクリプト退避に分岐)。

Step 2

① を実機検証

既知の著者名/説明文を持つ対象で、出力にそれらが含まれないこと・差分は残ること・K が規模通り変わることを確認。

Step 3

③ framing を agent→fork→本体の順で

賛成係/反対係の agent を先に作り、fork 単体で分身がちゃんと返るか(分身起動はグレーゾーン挙動)を検証してから本体で束ねる。

Step 4

仕上げと改善

各 skill 完成後に動作確認 + empirical-prompt-tuning(指示文をバイアスのない実行者に試して反復改善する道具)で磨く。完了後ログに残す。

原則を心構えで守る段階は終わり。賛成係と反対係を必ず立て、入力からは錨を抜く。バイアスを祈りでなく仕組みで潰す2本を、今から建てる。
