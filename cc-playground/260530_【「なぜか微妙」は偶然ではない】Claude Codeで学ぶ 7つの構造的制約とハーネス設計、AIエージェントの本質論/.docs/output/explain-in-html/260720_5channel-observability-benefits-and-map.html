01前提 — 5チャネル観測面設計とは

note K-2.3 の設計パターン。エージェントが世界を認識する経路は 5 つしかない、という整理よ: ファイルシステム (ファイルを読む) / コマンド (実行結果と終了コードを見る) / ログ (実行時の挙動記録を読む) / API (外部への問い合わせ結果を見る) / UI (画面を見る)。この 5 経路を「意識的に」設計せよ — 観測できないチャネルがあると、エージェントはそこで起きる異常に気づけない。たとえは車のダッシュボード: 燃料計が無ければ、ガス欠に気づくのはエンストしてからよ。

02Q1: 設計するとどんなメリットがあるか

  • 「たぶん大丈夫」が消える — 観測手段が無いエージェントは自己申告 (「コードを書いたので動くはずです」) しかできない。観測面があれば「200 が返り、ログに記録され、テストも緑」と実測で完了報告できる。報告の信頼性が根本から変わる。
  • 直す→確かめる、のループが成立する — 観測できないものは制御できない。修正の効果を観測できて初めて「修正→観測→再修正」の検証ループが回る。観測面は自動修復・自動検証の前提条件。
  • 沈黙死に気づける — 「配備した = 動いてる」は思い込み。アンタのハーネスは 07-16 に sandbox 設定202本が実は死んでいた等の「配備≠実効」を 3 クラス同日に踏んだ — それを拾えたのも、観測面 (実測・台帳) があったからよ。エンスト前に警告灯が点く。
  • 改修の判断材料が数字で届く — hook 発火台帳の月次集計が「rule X が今月 N 回発火 = 注意書きが効いてないサイン」を機械supply する。どこを直すべきかを勘でなく数字で決められる。
  • 後から他人 (未来の自分・別エージェント) が検証できる — 構造化された記録 (1行1JSON の台帳・証跡文書) が残るから、セッションを跨いだ引き継ぎや第三者レビューで「本当にそうだったか」を遡って確認できる。

04Q3: どのハーネス部品で機能するか

観測面は 1 つの部品じゃなく、「装置が作り、規律が使わせ、skill と subagent が消費する」3 層で機能してるわ。

装置hooks — 観測面を「作る」側

報告系 hook 3本セッション開始のたびに git 同期状況・hook 発火月次集計・認証情報の孤児を自動報告 (今セッションも「未push 1件」が発火→アンタが push して解消)
計装済み hook 20本発火のたびに台帳 (JSONL) へ 1 行追記する記録装置を内蔵
heartbeat観測センサー自体の死活を自己診断する二階の観測面

規律rules — 観測を「強制する」側

critical-thinking「状態変更は成功メッセージを信じず、独立確認 (コミットID・ファイル存在・件数) で着地を検証してから完了報告」— 観測面を使わない完了報告を禁止する
probe-before-persist知識を保存する直前に外部事実 (Read/ls/Bash) で裏を取れ、と hook が発火して要求する。さっき handoff の誤記 (「gitignore側」) を実際に捕まえた

消費者skills — 観測チャネルを「使う」側

verify / run変更後にアプリを実際に動かして挙動を観測する (テストだけで済ませない)
capturing-ui-evidenceUI チャネルの証跡化 (3視点スクショ + メタデータ)
review-harnessハーネス構成を 25 指標で診断 — 設定ファイルという観測面を読む
auditing-aio / auditing-nextjs-securitylive URL・ソースを実検査する監査系
pickup / handoff / logginggit 状態・作業状態を機械パース可能な形式 (frontmatter/JSONL) で残し、次セッションの観測面にする

検証者subagents — ツール付与 = 観測チャネルの割当

ここが面白いところ。検証系 subagent (team-auditor / team-tester / code-reviewer / 各 essentials-reviewer) には Read, Grep, Glob, Bash が付与されてる — これは note の 5 チャネルのうちファイルシステム + コマンド + ログをそのまま渡してるということ。逆に team-ui-designer や debater-pm には Bash が無い (Read, Grep, Glob のみ) = コマンド実行の観測チャネルを意図的に絞ってる。「subagent に与えるツール一覧」とは「そのエージェントが観測できる世界の定義」そのものなのよ。

05総評

5チャネル観測面設計のメリットは一言で「自己申告を実測に置き換えられること」— 完了報告の信頼性、検証ループの成立、沈黙死の検出、改修判断の数字化、セッション横断の検証可能性が全部ここから生まれる。かいじゅうのハーネスでは、hooks が観測面を作り (台帳658行/月・報告系3本・heartbeat)、rules が観測面の使用を強制し (独立確認・保存前裏取り)、skills と subagents が観測チャネルを消費する — という 3 層分業で実装されている。subagent へのツール付与が観測チャネルの割当そのものになっている点は、note の表をハーネス設計に翻訳した時の一番きれいな対応よ。
