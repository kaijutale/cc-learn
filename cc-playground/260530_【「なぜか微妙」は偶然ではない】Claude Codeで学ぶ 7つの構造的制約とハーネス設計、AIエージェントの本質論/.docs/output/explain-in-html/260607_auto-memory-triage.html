前提として 自動蓄積を OFF にした(Claude が学びを勝手に書き込まなくなる)。だが既に溜まった37本はそのまま残る。ここからは「蛇口を締めた後、溜まった水をどう仕分けるか」じゃ。

仕分けの軸は3つ + 保留が1つ:

  • rules移行 — 「常に効かせたい振る舞いルール」。memory の関連検索 (recall) に漏れると守られぬゆえ、毎セッション必ず読まれる ~/.claude/rules/ へ返す。
  • memory残置 — 「特定の作業でだけ要る技術ノート」「このプロジェクト専用の事実」「ツールの仕様」。常時持つ必要がなく、必要な時だけ呼び出す memory が正しい置き場。
  • 既存rules吸収 — 美学は frontend-aesthetics、egress 遮断などのセキュリティは security として、既にあるルールへ統合。
  • 要再判断 — 元の計画は rules 行きにしていたが、中身を見ると振る舞いではなく技術ノートで、rules に置くと常時ロードを汚すもの。ここが議論の本番じゃ。

要再判断 — 計画のまま進めると逆効果

元の計画 (plan) は、この9本も「振る舞いルールだから rules へ」と振り分けていた。だが中身を読むと、多くは 振る舞いの指示ではなく技術ノート ── 特定の作業をする時だけ要る具体知識じゃ。これを常時ロードの rules に置くと、先の収支検算で否定した「汚染を常時ロードへ引っ越す」になる。代表3つを解剖する。

A
巨大ノート

141行を rules に置く愚

multi-agent-debate-design(141行)を、元の計画は rules 行きに分類していた。だが rules は常時ロード ── 141行が 毎セッション必ず読み込まれる。

先の収支検算で否定した動き

「技術ノートを rules に移すと、特定作業でしか要らぬ知識が常時ロードを汚染する」。141行はその最たる例。memory の選択呼出(関連する時だけ読む)にこそ宿るべきじゃ。

暫定: memory 残置 または詳細遅延置き場へ 要再判断 ◆
B
仕様 か 振る舞い か

「skill の挙動」は法か、知識か

計画の判定

skill 作成グループとして rules移行。3本(skill-naming / disable-model-invocation / skill-self-reference)をまとめて。

中身の実態

「skill 選択は description 経由」等は客観的な仕様の事実。守らせる振る舞いではなく、skill を書く時だけ要る知識じゃ。

常に効かせたい振る舞いなら rules。作成時だけ要る仕様の事実なら memory。これらは後者寄りに見える。

暫定: memory 残置寄り(skill 作成時に呼出) 要再判断
C
手法 か ルール か

特定作業でだけ要る手順

計画が rules 行きにしていた、残る5本:

  • pnpm-workspace-isolation — pnpm 作業時のみ
  • tdd-red-stub-anti-pattern — TDD 実践時のみ
  • bash-oracle-authoring-traps — bash 検証器を書く時のみ
  • project-root-cwd-not-monorepo — パス解決の判断時
  • hooks-deny-install-via-tmp-cp — hook 設置時のみ
境界の問い

「その作業をする時だけ」要る具体ノウハウか、常に守らせたい汎用ルールか ── 1本ずつ性質が違う。hooks-deny や project-root は判断軸の色が濃く rules 寄り、pnpm や bash-oracle は技術ノート寄り。一括せず個別精査が要る。

暫定: 1本ずつ精査(残置 と rules が混在) 要再判断
