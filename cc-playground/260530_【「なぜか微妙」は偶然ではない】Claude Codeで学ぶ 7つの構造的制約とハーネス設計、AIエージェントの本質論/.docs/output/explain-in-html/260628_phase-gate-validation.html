01記事「Phase分割 + ゲート検証」とは

複雑なワークフローを番号付きの Phase(段階)に分割し、各 Phase の境界にゲート(機械的な合否チェック)を置く設計じゃ。ゲートを通過しなければ次の Phase に進めない。バグをその Phase で食い止め、後続への伝播を断つのが狙い。

02核心 — 乗算劣化の「分母を変える」

各ステップの成功率が 95% でも、長く繋げば全体の成功率は掛け算で落ちていく。ゲートを挟むと、この劣化の構造そのものを変えられる:

20ステップ 通し
0.95^20 ≈ 36%
1フェーズ(5ステップ)
0.95^5 ≈ 77%
フェーズ間の伝播
ゲートで ゼロ
実効成功率
大幅向上

20 ステップを一気に通すと全体 36%。だが 5 ステップ × 4 フェーズに割り、各フェーズの末尾でゲート検証してエラーをその場で修正すれば、フェーズ間にバグが伝播しない。失敗が積み上がらないぶん、実効成功率が跳ね上がる ── これが「分母を変える」の意味じゃ。

中核 1enforcing-strict-tdd-cycle / coder

VERIFY ゲート独立した検証担当(team-tester)がフレッシュな状態で GREEN 判定(自己申告でなく第三者検証)
テスト改ざん防止RED 時点のテスト指紋と照合(exit 0 = 不変)。偽 GREEN を弾く

ループ予算3周以内・飛び番なし・GREEN 後追加なしを第三者監査

中核 2orchestrating-team-development

フェーズゲートspec策定 → デザイン → テスト設計 → 実装 → 検証 → レビュー → FB反映 → 統合 を、タスク依存関係で順序を構造的に強制(口頭指示でなく依存でゲート)
behavior oracle(振る舞い検証)テスト全 PASS + lint exit 0 + spec 照合 Missing=0(自己申告は完了の証拠にしない)
二層検証behavior(振る舞い)+ structure(ファイルサイズ / 関数長 / 循環依存 / 重複)の両方通過で合格
