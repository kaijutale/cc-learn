02なぜ Progressive Disclosure か — 根拠の要約

Progressive Disclosure (段階的開示 — 詳細を外部ファイルへ置き、必要な時にだけ読み込ませる設計パターン) を rules/ に適用する根拠は、 調査ドキュメント .docs/research/progressive-disclosure-adoption-rationale.md (note 7 記事の横断調査) に集約済み。要点は 4 つ:

  • 帯域はゼロサム — コンテキストの枠に入る情報量には上限があり、ある情報を入れると別の集中力が薄まる (260405「机のたとえ」)。
  • 足すほど守られなくなる — 指示 10 個で遵守率 約90%、50 個以上で急落。無関係なルールの常駐は、核ルールの遵守率まで下げる。
  • 遅延ロードが原則 — 「重要な情報は早めに外部化し、必要な情報は遅延ロードする」(260405)。Anthropic 自身が Skills の中核原則としている (260501)。
  • paths は準決定論 — 「必要な時に Read してくれるか」というモデルの確率的判断を、ファイルマッチによる機械注入へ置き換えられる。ただし後述の通り発火条件に重大な制約がある。

04仕分けの意見 — issue #110 原案からの修正 5 点

原案は「26 行を paths 化」だが、上記の制約を踏まえると 一括適用は過適用。 先行して paths 化してよいのは 3 本 13 行 (Tier A)、残り 2 本 13 行 (Tier B) は paths 単独では成立しないため hook (ツール実行に連動して決定論的に発動する仕組み) との併用設計へ分離する。

理由UI 作業は既存コンポーネントの Read が先行することがほとんどで、Read 発火と相性が良い。新規ファイルをいきなり書く隙間は、常駐している designing-beautiful-frontends skill の説明文が補完する。

TIER Abuild-test-protocol (4 行) — paths 化するが glob を拡大 原案修正

原案テストファイルのみ (例: **/*.test.ts)
修正案コード全域: **/*.{ts,tsx,js,jsx,mjs,cjs,py,go,rs,sh}
理由この規約が必要なのは「コードを書いて出荷する時」全般。テストを読まないセッションこそ検証をサボる危険が高く、発火の門をテストファイルに置くのは本末転倒。コード全域なら実装作業では確実に発火し、markdown だけのセッション (本プロジェクトのような学習・文書作業) では消えて帯域を節約できる。

05実装フェーズ計画

  • Phase 0: 仕様の実測検証 (必須ゲート) — worktree + CLAUDE_CONFIG_DIR 隔離環境に、合言葉入りの canary rule (動作確認専用の使い捨て rule) を置いて 5 点を観測: (a) 起動時に載らないこと (b) マッチするファイルの Read で載ること (c) Write だけでは載らないこと (d) 現行バージョンで paths: が効くこと (効かなければ globs: 併記) (e) subagent へ届くか。
  • Phase 1: Tier A 3 本へ paths 付与 — 13 行を起動時注入から除外。1 rule = 1 コミットで committer による小コミット、PR を作り人のレビューを経てマージ。本体 ~/.claude の直接編集は絶対禁止。
  • Phase 2: Tier B の hook 併用設計 — ExitPlanMode への PreToolUse hook 等を検討し、別 issue に切り出す (paths とは別の仕組みゆえスコープを混ぜない)。
  • Phase 3: 観測と調整 — 数セッション運用し「効くべき場面で効いたか」を .docs/logs/ に記録。頻度を見て階層を上げ下げする (「頻度を見て下層から格上げ」の運用原則、260517)。

06リスクと対策

WARNWrite 発火なしの隙間 (Tier A にも残る)

シナリオ既存ファイルを一切 Read せず、新規 UI ファイルをいきなり Write する場合、frontend-aesthetics が効かない。
対策常駐する skill 説明文と PD ドキュメント参照で補完。Phase 3 の観測でこのシナリオの実際の頻度を計測し、問題なら注入へ戻す。

WARNsubagent への継承が未文書

シナリオrules が subagent に届かない場合、rule 前提で動く subagent 設計は静かに壊れる。
対策Phase 0 で実測。設計原則としても「rule はメイン会話向け、subagent は自身の定義文で完結」を維持し、依存を作らない。

WARNpaths: が効かない構成の報告 (#17204)

シナリオ公式記載の paths: 形式が動かず、非公式の globs: だけが動く環境がある。
対策Phase 0 の canary で自環境の挙動を確認。効かない場合のみ globs: を併記し、公式修正後に削除する前提でコメントを残す。

CONTAINED失敗時の影響範囲は小さく、完全に可逆

理由核 84 行は一切触れないため、最悪でも影響は Tier A の 13 行に限定。ロールバックは frontmatter を削除するだけで元の全注入状態に戻る。
