機械処理すると誤る3つの論点

40項目の大半は分類が明快じゃった。だが、ラベルだけ見て一括処理すると誤る論点が3つある。「散らばった規律をどこへ集めるか」「本当に消せる6本」「同じ事実が複数PJに重複する問題」── ここを解剖する。

A
規律17本の行方

散らばった規律と、移し先の逆説

散在している

未明文の規律17本のうち、6本は ios-memopo / pixel-leap / ios-ideas / learn に紛れ込んでいる。中身はどのPJでも通じる汎用ルールで、特定PJに置く必然がない。

だが全部 rules は危険

17本すべてを「常時読まれる rules」へ移すと、★ を付けた prefer-skill-over-rules-injection 自身が戒める常時ロードの肥大を招く。

逆説

規律を守ろうとして規律を破る形になる。学習PJの方針は既に「振る舞いルールは手動で厳選 (curate)、自動蓄積は OFF」と宣言済み。ゆえに正解は「全件移行」ではなく、critical-thinking に1-2行で凝縮 + 呼出文脈の明確なものは memory 残置。

判断: 厳選して手動昇格 + 一部 memory 残置 設計判断 ◆
B
本当に消せる6本

削除して情報を失わないもの

40本のうち、削除して何も失わないのは6本だけ:

  • カバー済 (4本) — handoff-reflects / no-raw-internal-labels / self-config-edit-denied (~/.claude) と git-write-ops-blocked (roi-fourre)。どれも CLAUDE.md か権限拒否で既に効いている
  • 完了ログ (2本) — thariq-implementation-notes (skill 化済み) と project_memory_timestamp_trial (検証用ダミー、archived)
1つだけ条件付き

self-config-edit-denied は本文末尾に「許可/拒否を読み違えて無駄な編集を試した」教訓がある。これだけは permission-path-semantics へ移してから消すこと。

判断: 6本削除 (1本は教訓を移してから) 明確
C
重複と越境

同じ事実が二重に住んでいる

memopo の二重持ち

同じ memopo アプリの記録が ios-memopo (実装フェーズ) と ios-ideas (企画フェーズ) の2PJに分散。正典がどちらかを決めて寄せたい。

warm-paper の越境

HTML美学の選好が ios-memopo にあるが、本文自身が「グローバル skill に反映済み」と書いている。実体はハーネス側、memory は経緯ノート。

休止PJの鮮度

toybox / ninmu は MEMORY.md に本文直書き。中身は再開時に効く技術メモで残す価値があるが、冒頭の「休止中 (2-3ヶ月前〜)」ステータスは古い。再開時に鮮度確認が要る。

判断: 正典を一本化、越境は経緯のみ残す 要整理
