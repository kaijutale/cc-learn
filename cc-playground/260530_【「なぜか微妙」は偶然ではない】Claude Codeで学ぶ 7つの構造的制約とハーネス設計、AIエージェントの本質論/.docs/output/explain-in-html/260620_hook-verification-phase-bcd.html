02検証の土台を 3 層に分けた

テストの足場 (lib) を関心ごとに 3 つへ分離した。1 つの部品が 1 つの仕事だけを持つようにし、後から読んでも責務が一目で取れる形にしてある。

  • harness.sh — hook を起動して、標準出力 (stdout) ・標準エラー (stderr) ・終了コードを捕捉する土台。今回 hooks/ の外に住む helper hook (三要素ハーネスのスクリプト) も起動できるよう、パスを解決する _resolve_hook を足した。テストの入力 JSON を組む補助 (PostToolUse / PostCompact / SessionStart) も追加。
  • assert.sh — 合否を判定するヘルパ。hook の「物の言い方」は 3 種類ある ―― 通知 (stdout に JSON) ・警告 (stderr + 終了コード 2) ・素通り (何も出さない)。これを別々の判定関数に分け、出力チャネルの取り違えという後戻りの劣化 (regression) まで捕まえられるようにした。
  • fixtures.sh (新規) — 隔離した一時ディレクトリの作成と後始末、最終更新時刻 (mtime) の操作、通知音や読み上げといった副作用コマンドの無効化 (stub) をまとめた。テストが macOS に音を鳴らしたり喋らせたりせず、本番環境を汚さず動く。

仕組み上の地雷も踏み抜いた。TMP="$(new_tmp)" のように一時ディレクトリ作成を コマンド置換 (副シェル) で呼ぶと、後始末リストへの登録が副シェル内で消える。そこで登録先をファイル (レジストリ) にし、副シェルをまたいでも親プロセスが確実に回収できるようにした (古い bash 3.2 での空配列エラーも同時に回避)。

03essence-gate に「壊れた hook を通さぬ門」

commit を見張る hook_pre_commit_essence_gate.sh (skill / agent / hook の改修を essence レビュー結果で gate する仕組み) に、新しい判定次元 (c) を後方互換で足した。ステージした hook 本体を抜き出し、その検証テストを run-all.sh --changed で走らせ、赤 (失敗) なら commit を拒否 (deny) する。既存の判定 (a: レビュー結果が不適格 / b: レビューが古い) は一切変えていない。

hooks/hook_pre_commit_essence_gate.sh + (c) hook-test gate−0 (既存 a/b は不変)
6464 [ -z "$STAGED" ] && exit 0 # ステージ無し = 対象外
65+# (c) 変更 hook の検証テストが赤なら deny
66+if [ -z "${ESSENCE_GATE_HOOK_TEST_RUNNING:-}" ]; then # 再帰防止
67+ RUNNER="${HOOK_TEST_RUNNER:-.../test/run-all.sh}" # テスト用 seam
68+ # ステージ済から hook 本体 (.sh) を抽出
69+ # 未カバー hook は stderr で可視化 (block はしない)
70+ TEST_OUT=$(... bash "$RUNNER" --changed $HOOK_FILES)
71+ [ "$TEST_RC" -ne 0 ] && deny "検証テストが赤"
72+fi
6573 # 以降は既存の verdict gate (a/b) ―― 変更なし

commit パスを直撃する改修ゆえ、落とし穴を 2 つ設計で塞いだ。無限再帰 ―― 門がテストを走らせ、そのテストが門自身を呼び戻す環 ―― は ESSENCE_GATE_HOOK_TEST_RUNNING という合言葉 (sentinel) で断ち切る。テストのしやすさは HOOK_TEST_RUNNER で検証ランナーを差し替え可能にし、赤 / 緑を模擬 (stub) して隔離検証できるようにした。

04レビュー指摘の反映

commit 前に harness-essentials-reviewer (ハーネス本質レビュー専任エージェント) を別コンテキストで走らせ、判定は CONDITIONAL GO (条件付き合格) / Critical 0・High 4・Medium 2。直せる指摘はその場で潰し、構造的なものは記録した。

05残した宿題

  • commit → push → PR — push はかいじゅうのみ。本変更は hooks/*.sh を含むゆえ、わらわが作った門がわらわの commit を試す (self-eval が要る、または記録済み前提で SKIP)。
  • マージ順序 — 兄弟計画 (over-block / #5) を先にマージし、本件を最後に重ねて 24 / 24 を確定する。両者が足したテストを二重に書かぬための順序じゃ。
  • live 検証 — $HOME/.claude へ反映した初回 commit で、保護パスへの過剰ブロックが起きないことを実証する (worktree では実証済)。

06総評

これまでの健診は「hook が配線されているか」までしか見なかった。今回の層はその先へ進み、各 hook が実際に意図どおり拒否し・通知し・警告するかを機械で確かめ、壊れた hook を commit の手前で止める。ハーネスがハーネス自身を検証する再帰構造が、口先の約束ではなく commit ゲートという実体を持った ―― それが Phase B/C/D の核心じゃ。
