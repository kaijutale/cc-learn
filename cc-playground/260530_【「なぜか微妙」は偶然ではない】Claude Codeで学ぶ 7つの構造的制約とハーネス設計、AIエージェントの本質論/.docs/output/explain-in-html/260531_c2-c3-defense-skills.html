// なぜ · 01

「気をつける」では防げない偏り

「7つの構造的制約」とは、AIエージェントが構造的に抱えてしまう弱点のこと。今回はそのうち2つに絞って、防御する道具を作った。

C-2 (フレーミング/アンカー) ── 同じ中身でも「提示のされ方」で結論が変わってしまう偏り。メリットを強調して見せられれば採用したくなり、リスクを強調して見せられれば却下したくなる。最初に示された枠組み (アンカー = 錨) に固定され、それ以外の側面が見えなくなる。

C-3 (迎合) ── 相手の期待を察すると、それに忖度して欠陥を見て見ぬふりしてしまう偏り。コードレビューなら「これを書いた人はこう言ってほしいだろうな」を読み取って、つい甘く評価する。

この2つが厄介なのは、偏っている本人には偏りが見えないこと。「気をつけよう」という心がけでは防げない。だから意志ではなく、入力や評価の段階に 仕組み (道具) を差し込んで、構造的に潰す方針を取った。

// 道具 · 02

偏りを潰す2本の道具

攻め口の違う2本を作った。① は入力からアンカー (錨) を剥がす方向、③ は出力を両極に振って相殺する方向。同じ偏りを、逆側から挟み撃ちにする。

01
C-3 迎合 + C-2 アンカー

blinding-review-prompt — 著者情報を剥がす「盲検化」

レビューを始める前に、著者名・変更の説明文・期待される結論を機械的に剥がし、コードの変更内容 (diff = 差分) だけを見せる道具。誰が書いたか分からなければ、忖度のしようがない。さらに「最低でも何件は指摘せよ」という下限を変更の規模から自動計算して課し、安易な「問題なし」を封じる。

剥がすもの (アンカー)

著者名・変更タイトル・説明文・期待される結論 ── 判断を引きずる手がかり

残すもの (中身)

コードの変更内容・ファイル数・追加削除の行数 ── 客観的に判断できる材料だけ

構成: 本体 + 判断基準の2ファイル (分岐起動なし) 変換ツール
03
C-2 フレーミング

detecting-framing-bias — 利点役とリスク役を相殺させる

同じ対象を「利点側に振り切った評価役」と「リスク側に振り切った評価役」に、互いの意見を見せずに別々で評価させる道具。両極の主張を突き合わせれば、提示のされ方による偏りが相殺される ── 「悪魔の代弁者 (Devil's Advocate = あえて反対意見を述べる役)」を自動化した装置。

利点役 (merit)

メリット・機会・成功シナリオに、根拠の範囲で最大限振り切る

リスク役 (risk)

欠点・失敗・最悪シナリオに、根拠の範囲で最大限振り切る

なぜ「別々に」評価するか

同じ場で「まず利点、次にリスク」と順番に考えると、先に出した利点が後のリスク評価の足枷になる ── まさに防ぎたい偏りが装置の中で再発する。だから互いを見せず、完全に分離して評価させる。

構成: 本体 + 評価役2種 × (起動役 + 中身) = 6ファイル 評価装置
// 検証 · 03

作って終わりにしない

道具は「書いた」だけでは「効く」か分からない。実際に動かし、第三者に使わせ、磨くまでを一周した。特に大事だったのは、書いた本人 (わたし) が「これで明確だ」と思っても、それは当てにならない ── 別の誰かに使わせて初めて曖昧さが見えるという点。

道具を「効く」状態にするまでの動線

実装 → 実機 → 自己評価 → 第三者で磨く → 合成

Step 01 · 実装

8ファイルを作成

① の2ファイルと ③ の6ファイル。最大の懸念だった「特殊な書き方が拒否されないか」は、実際に起動して拒否されないと確認できた。

Step 02 · 実機

分岐起動 (grayzone) が動くか確認

③ は別の独立した場所で子エージェントを起動する仕組み (公式には動作保証が曖昧な領域) を使う。本体に束ねる前に、単体でちゃんと子が応答を返すことを実機で確認した。

Step 03 · 自己評価

③ に ③ 自身を評価させる

「この6ファイル構成は妥当か」を ③ 自身にかけたら、リスク役が設計者が見落としていた急所を3件発見。その場で対策を組み込んだ (詳細は次セクション)。

Step 04 · 磨く

何も知らない第三者に使わせる

事情を知らない別の Claude に道具を渡して動かしてもらい、どこで迷ったかを報告させた。① で本物の曖昧さが2件出たので修正し、もう一度別の第三者にかけて「迷いが消えた」ことを確認した。

Step 05 · 合成

①の出力を③に流す二重防御

① で著者を隠した対象を ③ に評価させる連結を、実際の変更で試した。「ドキュメントだけだから安全」という見かけを ③ が相殺し、隠れた危険を炙り出した。

// 白眉 · 04

道具が、身内にも忖度しなかった

このセッションで一番効いたのは、道具が自分自身や身内の変更にも忖度しなかった瞬間だった。

自己評価の発見 ── ③ に ③ 自身の設計を評価させたら、リスク役が「評価をまとめる統合役自身も、対象を先に読んで偏ってしまう」という、設計者が完全に見落としていた急所を指摘した。子エージェント同士は分離したのに、最後にまとめる役が無防備だった。すぐに「まとめ役は自分の第一印象を記録して割り引く」という手順を組み込んだ。

合成での発見 ── ① と ③ をつなぎ、著者を隠した実際の変更を評価させたら、「ドキュメントだけ・コード変更ゼロだから安全」という見かけのフレームを ③ が相殺。リスク役が「レビューを経ずに、動作中の道具が1つ静かに消えていた」という隠れた問題を、実地調査つきで掘り当てた。利点役は「不要物の整理」と評価していたが、突き合わせると危うさが見えた。

身内の変更にも甘くならず、見かけのフレームに騙されず、本物の問題を掴む ── これが「偏りを潰す道具」の本懐だった。

検証した工程結果
特殊な書き方が拒否されないか (実機)拒否されず、想定通り動作
分岐起動で子エージェントが返るか動作 (公式に曖昧な領域を実機突破)
利点役 / リスク役の独立性互いの意見に一切言及せず
③ の自己欠陥の検出3件を発見 → その場で修正
第三者 (白紙) による曖昧さ検出① で2件 → 修正 → 再評価で効果確認
① × ③ 合成での危険検出見かけのフレームを相殺し隠れた問題を検出
既存資産への影響ゼロ (後方互換を死守)
