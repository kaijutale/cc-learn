2026.06.27設計解説担い手 = 所有 ≠ 読む

01ひとことで

重い調査の結果を research.md という独立ファイルに書き出し、それを踏み台に仕様書 (spec.md) を作る。そして 実装は仕様書だけを読み、調査の生データ (research.md) は読まない。これが技術記事「7つの構造的制約」(260405) のパターン2「Phase 分離」で、現ハーネスに欠けていた最初の段（Phase 0＝調査）の補完である。

research.md は「仕様を作るまでの踏み台」。用が済めば実装には持ち込まない — それが今回の設計の狙い。

03なぜ外部化するのか（最重要点）

実装フェーズは仕様書 (spec.md) だけを読み、research.md は読まない。こうすると、実装のコンテキスト（AI が一度に集中できる作業領域）が調査の生データで埋まらず、集中力が保たれる。これが「関心の分離」である。

今回の改修では、実装を担う skill 群に「research」という参照が 一つも無いことを機械検査 (grep) で確認した（0 件）。つまり調査の生データが実装に漏れ込む経路は、構造として存在しない。

04「担い手」の正体 — 読む人ではなく、決める人

ここが一番誤解されやすい。「担い手」とは research.md を読む人のことではない。「研究をいつ起こし、その何を仕様に活かすか」を 決める人のことだ。

下の表が核心。「誰が読むか」は案A でも案B でも同じ（spec係）。違うのは 「誰が決めるか」の列だけ。

　誰が research.md を読むか誰が "決める" か（＝担い手・所有者）
案A（却下）spec係spec係 本人　調査の要否も・何が効くかも自分で決める（指揮者は不在）
案B（採用）spec係指揮者　決めて渡す。spec係は指示されて読むだけ

読む手はどちらも spec係で 変わらない。だから「spec係が読むから案A」ではない。決定権の置き場所こそが案A／案B の分かれ目だ。所有者が握る「決定」は2つ — 発火判断（そもそも重い調査をするか）と 統合判断（調査結果のどれが仕様に効くか）。今回はこの2つを指揮者に持たせた（案B）。

05家づくりで例えると

三者の役割は、家を建てる現場に置き換えると一目で分かる。

役現場の例えハーネスでの担当やること
調べる人測量士調査の子エージェント土地を測り報告書 (research.md) を書く。大量の測定値（生データ）は自分の中だけに留め、監督には要約だけ渡す
決める人現場監督オーケストレーター（指揮者）測量するか決める・どのデータが効くか決める・報告書を設計士に渡す ＝ 担い手・所有者
描く人設計士spec-based-development報告書を読んで設計図 (spec) を描く。読む・書くが、決めるのは監督

設計士が報告書を読むのは案A でも案B でも同じ。案A は設計士が監督を兼ねる（末端が決定権まで持つ）、案B は監督が決めて設計士は描くことに徹する。今回選んだのは案B だ。

06まとめ — 3つの役割を混ぜない

  • 決める＝指揮者（オーケストレーター） — 調査するか（発火）・何が仕様に効くか（統合）を所有する。
  • 読んで書く＝spec係（spec-based-development） — research.md を読み、仕様 (spec.md) を書く。決定権は持たない。
  • 生データを背負う＝調査係（子エージェント） — 大量の検索結果は自分の中に閉じ込め、要約だけ返す。

07総評

「担い手」は research.md を読む人ではなく、研究をいつ起こし何を仕様に活かすかを決める人のこと。読むのは案A でも案B でも spec係で変わらない — 変わるのは "決める頭" が spec係本人か指揮者か、だけ。今回は指揮者に決定権を持たせ（案B）、spec係は読み手・書き手に徹させた。決める・読んで書く・生データを背負う、この三役を混ぜないことが今回の設計の核心である。
