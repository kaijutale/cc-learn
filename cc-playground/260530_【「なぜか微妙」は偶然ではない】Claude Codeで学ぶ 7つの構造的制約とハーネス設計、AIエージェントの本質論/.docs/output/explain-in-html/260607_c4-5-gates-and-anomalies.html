異常時と
5つのゲートを
1つずつ

「異常時」= 正常ルートがすんなり通らない時。そこには2系統あって、 coderが自分で立て直す系と、機械のゲートが弾く系。 後者の番犬(検証スクリプト)が 全5つ。1つずつ、何を見て・どう開閉するかを見ていくよ。

前回までで「ゲート = 判断しない機械の関門」と整理した。今回はその一歩先 ── ゲートが「いつ(=どんな異常のとき)」「どう(=5種それぞれ)」働くかを、具体的に開けていく。

// PART · 01

異常時とはどういう時?

まず正常時。これは テストを書く → 実装する → 検証する が、一発でスッと合格(GREEN)まで通る流れ。何も引っかからない。

異常時は、その流れがどこかで崩れる時。崩れ方は大きく 2系統 ある。

系統A — coderが自分で立て直す

実装が一発で正解にならず検証が不合格。coderが失敗ログを見て自分で直し、再検証する(最大3周)。2周目以降は「5人の専門家会議」(llm-debate)で方針を立て直す。これはゲートの仕事ではなく、coder自身の能力。

系統B — 機械のゲートが弾く

ズル(嘘の合格)・テスト不足・構造の崩れ・ループの誤魔化し。これらは coder の言葉では見抜けないので、判断しない機械の関門(=5つのゲート)が物理的事実で弾く。今回の主役はこっち。

つまり「異常時」と一口に言っても、自分で直せばいい異常(A)と、放っておくと不正がすり抜ける異常(B)がある。ゲートが守るのは B。次でその5つを1つずつ見るよ。

// PART · 02

5つのゲートを1つずつ

共通ルールは1つ ── どのゲートも「判断」しない。条件(物理的な事実)を満たせば開き(合格)、満たさなければ閉じる(不合格)。開閉の信号は 0 = 開く / 1 = 閉じる / 2 = 測定不能なので安全側に保留。

01
改ざん防止 · C-4の心臓

テストの指紋を凍結して照合する assert-tests-unchanged

何を見る

テストを書き終えた瞬間に、テストファイルの指紋(中身から決まるハッシュ値)を凍結保存。合格が出た時に、もう一度指紋を取る。

どう開閉する

2つの指紋が一致 → 開く。1文字でも書き換わった/消された → 閉じる(改ざん)。新しいテストの追加だけは正当なので開く。

捕まえる異常: テスト改ざん(嘘の合格=ズル) 最重要

例え: 答案に封印スタンプを押しておき、提出後に書き換えられていないか指紋で照合する。

02
テスト網羅 · 報告のみ

カバレッジ数値を基準と比べる assert-coverage

何を見る

テスト実行で出るカバレッジ(網羅率)の数値。コードの行を何%試したか、分岐(if の枝)を何%試したか。基準は 行80% / 分岐75%。

どう開閉する

基準以上 → 開く。基準未満 → 赤信号(閉じる)。測定器具(計測ツール)が無ければ安全側に倒して保留(勝手に合格にしない)。

捕まえる異常: テスト不足(※止めず報告。テストはcoderの担当外) 中

例え: 答案の何割が埋まっているかを測り、基準に届かなければ「ここ薄い」と知らせる。

03
構造の健全性

循環依存を見つける assert-no-cycles

何を見る

ファイル同士の参照(どのファイルがどのファイルを呼ぶか)を地図にする。

どう開閉する

A→B→A の堂々巡り(循環)があれば閉じる。無ければ開く。判定は地図をたどる機械処理だけ。

捕まえる異常: 循環依存(将来ほどけなくなる構造の崩れ) 中

例え: 部品どうしが互いに依存し合って外せなくなる「堂々巡り」を、地図で発見する。

04
記録係 · 証拠を残す

やり直し回数を台帳に記帳する record-loop-iteration

何をする

検証のたびに「何周目・結果(合格/不合格)」を、改ざんしにくい台帳ファイルに追記していく。

なぜ要る

coderが頭の中で数える回数(=自己申告)を、外部の台帳に出して証拠化するため。これ自体は弾かない。次の05が監査できるようにする土台。

役割: 監査の証拠を残す記録係(単体では合否を出さない) 土台

例え: ループの回数を、後から書き換えられない台帳に1行ずつ記帳していく記録係。

05
予算監査 · 第三者

台帳を監査してループ暴走を弾く assert-loop-budget

何を見る

04の記録係が残した台帳。coderの記憶でなく、外部に残った事実を読む。

どう開閉する

(1) 規定回数(3周)以内 (2) 回数が1から飛び番なし (3) 合格後に余計な反復なし ── すべて満たせば開く。1つでも破れば閉じる。

捕まえる異常: ループの暴走・誤魔化し(4周目/回数の飛ばし/合格後の続行) 中

例え: 台帳を第三者がチェックして「規定回数を超えてないか・改ざんがないか」を監査する。04(記録)+05(監査)で1セット。

// PART · 03

どの異常を誰が捕まえるか(早見表)

異常(何が起きた)担当判定の材料(物理的事実)
テスト改ざん=嘘の合格(ズル)01 改ざん防止テストの指紋(ハッシュ値)
テスト不足02 カバレッジ網羅率の数値(行/分岐%)
循環依存(構造の崩れ)03 循環検出参照の地図(import 解析)
ループの暴走・誤魔化し04 記録 + 05 監査台帳(回数と結果の記録)
一発で解けない(やり直し)ゲート外 — coder自身が修正 + 5人会議(機械でなくcoderの能力)

大事なのは最終行。「一発で解けない」はゲートの仕事ではない。これは coder が自分で立て直す異常(系統A)。ゲート(機械)が守るのは、放っておくと不正や不備がすり抜ける異常(系統B)だけ ── 客観的に白黒つけられるものに絞って、判断せず物理事実で開閉する。

ゲートは考えない。
事実が条件を満たすか、満たさないか。
だからズルが通らない。
