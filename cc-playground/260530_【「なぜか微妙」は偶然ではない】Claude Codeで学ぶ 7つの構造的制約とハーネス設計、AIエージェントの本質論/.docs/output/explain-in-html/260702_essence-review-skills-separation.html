03それぞれの人物像

単眼review-agent-essence — 町医者の設計相談

役割1つの対象を汎用原則に照らし、強み・弱み・改善案を返す。対象は直接修正しない。
特徴関連する原則だけを選定して当てる (全原則の機械的照合はしない)。1 fork で完結し、コンテキストの連続性を保つ。
注意orchestrator とは意図的併存 (Gotcha に must として明記)。旧世代の残骸ではなく、軽量な相談窓口として現役じゃ。

診断review-harness — 人間ドックの検査技師

役割skill 単体でなくハーネスという環境そのものを 25 のアンチパターン指標で採点し、文脈を考慮した改善案を出す。
特徴成熟度で出力を分岐する (設定ゼロの初心者環境にはセットアップガイド、成熟環境には本診断)。診断レポートの見た目が explain-in-html の美学の原典でもある。

複眼essence-reviewing-orchestrator — 3科合同の精密検査

役割ハーネス/skill/UI の3領域 fork を並列起動し、正本の全原則照合を Lead が統合判断。結果を機械可読形式で essence-review-records/ へ永続化する。
特徴検査の前に「基準そのものが古びていないか」を確認する同期検証 (check-essence-sync) を内蔵。進捗 JSON・エラー4分岐・self-eval 起動条件まで備えた最重装備じゃ。
唯一性3つの中で結果が機械の入力になるのはこれだけ — ここが accumulating との接続点になる。

04accumulating-reviewer-feedback は「一部」か？ — No、郵便受けで繋がる別工程

直答: orchestrator の構成要素ではない。根拠は3つ — (1) orchestrator の依存リストに accumulating は無い (依存は sub-skill 3体と評価基準のみ)、(2) orchestrator の副作用は「永続化ファイル書出のみ」で反映はしない、(3) accumulating は「明示呼出専用・自動誘発なし」でそなたが別途起動する。つまり検査 (orchestrator) と治療 (accumulating) は別の診察日じゃ。接続の実体はこの配管ぞ:

[検査工程]  essence-reviewing-orchestrator (そなたが明示起動)
              ├─ harness-essentials-reviewer-fork ┐
              ├─ skill-essentials-reviewer-fork   ├─ Lead が統合判断
              └─ ui-essentials-reviewer-fork      ┘
              ▼ 書き置き (file-as-message = ファイルを郵便受けにした非同期連携)
            ~/.claude/.docs/essence-review-records/
              ├─ <TS>_<target>.md               → essence-gate hook が commit 時に読む
              └─ <TS>_<skill>_self-eval-v<N>.md → accumulating が収集・版管理

[反映工程]  accumulating-reviewer-feedback (そなたが別途、明示起動)
              records を読む → severity 分類 → 再発照合 (段階2.5、PR #67)
              → 人間判断 (accept/defer/dismiss) → Gotcha 反映 → 二重記録

2つの skill は互いを呼ばない。繋がりは records ディレクトリと、self-eval ファイルの命名契約 _self-eval-v<N> (accumulating がこの命名で収集・版管理すると orchestrator 側の出力仕様書に明記) だけじゃ。さらに念の入ったことに、accumulating の改修禁止リストには orchestrator を含む協調ハーネス中核が載っており、反映装置が検査装置を書き換える事故は構造的に封じられておる。

05使い分け早見

  • 「この skill の設計、どう思う？」 — review-agent-essence。軽く1 fork、レポートで完結。
  • 「このプロジェクトのハーネス、健康か？」 — review-harness。構成全体を 25 指標で採点。
  • 「本気の全原則照合をして、記録も残したい」 — essence-reviewing-orchestrator。fork×3 の重装備ゆえ「3領域 essence レビュー」と明示して呼ぶ。
  • 「溜まった検査結果を skill に反映したい」 — accumulating-reviewer-feedback。records を読んで severity 分類し、そなたの決裁を経て Gotcha へ書き戻す。

06総評

3つの検査装置は「対象の広さ × 基準の正本性 × 結果を機械が読むか」で棲み分けられ、Gotcha にも意図的併存が must として明文化されておる。そして accumulating-reviewer-feedback は orchestrator の部品ではなく、検査記録の郵便受け (essence-review-records) を介して繋がる独立した反映工程 — 検査と治療を別 skill に分け、反映側には人間の決裁 (HITL) と改修禁止リストの二重の安全装置を付ける。この分離こそが、自己改修ハーネスが暴走しないための設計じゃ。
