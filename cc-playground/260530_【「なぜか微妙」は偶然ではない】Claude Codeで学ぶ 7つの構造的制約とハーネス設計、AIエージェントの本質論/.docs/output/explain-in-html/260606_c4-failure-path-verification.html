わざと失敗させて
番犬がズルを見破るか
を確かめる

AI に開発をさせる仕組み(ハーネス)は、正常に動くときはもう確認できた。 残っているのは、失敗やズルが起きたときに、ちゃんと止める・直すか。 これが failure path(失敗系の検証)。一番の山場は「嘘の合格を見破れるか」だよ。

まず、いま何の話をしているのかを一言で。わたしたちは AI にテスト駆動開発をさせる仕組み (ハーネス) を作って、それが本当に機能するかを確かめている。テスト駆動開発 = 「先にテストを書く → 実装する → 検証する」の順で進める開発スタイル。

そして全体を貫く合言葉が C-4「自己申告は完了の証拠にならない」。AI が「できました!」と言っても、その言葉は信じない。機械(検証スクリプト)が、書き換えようのない物理的な事実(ファイルのハッシュ値・カバレッジの数値など)で本当に確かめる ── という思想。

// PART · 01

登場人物は2人だけ

仕組みの中で動くのは、ざっくり 2 人 だと思えば十分。

coder(実装役の AI)

テストを書き、実装し、自分で検証する「作業者」。賢いが、放っておくと「できました」で済ませてしまう恐れがある。

検証器(番犬スクリプト)

coder の成果を機械的にチェックする見張り役。全 5 種。ズル・テスト不足・構造の崩れを、数値や指紋(ハッシュ値)で物理的に検出する。

AI が「できた」と言っても信じない。
番犬が物理的な事実で確かめる。 — C-4 の合言葉
// PART · 02

これまで分かったこと(=正常時は完璧)

ここまでの検証で、正常に進むときは完璧に動くことが確認できている。残すは「異常時」だけ、という状況。

✓番犬そのもの正常→通す / 違反→止める を実機で確認
✓coder 正常時ちゃんと番犬を呼ぶ(happy path)
✓root バグ番犬が場所を間違える不具合を修正
?失敗・異常時← これから(このページの主題)
// PART · 03

残り = わざと失敗させるテスト

例えるなら 自動運転車。「普通の道をまっすぐ走れる」のは確認できた。でも本当に大事なのは、急ブレーキ・障害物・ズルしてゴール のような異常時に、ちゃんと止まる・対処することだよね。

そこで今からやるのは、わざと異常を起こして番犬と coder の反応を見る ── つまり 「わざと失敗させるテスト」。これが failure path(失敗系の経路)。

普通の道は走れた。
次は、急ブレーキとズルでも壊れないかを試す。 — 自動運転の例え
// PART · 04

3つの「わざと失敗」テスト

F1
自律 · 本物の coder にやらせる

一発で解けない問題を出す

わざと何をする

最小実装が踏み外しやすい問題を出す(例: 銀行丸めで 2.5 → 2。素朴な実装だと 3 にしてしまい外す)。

何を見たい

coder が自分で何度も直し、途中で 「5人の専門家会議」(llm-debate) を開いて方針を立て直すか。

やり方: 本物の coder に自律でやらせる 自律で検証
F2
手動再現 · C-4 の心臓

テストをこっそり書き換えて「嘘の合格」を作る

わざと何をする

本来の厳しいテストを「ゆるいテスト」にすり替えて、雑な実装でも通ってしまう 嘘の合格(偽 GREEN) を作る。

何を見たい

番犬が「これは テストが改ざんされた ぞ」と指紋(ハッシュ値)の違いで見破って、合格を 却下 するか。

正直な注意

coder は自分でズル(テスト改ざん)を絶対にしないので、純粋な自律では起こせない。だから 人が「わざとズル」を仕込んで 番犬が見破るか試す = 手動再現。「AI が自分でズルを自浄した」という話ではない、と正直に区別する。

これこそ 「自己申告を信じない」の真価 最重要
F4
任意 · 軽め

テスト不足のまま実装する

わざと何をする

コードの一部しかテストしない状態(テストの網羅率=カバレッジが低い)にする。

何を見たい

番犬が「カバレッジが足りない」とちゃんと報告するか(※ここは止めはしない仕様。報告して人に判断を委ねる)。

やり方: 軽め。時間が余ればやる 任意

補足: 「3 周直しても解けない(=お手上げ報告する)」「番犬の予算オーバーを検知する」といった経路もあるけれど、これらは起こしにくい上に番犬の単体テストで既に確認済みなので、今回は F1 が 3 周で解けなかった場合に自然に観測される 範囲でカバーする。

// PART · 05

どう検証する(正直な注意つき)

使い捨ての 実験場(sandbox) を用意して、そこで 別の Claude セッションを立ち上げ、本物の coder を実際に走らせる。なぜ別セッションかというと、いまのこのセッションからは技術的な理由(作業フォルダの固定ができない)で coder を正しく起動できないから。

  • F1(問題を解かせる)は、本物の coder に 自律 でやらせる。
  • F2(嘘の合格)は、構造上 手動再現。coder は自分でズルしないので、人がわざとズルを仕込んで番犬が弾くかを見る ── ここは誇張せず正直に書く。
  • もう一つの教訓: 前回は検証のあと証拠を片付けてしまい、あとから独立に確かめ直せなかった。今回は 痕跡を消さず残す(掃除は次の人に任せる)。

そして毎回 「番犬が正しい場所を見ているか」 も確認する(さっき直した root バグの念押し)。具体的には、実験場の中だけに記録が残り、大もとのフォルダを汚していないかをチェックする。

ハーネスの本当の価値は、
正常時に動くことではなく、
失敗とズルをちゃんと捕まえること。
