gtr 系 worktree との違い

2026.06.27概念解説ブランチ戦略 vs worktree ツール

01まず結論

gitflow は Git の「機能」ではない。ブランチ(枝分かれした作業の系統)をどう切り、どう統合するかを取り決めた「運用ルール・戦略」じゃ。一方の worktree(作業ディレクトリ)は Git に組み込まれた本物の機能、gtr はその worktree を扱う道具(ツール)。

だから ── gtr 系 skill で worktree を作っても、gitflow が使われるとは限らない。ハーネスの中で gitflow を使うのは launching-gitflow-worktrees ただ一つじゃ。

02gitflow — 「機能」ではなく「戦略」

gitflow は 2010 年に Vincent Driessen が提唱したブランチ運用モデル(原題 "A successful Git branching model")。次の役割でブランチを切る「決め事」じゃ:

  • main — 本番リリース用。常に安定した状態を保つ
  • develop — 開発の統合先。各機能を集めて統合する場所
  • feature/* — 個別の機能。develop から枝分かれし、完成したら develop へ戻す
  • release/* · hotfix/* — リリース準備・本番の緊急修正

補助ツール git-flow(コマンド)は存在するが、それも普通の git コマンドを束ねたラッパー(包み紙)に過ぎない。Git 本体に「gitflow」という機能は無い ── あくまで人間が決めた使い方の約束事じゃ。料理に例えるなら、包丁や鍋(= Git の機能)ではなく、段取りを定めたレシピの手順書に当たる。

gitflow の狙いは「AI や開発者の成果物を、いきなり本番(main)に直結させない」こと。feature で作業し、develop で統合し、最後に人間が main へマージする ── この段取りの規約こそが gitflow じゃ。

03worktree と gtr — 「仕組み」と「道具」

worktree(ワークツリー)= Git の機能。1 つのリポジトリに対して複数の作業ディレクトリを同時に持てる仕組みじゃ。各 worktree は別々のブランチを開けるので、並列で作業しても互いのファイルを上書きせず、衝突を物理的に防げる。机を 1 つ増やして、別の作業を同じ部屋で同時に進めるイメージじゃ。

gtr(git-worktree-runner)= 道具。worktree を作る・消す・移動するのを楽にする補助ツール。中身は Git 標準の git worktree を包み、設定ファイルのコピーや作成後の自動処理(フック)などの便利機能を足したものじゃ。

04混同を解く — 4 つのレイヤー

この 4 つは別々の階層に属する。ここを一緒くたにすると見誤る:

レイヤー正体役割
Git基盤branch(枝分かれ) / merge(統合) / worktree の機能を提供する土台
worktreeGit の機能作業場所(作業ディレクトリ)を分離する
gtr道具(ツール)worktree を作る・消すのを楽にする
gitflow戦略(規約)ブランチの切り方・統合の仕方を取り決める

核心は ── gitflow は「枝の設計図」、worktree と gtr は「作業場所とその道具」。両者は軸が直交しておる。gitflow は worktree が無くても成立するし(1 つの作業場所でブランチを切り替えればよい)、worktree は gitflow が無くても成立する(どんなブランチでも作業場所を分けられる)。

05ハーネスの gtr 系 skill との対応

そなたのハーネスには gtr を使う skill が 3 つある。そのうち gitflow を使うのは一番下の 1 つだけじゃ:

skillgitflow作るもの
creating-gtr-worktreeなしworktree を 1 つ(ブランチ 1 本)だけ
launching-gtr-issue-worktreeなしissue-<番号>-<名前> というブランチ 1 本
launching-gitflow-worktreesありmain → develop → feature/team-* の階層 + 各 worktree + main/develop 保護フック

launching-gitflow-worktrees だけが「gtr を道具として使い、その上に gitflow の階層を建てる」── つまり道具と戦略を組み合わせた skill じゃ。複数チームが並列でファイルを書く規模になって初めて、本番(main)を守る gitflow の段取りが必要になる。

そなたが issue #50 で通した道は「gtr(道具)は使ったが、gitflow(戦略)は使っていない」典型例じゃ。単発の 1 作業に、main を守る gitflow の階層までは要らぬ。gitflow が出てくるのは、チーム開発級に並列度が上がったときだけじゃ。

07総評

gtr で worktree を作ること と gitflow は、別レイヤーの話じゃ。gtr は道具、worktree は Git の機能、gitflow はブランチの戦略 ── この三つを「同じもの」と捉えると設計を見誤る。普段の単発作業は gtr 系(gitflow なし)で十分、本番を守りながら複数チームを並列で走らせるときだけ launching-gitflow-worktrees で gitflow を建てる。この使い分けだけ押さえておけばよい。
