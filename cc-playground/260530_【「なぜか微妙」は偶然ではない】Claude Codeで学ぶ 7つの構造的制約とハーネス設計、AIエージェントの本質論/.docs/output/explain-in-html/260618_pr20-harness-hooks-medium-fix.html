個々のバグはバラバラの偶然ではなく、5つの設計上の癖が hook 層を横断していた。修正もこの根因に沿って束ねている。

#構造的根因あるべき方向
1人間が読む文章を正規表現で機械的に読む機械可読フィールド / 語の境界 / コード範囲に限定

036つのバッチで何を直したか

修正は「同じ性質のもの」でまとめ、6つのコミット (バッチ) に分けた。1つずつ実機で検証してから積んでいる。

A · 配線新しい編集ツールで検査がすり抜けていた

項目#17 (根因3)
症状MultiEdit (複数箇所を一度に書き換えるツール) で編集すると、行数・絵文字・命名規約などの検査 hook が一切発火しなかった。検査の発火条件が古いツール名しか見ていなかったため。
修正発火条件 (matcher) に MultiEdit を追加。点検中に、同じ取りこぼしが報告外の2箇所 (うち1つは保護用のセキュリティ hook) にも潜んでいたのを発見し、まとめて塞いだ。

B · 出力形式検査結果のJSONが壊れて片方が消える

項目#12, #4
症状lint hook が「文法エラー」と「型エラー」の両方を見つけた時、結果のJSONを2回に分けて出力し、片方が握り潰されていた。別の hook はブロック応答に非推奨の古い形式を使っており、将来のバージョンで無視される恐れがあった。
修正診断を1つにまとめて1回だけ出力。ブロック応答を現行の正式形式 (permissionDecision:"deny") へ。

C · 安全側へ秘密ファイル検査が異常時に素通りしていた

項目#1 (根因2)
症状秘密ファイル (.env や SSH 鍵) の読み取りを止める hook が、入力が壊れて対象パスを取れない時に「素通り=許可」していた。これが fail-open (失敗時に危険な側へ倒れる) の典型。
修正判定できないなら拒否側 (fail-closed) へ倒す。正常な読み取りは必ずパスを持つので、正当な操作を巻き込む誤ブロックは起きない。

D · 異常終了セキュリティhookが途中で落ちて素通り

項目#6, #7, #5 (根因2, 5)
症状司令塔モード (Macro 層) での書き込みや、チーム用エージェントの直接呼び出しを制限する hook が、内部の検索が空振りした瞬間に script ごと異常終了し、結果として制限がかからなかった。さらに作業フォルダ名に [ ] 等の特殊文字があるとパス判定が壊れた。
修正空振りで落ちないようガード (|| true) を追加し、変数をクォート。配線Aで見つけた残りのツール名取りこぼしも、この hook の修正と同じコミットで塞いだ。

E · 境界数え方・大小・閉じ忘れの取りこぼし

項目#14, #8, #15 (根因1, 4)
症状行数を「改行の数」で数えるため、末尾に改行がない501行のファイルを500行と誤判定して見逃す。パス検査が大文字小文字を区別し、小文字の /users/ を見逃す。SKILL.md の設定部 (frontmatter) の閉じ忘れを検出できない。
修正行数は行単位で数え、大小を無視して照合し、区切り (---) が2本ない閉じ忘れを検出するようにした。

F · 誤ブロック緩和正当な操作まで巻き込む過剰ブロックの是正

項目#2, #3, #9, #10, #11, #16 (根因1)
症状禁止コマンドの検査が部分一致で、無関係な語 (例: 「curly」) まで誤検知。保護フォルダ「から」外へコピーするバックアップ操作まで禁止。発言の言葉づかい検査が、直前がツール操作だと最新の発言を見逃す。参照リンク検査やサーバー名抽出の取りこぼし。
修正語の境界で照合し、書き込み先 (最終引数) だけを判定し、直近のテキスト発言まで遡って検査する、など6点を是正。このPRを作る間、実際にこのバグが私の正当なコマンドを巻き込んでブロックしたため、症状を実地で確認できた。

05監査の指示を鵜呑みにせず下した判断

点検レポートの記述をそのまま適用せず、安全性と回帰リスクを天秤にかけて以下を判断した。

  • 監査外のバグも発見・修正 — ツール名の取りこぼし (根因3) は報告された1箇所だけでなく、同じ原因で2箇所 (片方は保護用セキュリティ hook) にも潜んでいた。挙動を壊さない追加ゆえ、まとめて塞いだ。
  • 自動承認の挙動 (#4) は意図的に据置 — ブロック側だけ現行形式に直した。「ブロックされないコマンドを全部自動承認する」挙動を形式変更で触ると回帰の恐れがあるため、次フェーズ送りに。
  • 重い型チェック (#13) は現状維持 — 毎回プロジェクト全体を型チェックするのは重いが、外すと型エラー検出を失う。検査は残し、設計事項として持ち越し。
  • Windows パス検出は見送り — この環境は macOS 専用で発生確率がほぼ無く、ブロック hook の正規表現を複雑にする risk に見合わぬと判断。文書化済みの検査範囲に揃えた。

06どう検証したか

「動いた」で終わらせず、各 hook を一時フォルダ (/tmp) で実際に動かして両方向を確認した。live (本番) の ~/.claude は触らない。

  • 取りこぼし方向 — 直すべき入力が「見逃し → 捕捉」に変わるか。
  • 過剰ブロック方向 — 正当な操作が「誤ブロック → 通過」に変わるか。
  • 正常系維持 — 元々正しく動いていたケースを壊していないか。
  • ルール変更の faithful テスト — 検査 hook が本番のルールファイルを直接参照するため、hook を /tmp にコピーしてルール参照先を作業ブランチ側へ差し替え、実際の変更が効くことを確認した。
  • 機械チェック — 全11個のシェル hook が文法チェック (bash -n) を通過、変更したJSON 2本も妥当。
