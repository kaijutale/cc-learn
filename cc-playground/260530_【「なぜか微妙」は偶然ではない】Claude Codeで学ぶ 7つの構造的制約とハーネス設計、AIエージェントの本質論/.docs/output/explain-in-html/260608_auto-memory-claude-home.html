判断が割れる3つの論点

24本のうち大半は分類が明快じゃった。だが、機械的に処理すると誤る論点が3つある。削除に見えて教訓だけ救うべき本、規律だが「移す先」を間違えると逆効果になる本 ── ここを解剖する。

A
削除 · ただし条件付き

「消す前に1行だけ救う」本

reference-self-config-edit-denied は「CLAUDE.md は編集拒否」という事実で、これは権限拒否 (deny) として構造的に既にブロック済み ── だから削除候補 (Covered) じゃ。

機械削除すると失うもの

本文末尾に「許可と拒否の設定を読み違えて、無駄な編集試行をした」という教訓がある。これは構造ではカバーされぬ振る舞い。完全削除する前に、この1行を reference-claude-code-permission-path-semantics(残置する技術ノート)へ吸収しておくのが安全じゃ。

判断: 教訓を移してから削除 条件付き削除
B
規律 · 移す先が問題

検証系4本は rules か memory か

rules 移行に見える

「未読を捏造しない」「非存在は全 PJ 検索してから断定」「日付は意味を確認」「出典は原典まで遡る」── どれも常に効かせたい品質規律に見える。

memory 残置でも効く

4本とも呼び出す文脈が明確(出典を辿る時・存在を断定する時)。常時 rules に置かずとも、その作業の時だけ memory から呼べば規律は機能する。

対象は no-fabricating / search-all-projects / verify-date-source / verify-source-via-pdf の4本。常時注入のコストを避けるなら memory 残置が理にかなう。

判断: memory 残置寄り(呼出文脈が明確) 要判断
C
規律 · 自己言及の逆説

11本を全部 rules に移すと…

未明文の規律11本を「常時読まれる rules へ全部移そう」とすると、奇妙なことが起こる。その11本の中の1本 ── feedback-prefer-skill-over-rules-injection(表で ★ を付けた本)── が、まさにそれを戒めている。

逆説

「たまにしか使わぬ知識を rules(常時注入)に積むと、システムプロンプトが肥大して注意が散漫になる」。11本全部を rules に移せば、この規律自身が警告する常時ロードの肥大を招く。規律を守るために規律を破る形じゃ。

判断: critical-thinking に1-2行で凝縮 + 一部は memory 残置 設計判断 ◆
