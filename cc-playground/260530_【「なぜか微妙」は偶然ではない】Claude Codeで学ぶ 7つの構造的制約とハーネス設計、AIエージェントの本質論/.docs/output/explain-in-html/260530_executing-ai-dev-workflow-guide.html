調査からPRまでを
一本のレールにする

executing-ai-development-workflow は、思いつきの実装を「調べる → 計画する → 実装する → 検証する → PR を出す」という決まった順路に乗せるスキル。Light モードで素早く、Full モードで品質重視。場面に応じて重さを切り替えられる。

8 Steps (Full) Light は5ステップに短縮
2 Modes 軽量 Light / 品質 Full
4+2 Reviewers 独立した複数視点で点検
1→3 Human Gates 人間が判断を挟む回数

このスキルの正体を一言でいうと 「開発の段取りを毎回ブレさせない仕組み」。AI は同じ依頼でも、ある日は調査を省いたり、ある日はいきなり実装に走ったりと、進め方がブレやすい。このスキルはその進め方を 毎回同じ順路に固定する。

順路は大きく7つ — 調査 (Research)、計画 (Planning)、Issue とブランチ作成、実装 (Implementation)、多層レビュー (Review)、優先度ベースの修正 (Fix)、PR 作成。最後にレビュー対応とマージが続く。各ステップには「次に進む前に人間がOKを出す関所」が要所だけ置かれている。

// PART · 01

このスキルは何をするのか

前のやりとりで出てきた指揮者スキル (orchestrating-team-development) との違いを先に言っておくね。あちらは 複数のエージェントをチームで並走させる重装備。こちらは 1人 (メインの Claude) が順路に沿って一気通貫で開発する軽量レール。チームを組むほどではない、でも行き当たりばったりにもしたくない — その中間を埋めるのがこのスキルだよ。

  • 調査を必ず先にやる: いきなり書き始めず、既存の似た作りを Explore (コードベース探索エージェント) で調べてから設計する
  • 計画を一度ファイルに残す: _docs/plans/ に計画書を保存し、人間の承認を得てから実装に入る
  • 検証を仕組みで挟む: Full モードでは、生成したコードを別々の視点のレビュアーに点検させてから PR を出す
良いコードを祈るのではなく、良い順路を通れば自然に良くなる — 段取りを仕組みにするのがこのスキルの value。 executing-ai-development-workflow / 設計思想
// PART · 02

2つのモード — Light と Full

デフォルトは Light モード。素早さ優先で、調査 → 計画 → Issue/ブランチ → 実装 → PR の5ステップだけを通る。多層レビューは省く。日常の小さな機能追加はこれで十分。

一方、依頼の中に「重い」サインがあると Full モードに切り替わる。レビューと修正のステップが追加され、人間の確認関所も増える。下の2列で見比べてみて。

Light モード (既定)

順路: 1 → 2 → 3 → 4 → 7
調査・計画・実装・PR作成の最短ルート。
人間の関所は 1回 (計画の承認だけ)。
多層レビューなし。素早く回したい時。

Full モード (品質重視)

順路: 1 → 2 → 3 → 4 → 5 → 6 → 7 → 8
レビュー (Step 5)・修正 (Step 6)・PR対応 (Step 8) を追加。
人間の関所は 3回。
セキュリティや大規模変更の時。

どんな時に Full へ切り替わるか。次のどれか1つでも当てはまると Full になる仕組みだよ。

Full に切り替わる条件具体例
セキュリティに関わる機能認証・決済・権限管理・暗号化
10ファイル以上の変更見込み大規模リファクタリング
ユーザーが明示的に品質重視を指定「品質重視で」「多層レビューで」「Full mode で」
外部公開APIの変更互換性を壊す変更を含む API
データベース構造の変更マイグレーション (データ移行)

言い換えると 「壊れたときの被害が大きいもの」ほど Full。被害半径が大きい変更に手厚い検証を当てる、という考え方だね。

// PART · 03

8ステップの流れ

Full モードの全行程

Light モードは Step 5・6・8 を飛ばして 1→2→3→4→7 を通る

Step 1

Research (調査・探索)

コードベース探索エージェント (Explore) で既存の似た作り・設計の決まり・テストのやり方・影響範囲を把握する。いきなり書かない。

Step 2

Planning (計画 + 文書化)

調査をもとに実装の段取りを設計し、_docs/plans/ に計画書として保存。ここで人間がOKを出す (関所その1)。Light/Full のどちらで進むかもここで決める。

Step 3

Issue & Branch (記録と隔離)

GitHub の Issue を作り、専用ブランチ (feature/番号-機能名) を切る。本流から隔離した場所で作業する。

Step 4

Implementation (実装)

計画に沿って実装。必要なら専門エージェント (フロント担当・バックエンド担当・テスト担当など) に分けて任せる。テストも書く。

Step 5 · Full

Review (多層レビュー)

4つのレビュアーを同時並列で走らせる — コード品質・セキュリティ・設計・テスト網羅。UI を含むなら UX 観点・デザイン観点を追加。さらに /review と CodeRabbit (外部の自動レビューツール) も併用。

Step 6 · Full

Fix (優先度ベースの修正)

レビュー指摘を重大度で仕分け。重大 (Critical/High) は妥当なものを自動修正、中程度 (Medium) は 人間が修正/保留/却下を判断 (関所その2)、軽微 (Low) は記録だけ。

Step 7

PR Creation (PR作成)

変更をコミットしてプッシュ、PR を作成。Full では作成前に人間が最終確認 (関所その3)。レビュー結果のまとめも PR 本文に載せる。

Step 8 · Full

PR Response & Merge (対応とマージ)

PR 上に付く自動レビューコメントに対応。妥当な指摘だけ直し、誤検知は丁寧に却下。最後は人の目を入れてマージする。

// PART · 04

3つの使用例

スキルに付属する使用例 (examples.md) から、3つの典型パターンを引くね。依頼文の中の言葉でモードが自動で決まるのが分かるはず。

01
Light mode

UIコンポーネントの追加

依頼例:「機能開発ワークフローでユーザープロフィールカードを追加して」

重いサインがないので Light。既存のカード部品 (ProductCard / TeamCard) のパターンを調査 → 設計を _docs/plans/ に保存 → 承認 → Issue/ブランチ → 実装+テスト → PR作成。レビューのステップは通らず、最短で仕上がる。

02
Full mode

認証機能 (セキュリティ重視)

依頼例:「AI開発ワークフローでユーザー認証機能を品質重視で実装して」

「品質重視」+「認証」の二重サインで Full 確定。実装を3つの担当 (バックエンド/フロント/テスト) に分担。Step 5 で4レビュアーが並列点検し、SQLi (データベースへの不正注入) や XSS (悪意あるスクリプト埋め込み) を検出。Step 6 で重大な指摘 (パスワード未ハッシュ化など) を自動修正、中程度は人間判断。

03
Full mode

DBクエリの大規模リファクタリング

依頼例:「データベースクエリの大規模リファクタリングをAI開発ワークフローで」

「大規模」「リファクタリング」+ 15ファイル変更見込みで Full。N+1問題 (1件取得するたびに追加クエリが飛んで遅くなる典型的な性能問題) を特定し、インデックス追加・クエリ最適化・キャッシュ層を計画。設計・性能の観点でレビューしてから段階的に修正していく。

// PART · 05

レビュー指摘の優先度

Full モードのレビューで出た指摘は、重大度で4段階に仕分けされる (config.json で定義)。重いものは自動で直し、軽いものは記録だけ。すべてを機械的に直すのではなく、人間が判断する層をちゃんと残しているのがポイント。

優先度内容対応
Criticalセキュリティ脆弱性・重大バグ (不正注入・データ漏洩など)妥当なものを即自動修正
High性能・設計の問題 (メモリリーク・N+1・競合状態)妥当なものを即自動修正
Mediumコード品質・ベストプラクティス (重複・複雑さ・エラー処理)人間が妥当性を判断
Lowスタイル・命名・ドキュメント (書式・タイポ)記録のみ

そして大事な歯止めが2つ仕込まれている。1つは 「インフレ禁止」 — 全項目が満点だったら、レビューが甘いことを疑う。少なくとも1つは「No」が出るのが健全、という考え方。もう1つは 「フレッシュコンテキスト原則」 — レビュアーはそれぞれ独立した状態で起動する。同じ記憶を共有すると評価が仲良く甘い方へ収束する (専門用語でポジティビティバイアス) ので、それを構造で防いでいる。論文の二重盲検 (お互いを知らずに審査する方式) と同じ力学だよ。

// PART · 06

引数なしで呼ぶとどうなるか

2つめの質問 ——「単純に /executing-ai-development-workflow とだけ打つとどうなる?」への答え。結論から言うね。

起きること

スキルの手順書 (このガイドで説明した8ステップの設計図) が読み込まれる。そして既定の Light モードで構える。つまり「開発の順路レールが敷かれた」状態になる。

起きないこと

実装は始まらない。Step 1 の調査は「何を作るか」が決まって初めて動く。題材 (作りたい機能) を渡していないので、レールはあっても列車が発車できない。

なぜそうなるか。このスキルの最初のステップ Research は 「実装したい機能を起点にコードベースを調べる」工程だから。題材がなければ調べる対象が決まらない。スキルの構造上、開発対象は 必須の入力なんだ。

だから引数なしで呼んだ時の自然な挙動はこうなる —— Claude が 「何を実装しますか / どんな機能を開発しますか?」と題材を尋ねてくる。それを答えて初めて Step 1 の調査が回り出す。レールだけ敷いて「で、何を運ぶ?」と聞いてくる、というイメージだね。

  • 正しい呼び方: /executing-ai-development-workflow ユーザー認証を品質重視で実装して のように、題材を一緒に渡す
  • 引数なしでも害はない: 手順書が読まれるだけ。勝手にコードを書き始めたり、ブランチを切ったりはしない (計画承認の関所が必ず手前にある)
補足 · 検証範囲

上記はスキルの手順書 (SKILL.md) の構造から確実に言える挙動。「題材を渡さないと Step 1 が始まらない」はスキル設計上の必然。ただし「実際にターミナルで打って画面で確認した」観測ではなく、設計からの導出である点は正直に明記しておくね。

レールを敷くスキルであって、行き先を決めるスキルではない。だから題材を渡して初めて走り出す。
