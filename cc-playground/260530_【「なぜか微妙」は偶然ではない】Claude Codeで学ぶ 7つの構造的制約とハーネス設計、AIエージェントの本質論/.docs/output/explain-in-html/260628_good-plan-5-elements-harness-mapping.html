01良い計画の5つの必須要素とは

LLM のセッションは ステートレス (記憶を保持せず、セッション間で前回の文脈がゼロになる) じゃ。新しいセッションは「何も知らない新しい同僚」と同じ。だから計画は「あとよろしく」だけの TODO リスト ではなく、それ単体で再開・実行・検証できる 自己完結した引き継ぎ書 (実行可能ドキュメント) にせよ、というのが T-2.1 の主張じゃ。

5 つの要素は、それぞれ「TODO リストが落としてしまう問い」と「その要素が無いときに起きる事故」に一対一で対応しておる。

要素答える問い不在時に起きる事故
1. 目的何を達成するかドリフト (目標からの逸脱) の起点。手段が目的化する
2. 前提何が必要か (触るファイル / 環境 / 依存 / 参照実装)存在しないファイルを参照する、環境差で失敗する
3. 判断理由なぜこの方法か (却下した代替案を含む)次のセッションが別方式で再実装し、矛盾が生まれる
4. 検証コマンドどう確認するか (各ステップに exit 0 等の機械的合否)早期完了宣言 (C-4) を検出できない
5. 進捗どこまで終わったか (チェックボックス + commit hash + 現在地)同じ作業の重複実行、または未完了の見落とし

裏付けとして記事はこう引く — Steinberger 「計画ファイルを 再起動可能な状態 (restartable state) として設計せよ」、Ronacher 「5 ステップを超えるなら計画フェーズが必要」。

02あなたのハーネスでの担当マッピング

まず訂正する — 担当は「plan か handoff か」の二項ではない。記事が 1 つの 自己完結ドキュメントに込める 5 要素を、あなたのハーネスは spec (仕様) / ticket (three-elements) / handoff / build-test-protocol という 4 つの第一級の器に役割分割して実装しておる。plan の主務は「判断文脈の保持」じゃ。各担当は現物を確認した検証済みの対応ぞ。

要素主担当 (第一級の器)判定
1. 目的spec ## 背景・目的 ＋ ticket original_goal (書換禁止 = T-2.2 目標保全)専用器あり (二重)
2. 前提spec ## 技術仕様 / 既存との統合ポイント ＋ ticket depends_on ・ ## 参照専用器あり (分散)
3. 判断理由knowledge ## 判断根拠 / トレードオフ (却下理由を書く唯一の見出し) ＋ plan (判断文脈)部分的 = 最弱
4. 検証コマンドbuild-test-protocol (検証不可 = 出荷不可) ＋ ticket の 2 層オラクル ＋ spec Acceptance Criteria専用器あり
5. 進捗handoff frontmatter (session_commits / status / last_completed_phase / next_phase / ahead_count)専用器あり

補足: 要素4 の「2 層オラクル (合否判定の基準)」とは、ticket が持つ behavior_oracle (挙動の合否 = unit / e2e / integration テスト) と structural_oracle (構造の合否) の 2 つで、両方 pass して初めて done に遷移 する仕組みじゃ。要素5 の commit hash と現在地は handoff の frontmatter (ファイル冒頭のメタデータ領域) が、チェックボックスは spec / ticket 側が担う。

03唯一の弱点 — 要素3「判断理由」

WARN判断理由 (却下した代替案) だけが第一級の器を持たない

現状却下案を書く専用見出しは knowledge テンプレの ## 判断根拠 / トレードオフ (逐語: 「選択した理由。検討した代替案とその却下理由」) に部分的に在る。plan も「git log では実装は追えても動機は追えない」唯一の判断文脈の置き場として補う。
不足独立した第一級の ADR (アーキテクチャ決定記録。なぜその設計にしたかを残す文書) 専用の器 — 専用ディレクトリ + テンプレ + 検証スクリプト — は無い。2026-05-30 の gap-analysis K-2.2 の記録と現状は一致しておる。
誤解されがちdesigning-dd は観点が違う (UI 成果物を先に作って実装を駆動するプロセスであって、却下案の記録器ではない)。record-design-decision.py は判断根拠を必須化するが、font / color / layout の美学判断専用で射程が狭い。
補強策ADR 専用の器を新設するか、最低限 plan / spec のテンプレに「判断理由・却下案」を 必須見出し として昇格させる。

045要素に「無い」上乗せの防御層

逆に、あなたのハーネスは記事の 5 要素を 超える ものまで備えておる。これらは記事「7 つの構造的制約」由来の防御を、計画の最低ラインの上に重ねたものじゃ。

  • lineage (来歴追跡) — KPI → Story → Spec → Ticket を双方向リンクで繋ぎ、snapshot 比較で drift (目標逸脱) を検知する。「目的の trace」を 5 要素より厳格にした上位機構。
  • 権限分離 — assignee ≠ reviewer を done 遷移時に検証スクリプトで強制。評価と実装を別の手に分ける信頼境界。
  • 構造オラクル — 機能が動くか (挙動合格) だけでなく、責務境界・テスト可能性・ファイルスコープ規律 (構造合格) まで done の条件にする。
  • 出所追跡・改竄検知 — 外部由来の goal に verified_by / verified_at / integrity_hash を持たせ、前提 (要素2) の出所と改竄を追える。
  • blockers / self-eval メタ — 停止理由の構造化配列と、自己評価結果 (GO / CONDITIONAL / NO-GO + 件数) を handoff frontmatter に機械可読で残す。

05総評

良い計画の 5 要素のうち、目的・前提・検証・進捗の 4 つは spec / ticket / handoff / build-test-protocol という第一級の専用器を持つ。構造的に最弱なのは要素3「判断理由・却下案」ただ一つで、独立した ADR の器が無く knowledge テンプレの一節と plan / log で代替している。一方であなたのハーネスは、lineage・drift 検知・権限分離・構造オラクル・出所追跡という、記事の 7 制約由来の防御層を 5 要素の上に重ねて持つ。記事の最低ラインを満たした上で、さらに堅い造りになっておる。補強するなら、一手は要素3 の ADR 専用器じゃ。
