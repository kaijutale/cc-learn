03各原則の詳細

思想○ 数値✗K-2.1 ポインタは百科事典より強い

記事CLAUDE.md を60行以下のポインタ (目次カード) 型にする。詳細は Progressive Disclosure (段階開示 = 必要な時だけ読み込む) でポインタ先に置く。狙いは帯域の常時圧迫 (C-1) の回避。
実装rules/ の5本が詳細への導線を持ち、本体は .docs/progressive-disclosure/ に外置き。ポインタの実在は hook (特定操作に自動で割り込む小さなプログラム、hook_validate_claudemd.sh) が機械検証し、リンク切れを検出する。
ギャップCLAUDE.md は実測 112行 (60行目安の約1.87倍)。ただし中身は憲法 (人格・口調・応答規約) の本文 + 詳細ポインタのハイブリッドで、記事が警告する500〜1000行の百科事典型病理には陥っていない。

記事超えK-2.2 ドキュメントは構造化する

記事3層ドキュメント構造 (原則 / 戦術=ADR / 実行=SKILL.md) に分け、判断理由と却下案を永続化する。
実装knowledge-categories.md が分類の正本、frontmatter (ファイル先頭の機械可読メタデータ) で型を固定。validate-knowledge.py が pre-commit (コミット直前に走る検証) で列挙値・循環依存・ADR却下案の非空を機械検証する。
差記事は「構造化された文書を残せ」まで。ハーネスは検証を機械強制する段階まで進めている。

記事超えK-2.3 観測面を設計する

記事ファイル / コマンド / ログ / API / UI の5チャネルで世界を観測可能にする。観測できないものは制御できない (C-4 自己申告への対策)。
実装5チャネルを全カバーした上で、fire-report hook が「hook 自体が今月何回鳴ったか」を観測するメタ観測 (観測する仕組み自体の観測)、heartbeat で「静か = 平和か観測不能か」を判別、statusLine で使用状況を常時可視化。
差観測面そのものを観測する二階の仕組みまで到達している。

記事超えK-2.4 ルールは設定ファイルに分ける

記事制御方針をコードに埋め込まず、設定ファイル (YAML 等) へ外部化する。差分表示で方針の違いが見え、別プロジェクトへの移植も容易になる。
実装settings.json が hooks / permissions / env を分離、hooks/rules/*.json がルールをデータ化 (処理ロジックと分離)。さらにそのルールJSON自体を permissions.deny (操作を構造的に禁止する設定) で改竄禁止にする。
差外部化した器そのものを保護するメタ保護まで到達している。

04唯一のギャップ — CLAUDE.md 112行をどう見るか

4原則で唯一 amber なのは K-2.1 の CLAUDE.md 行数 (112行 vs 60行目安)。だが無理に60行へ削るべきではない。理由は3つ。

  • 百科事典型ではない — 112行は憲法本文 + ポインタのハイブリッド。記事が警告する500〜1000行の止まらない肥大とは別物。
  • 常時効かせたい核 — 人格・応答構造はポインタ先へ追いやると遅延ロードで効かなくなる。本文に置く合理性がある。
  • 数値より本質 — 「60行」は目安にすぎない。見るべきは「肥大化していないか」であり、肥大の兆しが出た時が外出しの合図。
