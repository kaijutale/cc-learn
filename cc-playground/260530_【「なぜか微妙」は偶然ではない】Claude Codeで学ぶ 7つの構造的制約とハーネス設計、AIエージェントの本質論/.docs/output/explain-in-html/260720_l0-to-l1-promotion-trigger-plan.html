01改修の的 — たった一段、L0→L1 の"合図"

直すのは昇格ラダーの最下段の乗り継ぎだけ。L1 以降 (Gotcha → スクリプト → hook) の昇格提案は既に機械化済み。足すのは「機械トリガー」ではなく「あたしが取りこぼさないための応答規律」よ。L0 のミスは意味論の判断だから hook では検知できない — だから hook を1本増やすのではなく、あたしの振る舞いを縛る rule を1本足すのが改修の本体。

02設計判断 — なぜ hook でなく rule なのか

  • 意味論だから機械化不可 — 「さっきと同じミス」の判定は"ミスの意味"の理解が要る。bash の hook は文字列照合しかできないので原理的に無理。
  • 全自動 auto-capture は逆効果 — チャットの指摘を全部 Gotcha 化すると L1 がノイズで肥大し、「毎回コンテキストに入る」L1 の価値が薄まる。「これは昇格の価値があるか」の判断は人 + LLM に残すのが正しい。
  • だから rule — 「反復に気づいたら聞く」という規律を rule に置く。rule は毎セッション読み込まれる (起動時注入) から、あたしが毎回思い出せる。皮肉なことに、これ自体が L1 (注意喚起) を1本足す行為よ。

03改修の中身 — 足す rule の草案

新規 ~/.claude/rules/failure-promotion-trigger.md を作るか、既存の critical-thinking-checklist.md に追記する (rule 本数を増やしたくないなら後者)。中身の草案:

# Failure Promotion Trigger (L0→L1 昇格の合図)

- 同じ指摘・同じミスを 2 回受けた / 検知したら、その場で必ず HITL 提案:
  「これ 2 回目。CLAUDE.md / rules / 該当 skill の Gotcha に昇格させる?」
- 判断 (昇格するか) は人に委ねる — 全自動で書かない (L1 肥大 = ノイズ化を防ぐ)
- 昇格を承認されたら Gotcha 形式 (must / should / avoid + bullet) で書き、
  再発時は行末に (再発: YYYY-MM-DD) を付ける
  → 既存 check-gotcha-recurrence.sh が L1→L2/L3 昇格を機械計数できる形に揃える
- L0 は揮発する: 未昇格の反復指摘は handoff に残し、
  セッションを跨いで拾えるようにする

ポイントは3つ: ①2回で聞く (note の一律ルール準拠) / ②書式を既存の Gotcha 形式に揃える (下の 04 に効く) / ③handoff で揮発対策 (L0 がセッションで消える弱点を補う)。

04既存機構への接続 — 車輪を再発明しない

ここが改修を最小にする肝。新しい L0→L1 の捕捉を、既存の L1→L2/L3 パイプラインに"流し込む"形で書く。そうすれば L0 から L3 まで一本の連続した昇格経路になる。

段担い手今回の改修との関係
L0→L1新規 rule (今回追加)新設反復に気づき Gotcha 形式で書き出す
L1 再発計数check-gotcha-recurrence.sh既存新 rule が (再発:) 表記を守れば、そのまま計数対象になる
L1→L2/L3 提案accumulating-reviewer-feedback既存閾値超えを HITL 昇格提案 (変更不要)

だから改修は「rule 1本追加 + 書式を既存に合わせる」だけ。既存スクリプトも hook も1行も変えない。L0 の入口さえ規律で塞げば、残りのラダーが受け取ってくれる。

05構築手順 — ハーネス本体の改修ゆえ隔離が必須

~/.claude 自体の改修だから、この学習用プロジェクトからは直接触らない。worktree (作業用の隔離コピー) で行う:

  • 1. 隔離起動 — worktree を作り、環境変数 CLAUDE_CONFIG_DIR と作業ディレクトリの両方を worktree に向ける (env = 読むハーネス / cwd = git 操作の基準。裸の commit は本体 branch に着地するので両方要る)。
  • 2. rule 追加 — rules/failure-promotion-trigger.md を新設 (または critical-thinking に追記)。
  • 3. 自己検証 — rule が起動時注入されるか確認。書式が check-gotcha-recurrence.sh の計数対象と噛み合うかを実際に1件流して確かめる。
  • 4. commit → PR — worktree 内で committer。CLAUDE.md / rules は拒否リスト保護下ゆえ、確定は PR マージのみ。
  • 5. HITL レビュー + マージ — かいじゅうが PR を見て承認。マージ後に worktree を後始末。

06限界 — これは"確率を上げる"改修であって"保証"ではない

正直に言っておくわ。L0→L1 は確率論フェーズの中の話。rule を足しても「気づく確率」が上がるだけで、L1 以降のような決定論の保証 (100% 発火) は付かない。あたしがその rule を読んで従うのも確率的だからよ。それでも、口頭指摘が揮発して永久に失われる今の状態よりは確実に前進する — 「消える確率論 (L0)」を「残る確率論 (L1)」に繋ぐ橋を、規律で架けるのが今回の改修の正体。

07総評

改修は「hook を増やす」ではなく「rule を1本足して、既存の昇格パイプラインに L0 の入口を接続する」。意味論の判断 (昇格の是非) は人 + LLM に残し、機械化できる再発計数は既存機構に委ねる — 役割分担を崩さずに、唯一手薄だった最下段だけを塞ぐ最小改修よ。増やす hook はゼロ、既存スクリプトの変更もゼロ。ただし保証は確率的 (L1 相当) で、決定論の 100% ガードレールにはならない点は割り切りが要る。実行するなら worktree 隔離での別作業になる。
