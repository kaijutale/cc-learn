01何を変えたか（要旨）

技術記事「7つの構造的制約」(260405) の パターン2「Phase 分離」は、開発を「調査 → 設計 → 実装 → 検証」の4つのフェーズに分け、各フェーズは前フェーズの成果物ファイルだけを読む、という関心の分離を説く。現ハーネスはこのうち Phase 0（調査 → research.md）だけが欠けていた。

修正前は、重い調査（大規模なコード把握や技術選定）の結果が独立した成果物にならず、仕様書 (spec.md) に畳み込まれるか会話の中に残り続けていた。すると実装フェーズに調査の生データが持ち込まれ、コンテキスト（AI が一度に集中できる作業領域）の集中力が落ちていた。

今回の変更は、重い調査を .docs/research/<topic>.md に外部化し、仕様策定がそれを入力として読む「2段分離」を、重い調査が要るタスクでのみ有効になる opt-in（使う時だけ有効・既定では従来通り）として導入した。

03設計の核心 — research.md の「担い手」は誰か

最大の論点は「research.md を読んで仕様に反映するのは誰か」だった。3案を比較し、案B（オーケストレーターが所有）に確定した。

案担い手判定理由
案Aspec-based-development が読む却下統合判断を末端の skill に押し込む（責務分離に反する）。発火の判断も持てない
案Bオーケストレーターが所有採用発火判断・統合判断 (T-2.3) を中枢が握る。spec-based-development は「読み手」に留める
案C調査駆動設計の新規 skill却下改修ポリシー「新規作成は退避策」に反する + skill の肥大化

つまり 「決める・束ねる」のは中枢のオーケストレーター、「読んで仕様に書く」のは spec-based-development という役割分担にした。調査の生データを背負うのは調査の子エージェントだけで、中枢には要約だけが返る。
