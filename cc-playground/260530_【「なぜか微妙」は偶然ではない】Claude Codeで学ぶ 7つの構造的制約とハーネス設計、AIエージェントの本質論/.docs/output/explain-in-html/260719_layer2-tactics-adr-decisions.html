013層の中での Layer 2 の位置

note K-2.2 は「ドキュメントは捨てるのではなく構造化する」ために、文書を変更頻度で 3 つの層に分ける。Layer 2「戦術」はその真ん中の層で、答える問いは「どう考えるか」。Layer 1 が「滅多に変わらない信念」、Layer 3 が「毎週変わる今日の手順」だとすると、Layer 2 は一度決めたら当分従うが、状況が変われば見直す「判断」を置く場所よ。

02decisions/ とは — 設計判断の記録 (ADR) の置き場

decisions/ は ADR (Architecture Decision Record。設計判断の記録) を連番で置くディレクトリよ。ADR は「何を決めたか」だけでなく「なぜ決めたか」「何を却下したか」まで書き残す文書。たとえるなら裁判の判決文ね — 主文 (結論) だけでなく、判決理由と退けられた主張まで残っているから、後から同じ争点を蒸し返さずに済むの。

03なぜ patterns.md ではなく decisions/ が該当するのか

note をよく読むと、Layer 2 の実装として docs/patterns.md (考え方のパターン集) と ADR の2 つが並記されている。つまり ADR はもともと note 自身が挙げる Layer 2 の実装形の 1 つなのよ。

  • ADR 側 → decisions/ が担う — 「この状況ではこう判断した、理由はこれ」という記録。Layer 2 の核心 (判断とその理由) を専用の器で実装している。
  • パターン集側 → 独立ファイルとしては無い — patterns.md という名前のファイルは実在しない。ただし「どう考えるか」の言語化 (例: 曖昧回答の禁止ルール、保存前に検証する手順) は rules/ 配下などに分散して存在する。
  • 照合は名前でなく機能で行った — K-2.2 の取り入れ確認では「Layer 2 の機能 = 判断の理由の永続化」を満たす器として decisions/ を対応物と判定。名前の一致 (patterns.md があるか) では見ていない。

04Layer 2 はどんな時に使用されるのか

使われる場面は 4 つ。書く・読む・見直す・機械ゲートで覚えるといいわ。

書く分岐点で「決めた」時四半期〜月

「どちらでも動くが、決めないと後で実装が揺れる」選択をした時に書く。例: ADR-0001 の外出し先の選択。逆に、バグ修正や手順の追加は Layer 3 (SKILL.md や hook) の仕事で、ADR は書かない。毎セッション書くものではない — 実際 decisions/ は 2026-07-10 に始動してまだ 2 本。判断が起きた時だけ増えるのが正常で、それが「四半期〜月単位」という頻度の意味よ。

読む同じ論点が再来した時

後のセッションや別のエージェントが過去の判断と矛盾する変更を提案した時、ADR を参照して却下できる。note の例: 「セッション管理を PostgreSQL に移行しましょう」という提案が来ても、ADR に「PostgreSQL は検討済みで、この理由で却下した」とあれば議論を最初からやり直さない。過去の失敗を繰り返さないための読み取り面がここ。

見直す撤退条件が観測された時

ADR の「撤退条件」に書いた観測値・事象が実際に出たら、status を superseded (置き換え済み) にして新しい ADR を書く。決定は「不変」ではなく、統制された手続きでだけ変わる。過去の判断を黙って上書きせず、置き換えの系譜 (どの判断がどの判断を置き換えたか) が残る。

自動commit の瞬間 (機械ゲート)

人が意識しなくても、decisions/ に触れる commit の瞬間に検証スクリプトが走り、却下代替案が未記入の ADR はブロックされる。「書く時に理由を残す」という規律を、散文の約束ではなく機構で強制している。

書くタイミングの目安を 1 行で言うなら: 「あとから『なんでこうなってるんだっけ?』と聞きたくなりそうな選択をしたら、それが ADR を書く合図」よ。

05総評

Layer 2「戦術」は、未来の自分とエージェントに宛てた判断の理由書を置く層。Layer 1 (滅多に変わらない信念) と Layer 3 (毎週変わる手順) の間で、「決めたこと・その理由・捨てた選択肢」を四半期〜月のリズムで残す。かいじゅうのハーネスではその器が decisions/ (ADR) で、判断の心臓部である却下理由の記入を hook が機械強制している点、そして撤退条件まで書かせる点で、note の推奨より一歩先の実装になっている。
