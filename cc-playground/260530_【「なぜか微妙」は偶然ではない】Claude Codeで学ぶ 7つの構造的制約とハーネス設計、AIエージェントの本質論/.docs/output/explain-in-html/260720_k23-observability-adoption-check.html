03原則の実装 — 「配備≠実効」の実痛から学んだ

面白いのは、このハーネスが「観測できないものは制御できない」を教科書からでなく実痛から学んでいること。07-16 の観測レビューで、「配備した = 動いてる」という思い込みが 3 クラス同時に崩れた: sandbox 設定202本が実は死んでいた / 防御設定 (deny) 7本が不発だった / hook 通知 9本が沈黙死していた。どれも「配備の記録はあるが、実効を観測する面が無かった」ことが原因よ。

その対策として作られたのが 02 の liveness 証跡と heartbeat。発火レポート hook のコメントには自己記録がこう残ってる:

「旧実装は jq/aggregator 不在を沈黙 skip しており、
 それ自体が #68 の指摘した無音死の盲点だった」

note の Before/After (「動くはずです」→ 実測で答える) も、ルール「状態変更は成功メッセージを信じず、独立確認 (コミットIDの変化・ファイル存在・件数) で着地を検証してから完了報告する」として規範化済み。

超過1二階の観測面 — 観測装置の死を観測する

note は「観測面を作れ」まで。ハーネスは heartbeat (死活監視) で「観測面そのものが死んだこと」をセッション開始時に検出する。「集計ゼロ = 平和」と「集計ゼロ = センサー死亡」を機械的に区別する装置で、死亡報告の抑制タイマーを定例報告と分ける理由 (共有すると死亡が沈黙する盲窓が生まれる) まで設計コメントに明文化されてる。

超過2発火数の意味論 — 数字を目標にしない規律

月次レポートは「rule X が今月 N 回発火 = 自然言語の注意喚起が効いていないサイン」として上流の改修判断材料を機械供給する。同時に「多発火を昇格シグナルと混同しない」と明文化 — 指標を目標にすると指標が壊れる (Goodhart の法則) への警戒が組み込み済み。

超過3観測面の自己汚染防止

台帳が検出した禁止語の本文を画面に出すと、それがセッションの文脈に入って別の hook を誤発火させる — 「観測装置が観測対象を汚す」事故を、報告は hook 名と件数のみに限定する設計で予防してる。
