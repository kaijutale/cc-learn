01まず結論 — 2つの問いへの答え

問い1: gitflow は必須ではない。worktree を作る skill(creating-gtr-worktree / launching-gtr-issue-worktree)を複数回起動すれば worktree が複数でき、チーム開発級の並列も技術的には回せる。gitflow を絶対に使わねばならぬ、ということはない(そなたの理解どおり)。

問い2: 協調系で worktree を作る担当は launching-gitflow-worktrees 一つ。中枢の指揮役 orchestrating-team-development が、複数チームを並列で走らせるとき Step 2.5 でこれを呼ぶ。

02問い1 — gitflow は「並列の必須条件」ではない

大事なのは、2 つを別レイヤーで捉えることじゃ:

  • worktree を複数作る — 並列の「手段」。gtr 系を何回起動してもよい。これだけで並列開発は成立する
  • gitflow — 並列の「安全管理」。並列が大規模化したときの事故を防ぐ仕組み

つまり gitflow は「並列を可能にするもの」ではなく「並列が大きくなったときに事故を防ぐもの」。worktree 複数 = 並列の実現、gitflow = 並列の安全運用じゃ。

03gitflow を使わないと「手動」になるもの

gtr 系の複数起動だけで並列する場合、gitflow が自動でやってくれる安全装置が抜ける。その分は自分で気をつける必要が出る:

gitflow が自動でやること無いとどうなるか
複数 feature を main 前に develop で一旦統合・検証する中間層各 feature が main へ直接マージされ、複数成果物を束ねた統合状態を本番前に検証する場が無い(専用ブランチ + PR レビューで個別検証は効くので、実害が出るのは大規模並列時)
pre-push hook で main/develop への push をブロック誤って本番ブランチへ push しないよう自分で注意
current-team.md でチームの担当情報を注入各 worktree に担当範囲・他チーム情報を手動で渡す
誤 merge(誤統合)の多層防止統合を慎重に手動管理

04規模での使い分け

規模推奨理由
小規模gtr 系を複数起動独立タスクが少数で main を同時に触らないなら、gitflow の階層は過剰。軽く済む
大規模launching-gitflow-worktrees複数チームが同じ develop に向かい main を守りたいなら、保護・緩衝・誤マージ防止の価値が出る

結論 ── gitflow は並列の必須条件ではなく、大規模並列の安全オプションじゃ。

05問い2 — 協調系で worktree を作るのは誰か

中枢の指揮役 orchestrating-team-development(複数の専門エージェントを束ねる L3 指揮者)の一次記述より、担当は 1 つに定まる:

担当launching-gitflow-worktrees

呼ぶ場所orchestrating-team-development の Step 2.5
発動条件複数チームが並列作業する場合
根拠「launching-gitflow-worktrees で gitflow + worktree 環境を構築」+ 接続スキル群にこれだけ載る(creating / issue 系は無い)

06重要な区別 — context:fork は worktree ではない

協調系でも、すべてが worktree を作るわけではない。coder / enforcing-strict-tdd-cycle / llm-debate などは context:fork(新しい会話コンテキストを分岐する仕組み)で隔離する ── これは worktree とは別物じゃ:

隔離手段何を分けるか使う skill の例
context:fork会話履歴(コンテキスト)の分離coder / enforcing-strict-tdd-cycle / llm-debate
worktreeファイル(作業ディレクトリ)の分離launching-gitflow-worktrees

協調系で「ファイルの worktree」を作るのは launching-gitflow-worktrees ただ一つ。他は「会話の分岐」で隔離しておる ── ここを混同すると、どの skill が何を分けているか見失う。

07総評 — 2つの問いの接点

gitflow は技術的に必須ではない(問い1)。だが、ハーネスのチーム協調 skill はあえて gitflow を選んでいる(問い2)。これは矛盾ではない ── チーム協調級の並列では、main 直結禁止・誤マージ防止という安全管理の価値が、gitflow のひと手間を上回るからじゃ。小さく回すなら gtr 系の複数起動で軽く済ませてよい。規模で選ぶ ── それが答えじゃ。
