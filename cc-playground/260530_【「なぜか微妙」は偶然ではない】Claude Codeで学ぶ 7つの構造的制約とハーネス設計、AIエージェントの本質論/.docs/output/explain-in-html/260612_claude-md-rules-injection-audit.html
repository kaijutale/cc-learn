前提となる実機事実 — 記憶ではなく実機出力のみ

本監査の判定はすべて、ls (ファイル一覧) / wc (バイト数計測) / grep (文字列検索) の実機出力と、本セッションへ実際に注入されたシステムテキストの突き合わせに基づく。

  • 8 本の rules すべてに frontmatter (ファイル先頭のメタ情報ブロック) がない — paths 指定による遅延読み込みは効いておらず、毎セッション全文が注入される。本セッションのシステム注入に 8 本全文が載っていることを確認済み。
  • 参照先はほぼ生きている — progressive disclosure (必要な時にだけ詳細を読み込ませる遅延開示) 先の 4 ドキュメントと hook_stop_words_rules.json はすべて実在。死んでいるのは citation-format.md 内の例パス 2 本のみ (前段の監査で確定済み)。
  • Secrets L1 の機械ブロックは監査中に実演された — 「.env」という文字列を含む検査コマンドが hook (操作を検査して止める自動チェック) に拒否された。L1 の記述は飾りではなく、実在する壁の説明書である。
  • 構造的発見 — rules が常時自動注入されるのに、CLAUDE.md 側に「reference: rules/X.md」とだけ書くポインタ専用の節が 4 つ残存。これは「rules は遅延ロードされる」という旧理解 (6/8 の大掃除で否定済み、auto-memory に記録あり) の名残で、Claude にとって情報量ゼロの純重複。
「ポインタの指す先が、既に毎セッション手元に届いておる。なら、そのポインタは空気じゃ」 — 本監査の核心
