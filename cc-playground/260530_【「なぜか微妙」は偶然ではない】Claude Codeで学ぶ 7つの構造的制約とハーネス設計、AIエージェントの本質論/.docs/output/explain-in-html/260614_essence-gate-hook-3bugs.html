043 つのバグはこう連鎖する

個々のバグも問題じゃが、組み合わさるとどちらに転んでも壊れるのが核心。これが「単なる書き間違い」を超えて深刻な理由じゃ。

  • いまの状態(バグ 1 + 2) — グロブが 5/14 を拾い、その前置き文から「Critical 1 / High 3」を誤抽出する。結果、いま ~/.claude で skill・agent・hook を含む commit が誤った理由で止められる。セッション開始時に出た「未 commit 7 件」に該当ファイルがあれば、すでに弾かれている可能性がある。
  • 仮にバグ 1 だけ直すと(バグ 3 が表面化) — グロブが最新 6/12 を拾えるようになるが、今度は抽出が 0/0 になり、High 1 件の不適格を誤って通してしまう。片方を直すと、もう片方の故障が顔を出す。
  • 結論 — 正しい commit を止め(止めすぎ)、不適格な commit を通す(通しすぎ)。両方向に誤るゲートは、もはやゲートとして信頼できない。
  • 根本原因 — バグ 2 と 3 は同じ根を持つ。合否を、決まった形式の項目ではなく「人間が読む散文を検索して抜こうとしている」こと。レポートの書式が少し変わるだけで、誤抽出(バグ 2)も抽出失敗(バグ 3)も起きる。

06総評

このフックの弱点は個別の書き間違いではなく、「人間が読む散文から機械が合否を抜く」という設計そのものにある。レポートの書式が変わるたびに、止めるべきを通し、通すべきを止める。安全ゲートは厳格に作るほど、壊れたときに静かに無効化されるのが恐ろしい——だからこそ合否は、散文ではなく機械可読の一点に固定すべきだ。
