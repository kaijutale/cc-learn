01S-1.2 — 安全性の判定を自然言語に委ねない

note: 「安全か?」を LLM の印象でなく機械的条件で判定する (「本番か?」→ echo $NODE_ENV / PreToolUse Hook の正規表現で本番操作を物理ブロック)。LLM の安全判定は迎合性 (C-3)・自己確信 (C-7) で不安定だから。

取り入れ済みhook 群が字句・exit code で機械判定

実体hook_pre_commands が 9 rule の正規表現/固定文字列で curl/npm/秘密読取/一括置換/find-delete 等を字句 deny。read_secret はパス一致で fail-closed (入力異常時も安全側へ)。hardcode_hygiene は machine 固有パスを block (「確率的→決定論的」)。verify-adr / assert-* 4本 が exit 0/1/2 で合否を返す。

記事超え機械化の「限界」も明文化した

核心note は「機械化せよ」まで。ハーネスは、どこまで機械化しどこを意味論に残すかを自覚している ── probe-before-persist L5「内容の真偽は機械判定しない (意味論判断ゆえ)」。fail-closed (防御系) と fail-open (advice 系) の使い分け、Keychain 削除可否は機械で「承認を通す」までで可否の判断は人間に留保。安全判定を機械化しつつ、意味論だけは機械に委ねない境界線じゃ。
