短い 2 行に見えて、設計判断が濃く詰まっておる:

  • 沈黙契約 — 記録部品は標準出力にも標準エラーにも一切出力しない。hook_stop_words のような hook は標準出力の JSON が判定結果そのものを運ぶため、1 バイトの混入も判定を壊す。呼出側も >/dev/null 2>&1 || true で二重に封じる。
  • fail-open (失敗しても本来の処理を止めない設計) — 記録部品は常に exit 0。台帳が書けなくても hook 本来のブロック判定は無傷。理由は明快で、台帳は観測センサーであって防御ゲートではないから。防御系の「失敗したら閉じる」規約とは意図的に逆に倒しておる。
  • 発火分岐の内側だけで記録 — 問題なしで通過した時は絶対に呼ばない。「記録ゼロ = 発火ゼロ」が集計の前提になる。
  • 受入検証はバイト単位 — 計装前後で hook の標準出力・標準エラーが完全一致することを diff 比較で実測済み (発火経路・通過経路の両方)。

04Sensor B — 未昇格 Gotcha の再発照合 (段階 2.5)

hook を持たない自然言語だけの Gotcha は、破られても何も発火しない — 再発を観測する装置が構造的に無かった。この PR は既存のフィードバック反映 skill accumulating-reviewer-feedback (5 段階フロー: 読込 → 分類 → 人間判断 → 反映 → 記録) に、段階 2.5「再発照合」を純加法で挿入する。既存の入出力契約・呼出方法は一切変えない後方互換改修じゃ。

段階 2.5 の中身は 3 手順:

  • 1. 照合 (LLM の意味判断) — 新しく検出された問題が既存 Gotcha 項目と「同じ失敗の再来」かを、文字列一致でなく根因の同一性で判定する。曖昧なら非再発に倒す (過剰マークは昇格候補のノイズになる)。
  • 2. 再発マーク (事実記録、人間承認は不要) — 再発した Gotcha 行の末尾へ (再発: YYYY-MM-DD) を追記する。「再発が起きた」は観測事実であって判断ではないから HITL を経ない。Edit は「既存行全体 → 同一行 + マーク」の完全一致形式に固定し、本文を書き換える事故を構造的に不可能にする。
  • 3. 昇格候補の決定論抽出 — check-gotcha-recurrence.sh がマーク数を機械計数する (コード例の中や Gotcha セクション外の偽物は数えない)。閾値は既定 1 マーク = 初回失敗 + 再発 1 回 = 通算 2 回目で昇格候補。候補は段階 3 の人間判断 (AskUserQuestion) へ「提案」として持ち込まれる — 自動昇格は実装しない。

マークの見た目はこうなる (例):

- **must**: ファイル名の日付は常にログ記録時刻を使う (理由: ...) (再発: 2026-07-02)

もう 1 つの要が冪等性ゲート (冪等性 = 何度実行しても結果が変わらない性質) じゃ。人間判断の中断や持越しで同じレビュー結果が再処理されると、同じ再発に二重マークが付き通算回数が水増しされる。そこで「この self-eval 記録を処理した痕跡が既に履歴にあるか」を決定論で判定し、再走行では照合・マークを skip して計数だけやり直す。これは後述のレビューで検出された High 指摘への修正でもある。

06設計上の自覚点と follow-up

この PR は自分の限界を文書に残しておる。ここが読みどころじゃ:

  • センサー自身の死は観測されない — 台帳が書けなくなっても fail-open ゆえ無音で止まる。「集計ゼロ」と「センサー死亡」が台帳上区別できない。生存確認 (heartbeat) の仕組みが follow-up 候補 1 号。
  • 発火数の読み方 (Goodhart の法則への防衛) — 指標を目標にすると指標が壊れる。多発火は「上流の自然言語規律が効いていない」サインであって、「発火数を下げる」こと自体を目標化してはならない、と規約に明記された。
  • 展開は 4/12 本 — 計装は issue 明記の 4 hook のみ。残り 8 本の block/warn hook への展開は、共有部品設計により各 2 行 + テストで済む。
  • matched_context は書くだけの非信頼データ — 検知文言は実際の禁止語を含むため、セッションへ引用しない前提で台帳にのみ保持。保持方針の再検討が follow-up 候補。
