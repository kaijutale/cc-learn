01結論 — 3要素は skill が、2要素は指揮者自身の機構が担う

そなたの問い「束ねる skill 達が 5 要素を担っておるか」へ、要素ごとに割って断定する。orchestrating-team-development (チーム開発の指揮者 skill。以下「指揮者」) は、5 要素を 2 つの系統 に振り分けて実装しておる — 委譲先の skill が担う系統 と、指揮者 自身の workflow 状態 (手綱) が担う系統 じゃ。

  • 要素1 目的 — skill が担う (spec-based-development)。YES
  • 要素2 前提 — skill が担う (spec-based-development / conducting-research-phase / designing-dd)。YES
  • 要素3 判断理由 — 専用 skill は無い。指揮者の機構 (Step7 の判断記録) で部分的に代替するのみ。最弱
  • 要素4 検証 — skill が担う (enforcing-strict-tdd-cycle / branch-validator / gep) + 指揮者の機械判定。最も多重。YES (最強)
  • 要素5 進捗 — 専用 skill ではなく、指揮者自身の機構 (共有タスクリスト + フェーズゲート) が担う。YES

つまり — 「skill が担う」が正確に当てはまるのは 目的・前提・検証 の 3 要素。進捗と判断理由の 2 要素は、委譲先 skill ではなく指揮者そのものの状態管理が担っておる。よって「skill 達が 5 要素を担う」は半分しか正しくない — どの半分かは次節以降で具体に割る。

LLM (大規模言語モデル) のセッションは ステートレス (記憶を持たず、セッションが変わると前回の文脈がゼロになる) じゃ。だから計画は「あとよろしく」の TODO リストではなく、それ単体で再開・実行・検証できる 自己完結した引き継ぎ書 にせよ、というのが T-2.1 の主張。その必須要素が以下の 5 つぞ。

要素答える問い不在時に起きる事故

04「skill が担う」の正確な切り分け — 2 系統

そなたの問いへの核心の補正はここじゃ。指揮者は 5 要素を「全部 skill に委譲」しておるのではない。情報の器とゲートを skill に出す系統と、指揮者の状態機械が直接抱える系統に分かれておる。

  • skill が直接担う系統 (3 要素: 目的・前提・検証) — spec-based-development が目的と前提の「器」を、conducting-research-phase / designing-dd が前提の調査・設計を、enforcing-strict-tdd-cycle / branch-validator / gep が検証の「ゲート」を提供する。ここは確かに skill が担っておる。
  • 指揮者自身の手綱が担う系統 (2 要素: 進捗・判断理由) — 共有タスクリスト・フェーズゲート・review-deferred.md は委譲先 skill ではなく、指揮者 (= L3 の Claude 自身) が握る workflow 状態。とりわけ 統合判断は委譲してはならない (T-2.3) と SKILL.md に明示され、進捗の確定も「自己申告を完了の証拠にしない」(C-4) ため指揮者の機械判定に閉じておる。

だから答えは — 3 要素 (目的・前提・検証) は skill が担い、2 要素 (進捗・判断理由) は skill ではなく指揮者自身の機構が担う。「skill 達が担う」は前者には当たり、後者には当たらぬ。

05唯一の弱点 — 要素3「判断理由」

WARN採用案の「却下した代替案」だけが、どの器にも残らない

担っている所Step7 統合で「この指摘は見送る」という見送り判断の理由は review-deferred.md に記録される。指揮者が統合判断を委譲しない原則 (T-2.3) も判断の主体を明確にする。
欠けている所「なぜこの設計を選び、どの代替案を却下したか」という採用側の判断理由を残す専用 skill・専用の器が指揮者の束に無い。spec は What を、テストは合否を残すが、Why の却下系譜は構造化されない。
整合ハーネス全体を見た先行分析 (260628 の whole-harness マッピング) でも、第一級の器を持たぬ唯一の要素は「判断理由」じゃった。指揮者単体で見ても同じ穴が出る — これは偶然でなく、ADR (設計判断記録) 専用器がハーネスに無いことの反映ぞ。
補強策spec / ticket のテンプレに「判断理由・却下案」を必須見出しとして昇格させるか、ADR 専用の器 (ディレクトリ + テンプレ + 検証スクリプト) を新設する。

06見落としやすい前提 — これは計画「文書」でなく「実行器」

もう一つ構造的な補正をしておく。T-2.1 の 5 要素は本来「再起動可能な単一の計画文書」(restartable な 1 枚の引き継ぎ書) が満たすべき中身じゃ。だが指揮者は計画文書を 1 枚書く skill ではなく、実行を回す指揮者 (オーケストレーター)。ゆえに 5 要素を 1 枚に集約せず、spec / research.md / 共有タスクリスト / oracle / review-deferred.md という複数の状態物に分散実装しておる。

  • 利点 — 各要素が専用の器とゲートを持ち、検証 (要素4) を多重化できる。実行時の堅さは単一文書より高い。
  • 代償 — 「1 枚読めば再開できる」restartable 性は分散する。セッション境界をまたぐ再開を担う handoff skill は指揮者の束には含まれない別系統で、指揮者は「実行中の状態」を spec + 共有タスクリスト + ticket に頼って再構成する。

つまり — 指揮者は「良い計画文書を 1 枚書く」のではなく、「良い計画の 5 つの情報役割を、実行パイプラインの状態として満たす」。問いを「skill が 5 要素を担うか」と立てると、この文書 vs 実行器のレイヤー差が見えにくくなる。

07総評

orchestrating-team-development が束ねる skill 群は、良い計画の 5 要素のうち 目的・前提・検証 の 3 要素を直接担う — spec-based-development が目的と前提の器を、conducting-research-phase と designing-dd が前提の調査を、enforcing-strict-tdd-cycle・branch-validator・gep が検証のゲートを提供する。残る 2 要素、進捗 と 判断理由 は、委譲先 skill ではなく指揮者自身の手綱 (共有タスクリスト・フェーズゲート・review-deferred.md と統合判断) が担う。そして 5 要素で構造的に最弱なのは、ハーネス全体で見たときと同じく要素3「判断理由」ただ一つ — 採用案の却下代替案を残す専用器が無い。最後に忘れてはならぬのは、指揮者は計画文書を 1 枚書く skill ではなく実行器であり、5 要素を複数の状態物に分散して満たしておる、ということじゃ。
