# Claude Codeソースコード流出が暴いた「AIの未来設計図」——KAIROSプロジェクトと自律AIエージェントの衝撃

2026年3月31日、AIの世界で前代未聞の出来事が起きた。

AnthropicのAIコーディングツール「Claude Code」の全ソースコード――約512,000行にも及ぶTypeScriptコードが、npmパッケージ経由で世界中に公開されてしまった。しかも、これは2025年2月に同じミスを犯してすでに修正済みだったにもかかわらずの、「2度目の人間的ミス」だった。

「$38bの企業がまさかこんなミスを？」とXは大喜騒状態になり、GitHubにはミラーリポジトリが乱立。わずか数時間で8,800以上のスターが付いた。

でも、私がこの出来事に興奮したのは「流出した」という事実そのものではない。

流出したコードの中に、Anthropicが「次にどこへ向かうのか」という未来の設計図が、克明に刻み込まれていたからだ。

元Amazon Operations Manager、物理学修士として複雑なシステム設計を見続けてきた私の目には、このコードが単なるエラーではなく「AIエージェント時代の本質」を映す鏡に見えた。ビジネスパーソンが今すぐ知っておくべき3つの洞察を、この記事でお伝えする。

## なぜ「2度目」のミスが起きたのか――AIでさえ防げない人間的失敗の構造

まず事実関係を整理しよう。

今回の流出は、ハッキングでも内部告発でもない。Anthropicが公開したnpmパッケージ`@anthropic-ai/claude-code@2.1.88`に、`cli.js.map`というソースマップファイルが混入していただけだ。Claude Codeが使っているBunというランタイムは、デフォルトでソースマップを生成する。明示的にオフにしなければ、自動的にバンドルされてしまう。

「たったそれだけの話」なのに、なぜ2度も繰り返したのか。

Amazonでオペレーションを管理していたとき、私は「人間はシステムが許す失敗を必ず繰り返す」という原則を骨の髄まで叩き込まれた。エラーは個人の注意不足ではなく、構造の問題だ。

Anthropicは世界最高水準のAI安全性を謳う企業でありながら、ビルドパイプラインのデフォルト設定という「構造的な穴」を塞げなかった。これはAnthropicへの批判ではない。どれほど優秀な組織も、システム設計に欠陥があれば失敗するという普遍的な教訓だ。

そして皮肉なことに、この「人間的ミス」こそが、Anthropicが秘密裏に進めていた「人間的ミスをなくすためのAI自律化計画」を白日の下にさらすことになった。

## 流出コードが語るKAIROSプロジェクト――AIが「夢を見る」時代へ

コードを精査した研究者たちが発見したのが、**「KAIROSプロジェクト」**の存在だ（一部では「KAIR OS」とも呼ばれている）。

KAIROSとは古代ギリシャ語で「好機・適切な時」を意味する。このプロジェクト名に、Anthropicの野望が凝縮されている。

### Claude Codeを「常駐・自律・プロアクティブ」な存在へ変革する計画

現在のClaude Codeは、あなたが呼びかけるたびに起動し、タスクが終わると停止する。いわば「呼び出しベルに応答するウェイター」だ。

KAIROSが目指すのは、そのウェイターを「あなたの好みを記憶し、翌日には先回りしてテーブルを準備しておくコンシェルジュ」へと進化させることだ。

コード内に発見された主な機能フラグは以下の通り：

- **セッション間記憶の保持**: 会話が終わっても文脈を維持し続ける
- **KAIR_OS_DREAM（自律記憶整理）**: 24時間ごとにAIが過去の会話を整理・統合・矛盾解消する「夢を見る」機能。300行以上の`autoDream.ts`として実装済み
- **プロアクティブ起動**: ユーザーが指示しなくても自律的にタスクを開始する能力
- **GitHub Webhook連携**: PRに自律的に反応してコードレビューや修正を行う

**「AIが夢を見る」――これはSFではない。**

人間が睡眠中に記憶を固定化・整理するように、AIも「ダウンタイム」に学習を深化させる。この設計思想は、AIを道具から「共に働くパートナー」へと変える根本的な転換点を示している。

## 内部コードネーム「Tengu（天狗）」が示す開発哲学

Claude Codeの内部プロジェクト名は「Tengu（天狗）」だ。日本の妖怪から名付けられたこのコードネームには、単なる遊び心以上の意味がある。

フラグ名は機能推測を防ぐため、`tengu_amber_stoat`（琥珀色のオコジョ）のようにランダムな組み合わせが使われていた。しかしコードの中身を読めば、107個ものFeature Flagが「これだけの問題認識がある」という証拠に他ならない。

**ゴールシークプロンプトの考え方で言えば、Feature Flagの数は「解決すべき課題の地図」だ。**

What（何が問題か）とWhy（なぜ重要か）を107個特定できている組織が、そのHowを一つ一つ実装している――これが「Anthropicの真の強さだ」と私は見ている。

## 「BUDDYシステム」が示す、AIはツールからコンパニオンへ

もう一つ見逃せないのが、4月1日リリース予定とされていた隠し機能「BUDDYシステム」の発見だ。

アヒル、ドラゴン、アホロートル、カピバラなど18種族のAIペットがターミナルの入力欄横に住み、スピーチバブルでコメントしてくれる。レアリティは「コモン」から「レジェンダリー（1%）」まであり、ユーザーIDのハッシュ値によってペットが決定論的に割り当てられる設計になっている。ガチャシステム、手続き生成ステータス、ASCIIアートアニメーション、そしてClaudeが生成する「魂の説明文」まで――エンジニアが本気で作り込んだ、完成度の高いシステムだ。

「CLIツールのイースターエッグにここまでリソースを投じる意味はあるのか」という疑問は的外れだ。

**人間がAIに求めるのは、純粋な機能だけではない。**

日々の仕事に伴走してくれる「存在感」こそが、長期的な利用継続を生む――Anthropicはそれを洞察した上でBUDDYを設計している。これはビジネスにおける顧客体験設計と同じ論理だ。機能は模倣できるが、感情的な結びつきは模倣しづらい。

## ビジネスパーソンが今すぐ学ぶべき「ゼロトラスト」の精神

この事件の裏で、多くのメディアがあまり報じなかった深刻な話がある。

Claude Codeのソースコード流出とほぼ同時刻、JavaScriptエコシステムで週間ダウンロード数約1億回を誇るaxiosライブラリの主要メンテナーのnpmアカウントが乗っ取られ、悪意あるバージョンが公開された。`npm install`を実行するだけで、クロスプラットフォーム対応のリモートアクセス型トロイの木馬が自動的にインストールされる仕組みだった。

特に3月31日に`npm update`を実行した開発者は、このaxios問題との複合被害を確認すべき状況となった。

### 流出コードから発見された「17の攻撃ポイント」

コード解析により、Claude Code自体にもCritical級を含む17もの攻撃ポイントが特定された。特に注意すべきは以下だ：

- **`claude -p`（非対話モード）**: 通常の確認フローをスキップするため、悪意あるリポジトリで実行すると危険
- **`.mcp.json`**: 設定ファイルに見えるが、実際にはコマンド実行面として機能する可能性がある
- **`claude doctor`**: 診断コマンドのつもりで実行しても、内部でプロセスが起動する場合がある
- **`helpers`経由の`shell: true`実行**: コマンドインジェクションのリスク

### 今すぐ取るべきアクション：

1. 知らないリポジトリで`claude -p`を使わない
2. `.mcp.json`、`hooks`、`helper`はレビューなしで実行させない
3. プロジェクト設定ファイルを「実行可能コンテンツ」として扱う
4. 3月31日前後に`npm update`した人はaxios侵害との複合被害を確認する

Anthropicはこの問題に対し、中国のAIラボによる模倣攻撃への対抗策として「Anti-Distillation（偽のツール定義注入）」や「Client Attestation（非正規クライアント検出）」も実装していた。IP保護への強い意識が伺える。

この時代に求められるのは、**「ゼロトラスト」の精神**――あらゆるアクセスをデフォルトで疑い、都度検証するセキュリティ哲学だ。AIツールが業務の中枢に入り込む今、この考え方はエンジニアだけでなく、すべてのビジネスパーソンに必須の教養となっている。

### AIエージェント時代のセキュリティ3原則

AIエージェントが自律的にコードを書き、ファイルを操作し、外部サービスに接続する時代になった今、従来の「入口を守れば安全」という境界型セキュリティは機能しない。私が5,000人以上のAIコミュニティで繰り返し伝えているのが、以下の3原則だ。

**原則1：最小権限の徹底**――AIエージェントに与える権限は、そのタスクに必要な最小限に絞る。「全部任せ」の便利さと「万が一の被害範囲」はトレードオフだ。

**原則2：設定ファイルを「コード」として扱う**――`.mcp.json`、`CLAUDE.md`、`hooks`設定は単なるテキストファイルではなく、実行される可能性のある「コード」だ。レビューなしに適用しない。

**原則3：サプライチェーンを常に疑う**――今回のaxios侵害が示したように、あなた自身が書いたコードよりも、使っているライブラリの方が攻撃に脆弱なことがある。依存関係の更新は慎重に行う。

## 「AIが書いたコード」の生々しい現実――盲信するな、活用せよ

この事件が暴いたもう一つの真実がある。Anthropicが「世界最強のコーディングAI」を目指していることは疑いようがないが、AIが書いたコードの品質については冷静に見る必要がある。

流出した512,000行の中で発見された指標：

- **`as unknown as`（型安全の放棄）**: 1,884件
- **`eslint-disable`（リンタールール無視）**: 460件
- **`TODO`（放置された課題）**: 138件
- **`HACK`（認識されたハック）**: 23件
- **`FIXME`（壊れてると認識）**: 6件

これはAnthropicへの批判ではない。むしろ逆だ。

**問題を認識し、フラグとして記録し、段階的に改善していく――これが「生きているコードベース」の証だ。**

Amazonでのオペレーション管理経験から言えば、完璧な初期状態より「問題が可視化されている状態」の方が組織として健全だ。問題が見えないシステムは、問題がないのではなく問題が隠れている。

ただし、これはビジネス側にも重要な示唆を与える。AIが生成するアウトプット――コードであれ、文書であれ、分析であれ――は「完成品」ではなく「叩き台」として扱うべきだということだ。最終的な品質保証には、人間の目と判断が不可欠だ。

### なぜ「AIが書いたコード」にも型安全の放棄が生まれるのか

TypeScriptの`as unknown as`は、型チェックを強制的にバイパスする「型の嘘つき宣言」だ。1,884件というのは、相当な数の「型の嘘」が積み重なっていることを意味する。

これはAnthropicのエンジニアが怠慢だったからではない。高速に進化するコードベースでは、型システムが「現実の複雑さ」に追いつかないことがある。その場合、完璧さより前進を選ぶという実用主義的な判断が働く。

AIツールを使って成果物を作る私たちも、同じジレンマを抱えている。完璧を追求して立ち止まるより、「動く叩き台」を出して改善していく方が、多くの場合で優れた結果を生む。重要なのは、「これは叩き台だ」という認識を持ち続けることだ。

## 実践者が見る「KAIROSの本当の意味」――すでに動かしている人たちへ

この流出から私が引き出す最も重要な洞察は、技術的な詳細ではない。

**「Anthropicがfeature flagで『これから作る』と言っているものを、実践者たちはすでに動かしている」**という事実だ。

常駐型AIアシスタント、セッション間記憶の保持、プロアクティブなタスク実行、GitHub連携による自律レビュー――これらはKAIROSが「将来実装する」と宣言しているものだが、マルチエージェントの本格運用を行っている開発者コミュニティでは、すでに類似の仕組みが動いている。

私が開発しているMiyabiフレームワーク（GitHub: ShunsukeHayashi/Miyabi）も、まさにこの「常駐・自律・記憶保持」を実現するための階層型AIエージェント設計だ。Anthropicが「これから向かう」方向と、実践者たちが「すでに作り始めている」方向が一致している――これは偶然ではなく、AIエージェント化という不可逆の潮流の証拠だ。

## ゴールシークプロンプトとKAIROSの共鳴

ゴールシークプロンプトの核心は「How（どうやって）より先にWhat/Why（何を/なぜ）を定義する」ことだ。

KAIROSプロジェクトのフラグ設計もまったく同じ構造を持っている。Anthropicはまず「AIはどんな問題を解くべきか（What/Why）」を107個の課題として定義し、それぞれにfeature flagというHowの実装を割り当てている。

**問題の解像度が高い組織だけが、正しいソリューションを作れる。**

この事件は、「最先端のAI企業がいかに問題を見る目を持っているか」を、意図せずして世界に公開したのだ。

## まとめ：「未来からの設計図」を読み取った後、あなたが取るべきアクション

Claude Codeのソースコード流出は、3つの重要な事実を教えてくれた。

1. **人間的ミスはシステムが変わらない限り繰り返す**――AIツールを使う側も、ビルドパイプラインや権限設計など「構造的な穴」を定期的に見直す必要がある。

2. **次世代AIは「常駐・自律・記憶保持」へ向かう**――KAIROSが示す方向性は、AIとの関係を「道具を使う」から「共に働く」へと変える。この変化に備えることが、今後3年間の最大の競争優位になる。

3. **AIの出力を盲信せず、セキュリティとガバナンスを先に整える**――便利さの裏にあるリスクを理解した上で活用する組織と、そうでない組織の差は、AIの普及とともに拡大し続ける。

Anthropicが「これから作る」と宣言したものは、実践者たちが「すでに作り始めている」。この潮流に乗るために必要なのは、最新技術の追いかけではなく、**AIをどう使いたいかという明確な目的（What/Why）の設計だ。**

それこそが、ゴールシークプロンプトが一貫して提唱してきた考え方であり、AI時代を生き抜く最強の武器だ。
