# 本質ドキュメントを構築するための各ステップ

## step1. AIエージェント・ハーネスエンジニアリングとは

- まず「自分のAIエージェントに何を評価させたいか」を1つ書き出してみよう。それが本質ドキュメントの出発点になる。

## step2. 本質ドキュメントの設計思想

- 自分の領域（スキル設計、UI、コンテンツ等）で「誰が見ても正しいと言える原則」を3つ書き出してみよう。それが本質ドキュメントの種になる。

## step3. レビューアパターンとフィードバックループ

- 既存のスキルやワークフローに「レビューア呼び出し→フィードバック→再実行」の3ステップを1つ追加してみよう。

## step4. コンテキストフォークとサブエージェント設計

- メインエージェントからコンテキストフォークで呼び出されるサブエージェントには、以下の役割がある：
  - 役割：探索者（Explorer）｜何をするか？：情報収集・コードサーチ｜なぜ分離するのか？：不要な情報もコンテキストに入るため｜Claude Codeの対応機能：Explore agent
  - 役割：計画者（Planner）｜何をするか？：構成・オーケストレーション｜なぜ分離するのか？：計画特有のプロンプトが注入されるため｜Claude Codeの対応機能：Plan mode
  - 役割：実装者（Executor）｜何をするか？：コード生成・成果物作成｜なぜ分離するのか？：実装に集中させるため｜Claude Codeの対応機能：general-purpose agent
  - 役割：レビューア（Reviewer）｜何をするか？：品質評価・フィードバック｜なぜ分離するのか？：本質ドキュメントに基づく評価のため｜Claude Codeの対応機能：カスタムスキル

- 実装者サブエージェントが困った時に自律的にレビュースキルを呼び出してフィードバックを受けるという構造が実現
```
メインエージェント
  └─ スキル経由 → サブエージェント（実装者）
       └─ スキル経由 → マゴエージェント（レビューア）
            └─ 本質ドキュメント参照
```

- 現在1つのエージェントに任せているタスクを「探索→計画→実装→レビュー」に分解し、最も効果の高い1つをサブエージェント化してみよう。

## step5. 決定論的制御と確率的制御のバランス

- 自分のワークフローで「毎回同じ結果が必要な処理」を1つ見つけ、スクリプトまたはHookに置き換えてみよう。

## step6. 実践パターン ― 今日から使えるテクニック

- 最もよく使うスキルのMarkdown末尾に `## Gotcha` セクションを追加し、過去に遭遇した失敗パターンを3つ書き出してみよう。
