動いていた半分と、
動かせない半分

「PR駆動の本質ドキュメント自動提案フロー」を、記憶や設計メモではなく 稼働中の実体で確認した。提案を変更依頼の形にまとめる構想のうち、 最初の一段(Phase A)はすでに実装済み。 だがグローバル設定が git 管理の外にあるため、 肝心の対象を変更依頼にできない —— 進めない本当の理由はそこにあった。

1Implemented · 実装済Phase A (PR を自動で作る機能) は稼働可能
2Unbuilt · 未実装Phase B (通知) と D (統合 + 手順書)
1Dropped · 廃止Phase C (AI 自動レビュー)
0Startable Now · 即着手可構造的な前提が未充足で保留

このフローは、技術記事「継続的改善 #8」で提案された仕組みで、 skill や agent の改善を 変更依頼(PR: Pull Request、コードへの変更提案) の形にまとめ、「①収集 → ②提案 → ③通知 → ④人間レビュー → ⑤取り込み」の 5 段階で半自動的に回すものだ。

土台になるのは既存の accumulating-reviewer-feedback (改善点を 1 件ずつ人間が承認して反映していく skill)。 このフローは、その承認の単位を「ファイル 1 つ」から「変更依頼 1 つ」へ 一段引き上げる —— つまり人間が介在して承認する仕組み(HITL: Human-in-the-loop)を、より大きな単位へ昇格させる構造になっている。

// STATUS · Phase 別の現在地

4 つの段階、それぞれの実体

A
機能 · 実装済

--pr フラグ — 変更依頼を自動で作る

accumulating-reviewer-feedback を --pr 付きで起動すると、 変更を専用ブランチ(枝分かれした作業場)に切り分けて、変更依頼を下書きで作成する。 ブランチの命名規約や、人間が判断を挟む 2 つのポイントまで、 skill 本体(SKILL.md)にすでに書き込まれていて、そのまま動かせる状態だった。

実体: SKILL.md に実装済み 稼働可能
B
機能 · 未実装

Discord 通知 — 変更依頼ができたら知らせる

GitHub の自動実行係(GitHub Actions)が、変更依頼の作成を見張って、 Discord の投函口(webhook)へ「できたよ」と通知を飛ばす仕組み。 その指示書ファイル(notify-skill-update.yml)がまだ存在しないので未実装。 外部のインフラ(GitHub Actions + Discord)が前提になる。

設計の抜け · 3 つ (実装着手時に要対応)

① 投函口 URL が未登録のときに止める安全チェックが無い ／ ② 通知に載せたい情報は 5 つあるのに、設計例には 1 つ(変更依頼の URL)しか書かれていない ／ ③ その情報のうち 3 つ(指摘件数・集計・対象 skill 名)を「どこから取ってくるか」が未設計

前提: GitHub Actions + Discord webhook 未実装
C
機能 · 廃止

AI 自動レビュー — 廃止が確定済み

AI が変更依頼を自動でレビューする案。2026-05-17 に廃止が確定している。 理由は 2 つ —— (1) Claude API は使った分だけ課金される従量制(月 $5-20 想定)で、 「定額契約(subscription)の範囲内で使う」というハーネスの大原則に反する ／ (2) 元の技術記事が掲げる「人間レビューを強める」精神に反する (AI の事前審査を挟むと、人間の判断が薄まる方向に働く)。

代替: 人間 100% レビュー 廃止確定
D
機能 + doc · 未実装

統合と運用マニュアル

統合 = 機能

Phase A(変更依頼の作成)と B(通知)を繋いで、一連の流れとして動かす仕上げ。 B が無いと繋ぐ相手がいないので、B 待ちで止まっている。

runbook = doc

運用手順書(runbook)を .docs/runbooks/ に書く。 これは かもね(人間)が読む運用マニュアルで、 AI が読む実行手順は skill 本体に既にある。

前提: Phase B の完成 未実装
先送り(defer)の本当の壁は「時間」ではなかった。
グローバル設定が git で管理されない限り、
変更依頼にする対象そのものが、生まれない。 — 根本制約 · アーキテクチャの問題

ここが最大の山場だ。このフローは「skill や本質ドキュメントの改善を変更依頼にする」もの。 だが、その skill や本質ドキュメントの本体は ~/.claude/(グローバル設定)に置かれていて、 そこは意図的に git 管理されていない。 変更依頼(PR)は git で管理された場所のファイルにしか作れないから、 変更依頼にする対象が、現在の構成では存在しない。

だから「本番運用に昇格したら着手」という先送りの読み方は、半分しか正しくない。 本当の前提は「時間が経てば」ではなく、 「グローバル設定を git 管理するという構成変更をするかどうか」。 待っていても、対象は生まれてこない。

// VERDICT · 処遇の精密化

「全体先送り」を、実体に合わせて分解する

引き継ぎメモ(handoff)は、このフローを「全体を先送り(defer)」と一括で扱っていた。 方向は正しいが粗く、Phase A がすでに完成していることを見落としていた。 実体に即して分解すると、こうなる。
