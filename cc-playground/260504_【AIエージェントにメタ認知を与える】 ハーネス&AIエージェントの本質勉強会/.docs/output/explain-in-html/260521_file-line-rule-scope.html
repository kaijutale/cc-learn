CLAUDE.md · 500行ルールの設計

500行ルールの
「対象」とは何か [ A or B ? ]

「ファイルは 500 行以内」という決まりを、命令(上限)から合図に直すのがゴール。
でも「合図を出す対象」を決めるには、どのファイルに効かせて、どれを外すかを言い切る必要がある。
その境界線と、まだ決まっていない A / B の分かれ道を整理する。

500 行は「ここで分割しろ」という上限ではなく、
「分割を検討してみては?」という合図だ。 — 出発点となる考え方
// PART · 01 — 境界を分ける一本の基準

合図を「効かせる側」と「外す側」

対象かどうかを分ける基準は、たった一つ。「分割するか否かを人が判断できて、その分割に意味があるか」。この一本でスパッと割れる。

対象 (合図を効かせる)

人が書いて保守するソースコード。
.ts .tsx .js .py .go .rs など。

人が構造を決め、読んで、差分(diff)を取り、直し続ける。だから「責務ごとに分ける」判断に意味がある。500 行の合図が活きるのはここだけ。

対象外 (合図を出さない)

行数を人が制御しない、または分割しても無意味なファイル。

機械が吐き出すもの・データ・ロックファイルなど。ここに合図を出しても「分割しろ」が成り立たない(分けようがない / 分ける意味がない)ので、ノイズにしかならない。

// PART · 02 — 対象外の中身

「外す側」は 4 種類に分かれる

分類なぜ外すか具体例
① 機械生成物行数を人が制御しない生成HTML(.docs/output/**)、ビルド出力、コード自動生成
② データ分割が無意味.json / .csv / .tsv
③ ロックファイルツールが管理、人は触らない*.lock / *-lock.json
④ ベクタ・圧縮・記録系機械フォーマットで分割不可.svg / .map / .min.* / .snap

②③④ は今のフックが既に外している。さっき HTML が警告に引っかかったのは、① の生成HTMLが漏れていたから。.docs/output/** を足せば、フックが元々意図していた「生成物は外す」が完成する。

// PART · 03 — まだ決まっていない分かれ道

Markdown や設定ファイルは、どっち?

ここが宙ぶらりん。.md(解説ドキュメント)や .yaml(設定)は「コード」ではないけれど、生成物でもない。対象に入れるか外すかで、2 つの設計に分かれる。

案 A — コード限定 (推奨)

コードの拡張子だけを対象にする。
.md や .yaml は対象外。

長所: 元の「コードファイル」という言葉に忠実で、境界が明確。長文の解説ドキュメントに警告が出ない。
短所: 対象とする拡張子の一覧を持つ必要がある。

案 B — 生成物以外は全部

外す 4 種以外は、すべて対象。
.md も .yaml も 500 行超なら警告。

長所: 仕組みが単純(今のフックのまま)。
短所: 600 行の解説ドキュメントにも警告が出る。長文が妥当な場面でもノイズになる。

わたしの推奨は案 A(コード限定)。理由は 2 つ。元のルールが「コードファイル」と限定していた意図に忠実であること。そして、解説 Markdown や設定 YAML は「責務ごとに分割する」対象ではなく、長文が自然な場面が多いから。500 行の合図は、人が凝集度(=一つの責務にまとまっているか)で割るべきコードにこそ効く。

行数は「上限」ではなく「合図」。
そして合図を鳴らす相手は、
人が凝集度で割れるコードだけ。
