ワークフロー全体像と AI 自動レビューの位置

essence-review finding から merge までの 7 段

Phase A → B → C → D の自動化レイヤー

既存 · 既に動いてる

essence-review finding 永続化

レビュー結果 (finding) を accumulating-reviewer-feedback skill で集約、Apply 候補を確定する段階。これは既に Session N+4/N+5 で実機実行済み。

既存 · 既に動いてる

skill ファイル改修 + commit

finding を Apply して skill ファイルを書き換え、committer 経由でローカル commit する段階。これも既に動いている。

Phase A · 新規 · 最小実装

--pr フラグで PR 自動作成

accumulating-reviewer-feedback に --pr フラグを足し、commit ではなく gh pr create --draft で pull request (差分レビュー単位) を作る段階。subscription Max Plan 内で完結。

Phase B · 新規 · 通知

Discord webhook で通知

GitHub Actions が pull_request.opened イベントを受けて Discord の channel に webhook (受信フック) で通知する段階。設計の抜けが 3 つ残ったままのセクション。

Phase C · ★ AI 自動レビューはここ

AI が PR の差分を自動レビュー

PR 作成された後、camone が承認する手前に挟まる「機械的なフィルタ層」。GitHub Actions 上で Claude API を起動し、24 原則 (ハーネス 8 + skill 8 + UI 8) と照合、違反兆候を severity (Critical / High / Medium / Low) 付きで PR コメントに投稿する設計。月 $5-20 の API 課金が発生 = subscription Max Plan 違反のため削除候補。

Phase D · 統合

HITL (人間判断) → 承認 or 修正依頼

camone が PR の AI レビュー結果を見て、approve (承認) して merge するか、request changes (修正依頼) で再 commit を促すか判断する段階。HITL = Human-In-The-Loop = 人間が判断する点を残す仕組み。

最終 · 完了

merge 完了 → skill 反映

main ブランチに merge され、skill 改修が確定する段階。次の essence-review サイクルへ。

AI 自動レビューは「PR が作られた後、人間 (camone) が承認する手前」に挟まる、機械的フィルタ層である。 — Phase C の位置づけ
// GAPS · 02

Phase B (Discord 通知) の "抜け" 3 つ

01
Gap · Validate step

webhook URL 未登録時のチェックが design doc にない

現状の問題 (Silent Failure)

GitHub Secrets (秘匿変数の保管場所) に DISCORD_WEBHOOK_URL を登録し忘れたまま PR を作ると、curl コマンド (HTTP リクエストを送るツール) が空 URL で実行され、ワークフローが赤エラーで停止する。しかしエラーメッセージは curl: (3) URL using bad/illegal format だけで、「Secret 未登録だよ」という親切なメッセージは出ない。原因を camone が能動的に Actions タブで調べる必要がある。

Before (現状の design doc)

YAML 実装例には curl -X POST "$DISCORD_WEBHOOK_URL" ... しか書かれていない。URL が空でもそのまま実行される。

After (改善案)

curl の手前に Validate step を 1 つ足す。if [ -z "$DISCORD_WEBHOOK_URL" ]; then echo "::error::..." ; exit 1; fi で fail-fast (失敗を即座に伝える)。

改善コスト: YAML に 7 行追加 Low · 局所修正
02
Gap · Payload 乖離

「通知する情報」の要件と実装例が一致してない

現状の問題 (Design Inconsistency)

design doc §3-2 の本文では「通知に 5 情報を載せる」と書かれているのに、同じ §3-2 内の YAML 実装例には PR URL の 1 情報しか載っていない。要件 5 情報のうち 4 情報 (PR title / 対象 skill / finding 件数 / accept-defer-dismiss 集計) が実装例から抜けている。設計と実装が同じファイル内で食い違っている状態。

要件 (§3-2 本文)

5 情報を通知: PR title / 対象 skill / finding 件数 / accept・defer・dismiss 集計 / PR URL

実装例 (§3-2 YAML)

1 情報のみ:
"📝 skill 改修 PR 作成: <URL>"

改善方向: 要件を縮小 or 実装を embed 化で拡張 Medium · 方針判断
03
Gap · データ取得

「finding 件数」「集計」「対象 skill」の抽出ロジック未記載

現状の問題 (Data Source Undefined)

要件 5 情報のうち、PR title と PR URL は GitHub Actions の github.event.pull_request.* 変数から即時取得できる。しかし残り 3 情報 (finding 件数 / accept・defer・dismiss 集計 / 対象 skill) の取得方法が design doc に一切書かれてない。実装着手時にゼロから設計する必要がある。

取得済み (即時可能)

pull_request.title = PR title
pull_request.html_url = PR URL

未設計 (要追加実装)

finding 件数 = PR description を正規表現で grep 必要
集計 = 同上、書式依存で脆弱
対象 skill = changed files から path 抽出 + unique 化必要

改善コスト: 抽出パーサー (解析プログラム) を別途設計 Medium · 設計追加

camone が決めること: 2 つの質問

Q1
Phase C 代替案

AI 自動レビューを subscription 内でどう実現するか

案仕組み自動化度注意
A. self-hosted runner 自分のマシンに GitHub Actions runner を立てて claude CLI を起動 高 常時起動マシン要
B. local 手動 claude 起動 Discord 通知を見て camone が手元で claude を起動 中 手間 +、価値 3/3 保持
C. pre-PR レビュー PR 作る前の段階でレビュー完了させる 中 PR 段階の独立性消失
D. CodeRabbit 外部 SaaS で AI レビュー 高 別課金、subscription 外
E. AI レビュー廃止 機能諦め、camone 手動レビューに戻す 低 機能放棄、シンプル
わたしのおすすめ: B または E High · 機能選択

Phase B 抜けの扱い

抜け 3 つを design doc 上でどう処理するか

案処理内容コスト

B. メモだけ追加 design doc に「未解決事項」セクションを足し、3 抜けを箇条書きで残す。本体は触らない 小
C. 今すぐ直す Phase C 削除と同時に design doc 本体を改修 (Validate step / payload / 抽出ロジック) 中
わたしのおすすめ: B (メモだけ追加) Medium · 運用判断
