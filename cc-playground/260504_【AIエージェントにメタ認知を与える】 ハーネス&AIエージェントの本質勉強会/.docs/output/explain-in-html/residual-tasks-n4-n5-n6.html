2 つの構造的制約 — 触れてはならない線

「部分実装は欠陥ではなく、
次サイクルへの手紙である」
— そして同じ手紙が、改修してはならない場所も指し示す。 — Session N+3 完了時の Lead 独自観察より

Session N+4 以降でも、絶対に触ってはいけない skill / agent / ドキュメントがある。 これらは「協調ハーネス系の既存資産」で、改修すると影響が芋づる式に広がるためマルチエージェント協調が崩壊する。 改修禁止リストは handoff-state.md の ### 改修禁止 セクションと完全同期しているよ。

カテゴリ パス 役割

協調 skill ~/.claude/skills/orchestrating-team-development/ L6 指揮者ワークフロー (オーケストレーション中枢)
協調 skill ~/.claude/skills/enforcing-strict-tdd-cycle/ 厳密な TDD サイクル (Red-Green-Refactor) 強制

評価基準 ~/.claude/.docs/essence/{harness,skill,ui}-essentials.md 3 領域 × 8 原則 = 24 原則の評価基準 (変えると全 finding が無効化)

Why · 改修禁止の理由

memory feedback_no-existing-harness-modification.md より — 「マルチエージェント協調ハーネス系の既存 skill / subagent は絶対改修禁止。 新基準には新規ハーネス作成。改修=負の連鎖」。 評価基準を変えると、過去の self-eval ログ (10 件以上) との比較ができなくなり、 品質階段現象の観測精度が崩れるよ。
