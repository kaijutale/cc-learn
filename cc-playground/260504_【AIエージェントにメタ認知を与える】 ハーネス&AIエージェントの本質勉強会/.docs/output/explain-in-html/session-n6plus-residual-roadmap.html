// PART · 02

なぜ #8 が必要か

accumulating-reviewer-feedback skill は Session N+4 と N+5 で 3 回起動済で、HITL ループの実機実証は完了している。しかし、現状の skill には 3 つの構造的限界がある。

  • PR 単位の差分レビューができない — skill ファイル単位の反映 (Apply) は完了するが、複数 skill 横断改修の整合性チェック (= 変更が他の skill に矛盾を生んでいないかの確認) は未対応。
  • チーム共有ができない — commit (変更記録) はローカルに留まり、Discord や Slack などの協業ツール連携がない。
  • 自動収集ができない — Lead (主導役の Claude) が手動で skill を呼ぶ必要があり、CI (= 継続的インテグレーション、変更が自動でテストされる仕組み) 連携がない。

#8 はこれら 3 つを「pull request + webhook + 人間レビュー」で構造的に解消する設計。

skill ファイル単位の HITL から、pull request 単位の HITL へ。これは 1 段階上の構造への昇格である。 — 設計 doc 2026-05-16 より
