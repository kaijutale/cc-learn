# C-1〜C-7 の「C」は Constraint(制約)の頭文字

> 記事 Part I「LLMの構造的制約」の C-1〜C-7 ラベルの「C」が何かという質問。C = Constraint(制約)で、LLMが構造的に抱える7つの弱点に振った通し番号。

## 概要

参照記事「【なぜか微妙は偶然ではない】Claude Codeで学ぶ 7つの構造的制約とハーネス設計、AIエージェントの本質論」(note / masa_wunder) の **Part I. LLMの構造的制約 — 変えられない前提を知る** セクションに登場する `C-1`〜`C-7` というラベルの「C」が何の略かを問われた。

記事は2部構成:
- Part I = 「なぜそうなるのか」を **7つの構造的制約 (C-1〜C-7)** として整理
- Part II = 「どう対処するか」を具体的な設計パターンとして提示

## 内容

### 質問

- 「Part I のLLMの7つの構造的制約の C-1〜7 について。C ってなに?」

### 回答

- **C = Constraint(制約)** の頭文字。
- 記事独自のラベリングで、「LLMの7つの構造的制約 (structural Constraint)」のそれぞれに `C-1`〜`C-7` と通し番号を付けている。「C-1」=「制約その1」の意。
- これらは「プロンプトの工夫だけでは回避できない、LLMが構造的に抱えている弱点」であり、記事のいう「変えられない前提」。だから Part II では環境(ハーネス)側で対処する流れになる。

### 7つの構造的制約 (Part I 冒頭の全体図より)

| ラベル | 制約名 | 要点 |
|---|---|---|
| C-1 | コンテキスト帯域は有限(ゼロサム) | 情報を入れるほど集中が分散 |
| C-2 | 入力バイアスの増幅 | 最初の情報に引きずられる |
| C-3 | 迎合性 | ユーザーの期待に合わせてしまう |
| C-4 | 自己申告は証拠にならない | 「できました」を鵜呑みにできない |
| C-5 | 報酬ハッキング | 評価基準の抜け穴を突く |
| C-6 | 訓練データの偏り | 学習データの偏りが出力に出る |
| C-7 | 校正盲 | 知らないことを知らない |
