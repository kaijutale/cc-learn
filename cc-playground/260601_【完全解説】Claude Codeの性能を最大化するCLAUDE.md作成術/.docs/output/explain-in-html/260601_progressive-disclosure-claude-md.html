// PART · 01

なぜ詰め込むと AI が鈍るのか

大規模言語モデル (LLM) にはステートレス (過去の対話を一切覚えておらず、毎回まっさらな状態から始まる性質) という特性があります。だから Claude Code はセッション開始時に CLAUDE.md を読み込み、それを「プロジェクトの取扱説明書」として頼ります。

ところが、AI が一度に確実に従える指示の数には限界があります。研究では最先端モデルでもおよそ 150〜200 個が上限とされ、しかも Claude Code は内部のシステムプロンプトですでに約 50 個を使っています。残りの「指示のスロット」は最初から限られているのです。

ここに無関係な情報や細かすぎるルールを大量に書き込むと、AI の注意力が散漫になり、本当に重要な指示まで無視されかねません。コンテキストウィンドウ (AI が一度に読み込める情報の作業領域) を、関係ないノイズで汚してしまうわけです。

// PART · 02

段階的開示とは何か

CLAUDE.md は短く保ちたい。でもプロジェクトには詳細なドキュメントが必要だ ── このジレンマを解くのが段階的開示 (Progressive Disclosure) です。考え方は一文に集約できます。

全ての情報を一度に見せるのではなく、必要な時に必要な情報へアクセスする「方法」を教える。 — note「CLAUDE.md 作成術」より

つまり CLAUDE.md には詳細そのものを書かず、「どこに」「何の情報があるか」という道しるべ (ファイルパスと短い説明) だけを置きます。AI は普段はその道しるべだけを読み、必要になったときに自分で詳細ドキュメントを開きに行く。こうして CLAUDE.md は常にクリーンなまま、AI はタスクに集中し続けられます。

// PART · 03

3 ステップで実装する

01
置き場を作る

docs/ ディレクトリを用意する

プロジェクト内に docs/ のようなドキュメント専用ディレクトリを作ります。詳細情報の「逃がし先」をまず確保するステップです。
02
詳細を分割する

タスク固有の情報を別ファイルへ

API 仕様書・コーディング規約・データベース設計など、タスク固有の詳細を、それぞれ独立した Markdown ファイルとして docs/ の中に保存します。1 ファイル = 1 トピックが目安です。
03
道しるべを書く

CLAUDE.md にはポインタだけ

CLAUDE.md には詳細を直接書き込まず、ファイルパスと一行の説明だけを並べます。これが「必要な時に開きに行く方法」を AI に教える道しるべ (ポインタ) になります。
// PART · 04

ディレクトリ構成例

3 ステップを実際のファイル構成に落とすと、次のようになります。CLAUDE.md は入口に置き、詳細は docs/ の下に役割ごとに分けます。

project root — directory layout 
my-project/
├── CLAUDE.md                  # 道しるべのみ。300 行未満に保つ
├── docs/                      # 詳細ドキュメントの逃がし先
│   ├── api-spec.md           # API 仕様書
│   ├── coding-standards.md   # コーディング規約
│   ├── database-design.md    # データベース設計
│   └── deployment.md         # デプロイ手順
├── src/
│   └── ...                   # アプリ本体
└── package.json

ポイントは CLAUDE.md と docs/ の役割分担です。CLAUDE.md は「地図の目次」、docs/ 配下の各ファイルは「個別の詳細地図」。AI は目次を読んで、必要な詳細地図だけを取りに行きます。

// PART · 05

CLAUDE.md の書き方 — Before / After

Before — 全部詰め込む

API の全エンドポイント、規約の全項目を CLAUDE.md に直書き。

  • ファイルが数百行に膨張する
  • 指示スロットを食い潰す
  • 重要ルールが埋もれて無視される
  • コンテキストがノイズで汚れる

After — 道しるべだけ

詳細は docs/ へ逃がし、CLAUDE.md はパスと一行説明のみ。

  • 300 行未満を保てる
  • 指示スロットを節約できる
  • 重要ルールが際立つ
  • AI が必要時に自分で参照する

道しるべ側 (After) の CLAUDE.md は、たとえば次のように書きます。詳細は一切書かず、どこに何があるかだけを示すのがコツです。

CLAUDE.md — pointer style (After) 
# CLAUDE.md

## WHAT — 技術構成
- Next.js (App Router) + TypeScript + Tailwind CSS

## 詳細ドキュメント (必要に応じて参照)
- API 仕様:        docs/api-spec.md
- コーディング規約: docs/coding-standards.md
- DB 設計:         docs/database-design.md
- デプロイ手順:     docs/deployment.md

あわせて記事は、コードスタイル (インデントや引用符の種類) のルールは CLAUDE.md に書かず、ESLint や Prettier などのリンター (コードを自動で整える専門ツール) に任せるべきだと説きます。機械的な整形はツールへ、設計やロジックといった創造的な仕事は AI へ ── この役割分担もまた、貴重な指示スロットを守る段階的開示の延長線上にあります。

観点直接書き込み段階的開示

指示スロット細部で食い潰す重要ルールに温存
