01

情報密度 — 1 ファイルに表・SVG・JS が同居

Markdown は 線形テキスト。空間的関係 (依存グラフ・フロー・複数案の並列) を表現するには 読み手の脳内で組み立て直す 必要がある。HTML は CSS Grid + SVG + inline JS を 1 ファイルに同居させ、空間情報をそのまま視覚化できる。

Thariq の表現: "Diffs and call-graphs are spatial information; markdown flattens them."

02

視覚的明快さ — 色分けで重大度を即把握

PR レビュー HTML を例に挙げる。Blocking / Nit / Nice-to-have を 色分け severity badge で示すと、読まずに重大度を理解できる。

HIGH MED LOW INFO SUCCESS

このサンプルでは Thariq 流の warm tinted badges (clay #D97757 / oat #E3DACC / olive #8FA672 系) を dark 背景用に rgba 透明度で調整。box-shadow を使わず、 1.5px の #3D3D3A border で hierarchy を作る。

03

共有のしやすさ — URL 1 本でブラウザ即動作

self-contained HTML は 外部依存ゼロ。CDN 切断、Gmail 添付、Slack 共有、すべて「リンク 1 本で開いたら動く」。Markdown は GitHub / Confluence の renderer 経由必須。

Thariq 観測: PDF メール送信より 開封率が約 2 倍。本サンプルも CSS は inline、フォントは system-ui 系で 外部 CDN 依存ゼロ (camone v4 は Google Fonts CDN 依存あり、ここが差別ポイント)。

04

双方向性 — slider / toggle / copy で値を返す

HTML の真骨頂は 「触れる」。slider でパラメータを動かす、toggle で詳細を展開する、copy button で「LLM に渡す prompt」を生成する。

本サンプルは 静的 だが、実用版では <input type="range"> + 微量 vanilla JS で十分。Thariq 公式の Consistent Hashing 解説 (15-research-concept-explainer.html) では range slider と SVG ring が連動している。

05

context 幅拡大 — MCP + git + web を 1 面に

HTML は 複数情報源を 1 つの「レビュー面」に圧縮 できる。MCP の最新データ、git の diff、web スクレイピング結果を inline SVG + 表 + コードブロックで混在配置可能。

Thariq の MCP ダッシュボード例: "開くたびに Stripe や Git から最新データを取って表示する HTML 軽量エディタ"。静的レポートではなく「ライブダッシュボード」。

06

住み分け判定 — 4 段階の意思決定ルール

Thariq は「HTML 万能論」ではなく 住み分け を結論にしている。

  1. 短文か? → Markdown (HTML は過剰)
  2. git で追跡するか? → Markdown (HTML diff はノイジー)
  3. 100 行超か? → HTML を検討 (Markdown は scroll 専用に堕ちる)
  4. 他者に送るか? → HTML 強く推奨 (決定的要因)
Trade-off Severity Mitigation
生成時間が Markdown の 2–4 倍 MED Opus 1M context で許容、長文・配布前提のみ HTML 化
HTML diff が git で読みにくい HIGH ソースは Markdown / JSON、HTML はオンデマンド生成、gitignore
20 行以下では HTML は過剰 LOW 短文判定ルールで Markdown 自動分岐
