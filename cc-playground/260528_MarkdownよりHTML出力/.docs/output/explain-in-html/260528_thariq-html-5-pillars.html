// PILLAR · 01

情報密度 — 1 ファイルに表 + SVG + JS が同居

Markdown は 線形テキスト。空間的関係 (依存グラフ、フロー、複数案の並列) を表現すると 読み手の脳内で組み立て直す 必要がある。HTML は CSS Grid + SVG + inline JS を 1 ファイルに同居させ、空間情報をそのまま視覚化できる。

Thariq の元主張: 「diff も call-graph も spatial information で、markdown はそれを潰してしまう」。本ページの .stat-bar や .cards のような 2 次元配置は、Markdown では再現不能。

// PILLAR · 02

視覚的明快さ — 色分けで重大度を可視化

Thariq の代表例は PR レビュー HTML。Blocking (赤) / Nit (黄) / Nice-to-have (緑) を 色分け severity badge で示すと、長文を読まずに重大度が即座に分かる。

本 skill ではこれを .severity-badge の 7 段階 (crit / high / med / low / info / pass / harmless) として実装。terracotta 系で「赤」、teal 系で「無害・成功」の二極を作り、Bootstrap red との同質化を避ける。

severity の見本: CRIT HIGH MED LOW INFO PASS HARMLESS

// PILLAR · 03

共有のしやすさ — URL 1 本でブラウザ即動作

self-contained HTML は 外部依存ゼロ。CDN 切断、Gmail 添付、Slack 共有、すべてで「リンク 1 本 → ブラウザで開いたら即動作」。Markdown は GitHub / Confluence 等の renderer を経由しないと表組すら表示されない。

Thariq 観測: PDF メール送信より 開封率が約 2 倍。本 skill では Google Fonts のみが外部依存だが、fallback chain (Hiragino → Yu Gothic UI → Meiryo) を強化して CDN 切断時も Warm Paper の「丸み」が残るよう設計。

// PILLAR · 04

双方向性 — slider/toggle/copy で値を返す

HTML の真骨頂は 「触れる」。slider でパラメータを動かして結果を確認、toggle で詳細を選択的に展開、copy button で「この内容を LLM に渡す prompt」を生成。

本 skill v4 で実装: .copy-btn (clipboard) / .copy-prompt-bar (ページ全体を prompt 化) / .collapsible (details 折り畳み) / .slider-control (whitelist 式評価) / .filter-table (検索 + ソート)。inline vanilla JS で実装、合計 5KB 以内。

collapsible の実例 (クリックで展開) ▼

これは <details class="collapsible"> の動作例。JS 無効でも native の <details> として動作し、JS 有効時は height transition が滑らかに適用される (progressive enhancement)。

補足: 結論や主要情報を畳むのは anti-pattern。「初見では飛ばしてよい補足・引用元・raw data」だけを collapsible にする。

// PILLAR · 05

context 幅拡大 — MCP + git + web を横断

HTML は 複数情報源を 1 つの「レビュー面」に圧縮 できる。MCP で取った最新データ、git の diff、web スクレイピング結果、すべてを inline SVG + 表 + コードブロックとして混在配置可能。

Thariq の MCP ダッシュボード例: 「開くたびに Stripe や Git から最新データを取って表示する HTML 軽量エディタ」。静的レポートではなく「ライブダッシュボード」。本 skill では .filter-table + .svg-diagram + .collapsible 組み合わせで dashboard ユースケースをカバー。

// JUDGMENT · 06

住み分け判定 — 4 段階の意思決定ルール

Thariq は「HTML 万能論」ではなく 住み分けを結論にしている。4 段階で判定:

  1. 短文か? → Markdown (HTML は過剰)
  2. git で追跡するか? → Markdown (HTML の diff は属性ノイズで読みにくい)
  3. 100 行超か? → HTML を検討 (Markdown は scroll 専用に堕ちる)
  4. 他者に送るか? → HTML 強く推奨 (決定的要因)

本 skill は (3) (4) に該当するユースケース (audit / plan / research / dynamic / dashboard / explainer) を 6 種類のテンプレに整理。短文の概念解説では interaction layer を最小限 (TOC のみ条件付き)、長文 / 配布前提では完全装備で生成する。

機械が読むなら Markdown、人間が判断するなら HTML。
その境界を 100 行と「他者送付」で引け。 — Thariq Shihipar (要約)
